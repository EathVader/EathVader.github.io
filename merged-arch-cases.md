## 一、架构演进模式：从单体到微服务

### 1.1 演进的普遍规律

几乎所有互联网公司的架构都经历了相似的演进路径：

```
单体应用 → 垂直拆分 → 分布式服务化 → 微服务架构 → 云原生/容器化
```

- **单体阶段**：所有功能集中在一个应用中，适合创业初期快速试错，研发速度快、学习成本低、运维简单。
- **垂直拆分阶段**：按业务线拆分为多个独立子系统，各系统独立部署，DB 按系统级拆分并实现物理隔离。
- **服务化阶段**：公共服务从业务系统中解耦，独立部署维护；系统间通过 RPC（如 Hessian、Dubbo）通信。
- **微服务阶段**：以服务为中心，每个微服务围绕具体业务构建，独立部署、独立维护、独立扩展。
- **云原生阶段**：Docker 容器化部署、API 网关、跨 IDC 多机房部署、弹性伸缩。

### 1.2 架构演进的驱动力

| 驱动因素 | 具体表现 |
|---------|---------|
| 业务复杂度增长 | 业务线增多，代码臃肿，迭代速度下降 |
| 用户规模爆发 | 高并发压力，单机性能瓶颈 |
| 团队规模扩大 | 多人协作困难，缺少并行研发能力 |
| 数据量膨胀 | 单库单表瓶颈，需要分库分表 |
| 可用性要求提升 | 单点故障风险，需要高可用架构 |

### 1.3 各大互联网公司架构演进案例索引

- **阿里系**：支付宝高可用与容灾架构、聚划算架构演进、淘宝交易系统演进、手淘移动端架构
- **腾讯系**：微信后台系统演进、微信红包架构、QQ 音乐架构演进
- **京东**：峰值系统设计、咚咚架构演进
- **新浪微博**：平台架构、图床架构、推荐架构演进
- **携程/去哪儿**：技术演进之路、机票搜索/支付系统架构
- **国际公司**：LinkedIn 十年架构、Facebook 软件架构、Uber 四年增长 40 倍架构
- **垂直领域**：知乎架构变迁、豆瓣基础架构、小米抢购系统、饿了么移动架构

### 1.4 架构选型原则

1. **选择合适的技术而非最好的技术**：权衡研发效率和产品目标
2. **架构围绕业务**：不同业务时期需要不同架构
3. **提前规划扩展性**：单体阶段就在业务层面规划好未来解耦路径
4. **渐进式演进**：在维护现有系统的基础上同时进行架构重构，不能停下业务

## 二、去哪儿网案例：搜索系统与支付系统

### 2.1 机票搜索系统

#### 2.1.1 业务背景与挑战

去哪儿网机票搜索系统在高峰期承载**日均十亿级 PV、月均上亿 UV**，核心指标：
- 搜索缓存命中率 > 99.7%
- 每台搜索实例 QPS > 1000（50~60 台虚拟机实例）
- 搜索结果 Average-Time < 3 秒

机票报价搜索的特殊性：
- 不是简单的数据检索，更像**实时计算系统**
- 报价计算链路长：航司运价 + 库存情况 + 代理商调价 + 退改签/税费 + 附加产品
- 数据变更频繁：航司运价规则 1000 万+、代理商调价规则 1 亿+、航班数据 200 万+
- 部分基础资源在 GDS（全球分销系统）手中，需付费获取

#### 2.1.2 系统设计思路

两种方案对比：

| 方案 | 优点 | 缺点 |
|-----|------|------|
| 全量预计算 | 响应快、准确性好 | 资源消耗巨大，变更频繁时重复计算量大 |
| 按需实时计算 | 资源要求低、扩展性好 | 响应慢、缓存影响有效性 |

最终选择：**按需实时计算 + 报价结果缓存 + 缓存闭环刷新**

#### 2.1.3 整体架构（水平分层 + 垂直按引擎划分）

```
终端渠道层（APP/WWW/低价）
    ↓
报价聚合层（全局缓存 + Cache Manager + 有效性检查）
    ↓ MQ + Redis（数据交换区，解耦）
报价计算引擎层
  ├── 上层：航路规划、产品包装、大交通
  └── 下层：TTS、旗舰店、抓取等多渠道并行计算
    ↓
基础数据层（运价、调价规则、航班数据）
```

**关键设计点**：
- **MQ + Redis 交互方式**：解耦聚合层与引擎层，新增/减少引擎不影响聚合层；将单向计算流展开，无需层层回调，节省大量网络流量
- **缓存管理**：所有影响报价的变更收拢到 Cache Manager，报出缓存前检查有效性
- **多引擎并行**：各渠道报价引擎并行计算，形成互补优势

#### 2.1.4 前台搜索（读缓存 + 航线计算）

- 基于 Web 服务，高峰期 50 台 Tomcat 实例
- Nginx 负载均衡 + Lua 脚本区分国际/国内航线
- 缓存存储粒度：出发城市 + 到达城市 + 出发日期
- 航线组合本质是**有向图最优路径问题**（最短飞行时间/最低价格/最少中转）
- **大量使用 Java 多线程并行计算**：FutureTask 并行处理各段航班
- **自定义 txt 格式传输数据**：压缩到原来 30%~40%，极大节约带宽和 CPU

#### 2.1.5 后台搜索（GDS 数据抓取 + 异步缓存更新）

GDS 系统的挑战：
- 服务器遍布全球，通讯协议各异（HTTP/Socket）
- 服务不稳定，尤其国外 GDS
- 按次付费查询，无法承载高并发

解决方案：
- **引入 NIO 框架**：异步非阻塞 I/O，支持更长超时时间，降低 CPU 和内存开销
- **消息队列异步更新缓存**：前台搜索通知 MQ → Router 路由 → GDS 查询 → 异步更新缓存
- **GDS 抽象为虚拟 Node**：根据稳定性和性能计算权重，分配虚拟 Node 节点，由 Node 池统一管理和动态分配
- **Router 路由规则**：可细化到具体航段的缓存有效期；指定航线使用特定 GDS 数据源（"去伪存真"）

#### 2.1.6 监控系统

- 系统层：基于 CACTI + NAGIOS，多维度 Dashboard
- 业务层：自研监控插件，数据先存内存，每分钟上传监控平台，即插即用，支持自定义统计
- 24 小时 NOC 小组处理线上报警

### 2.2 支付系统架构演进

#### 2.2.1 支付系统 1.0（单体）

所有功能耦合在一个系统：收银台、交易、支付、网关、账务。

问题：
- 容灾能力差，一个功能出问题影响全局
- 扩容效果差（木桶效应）
- 开发效率低，多项目并行困难
- 系统职责混乱

#### 2.2.2 支付系统 2.0（服务化拆分）

按**交易时序**拆分：下单 → 交易 → 收银台 → 支付 → 网关 → 银行

**网关拆分**：
- 高度抽象为标准服务（支付、退款、查询）
- 路由系统支持多通道智能选择

**账务系统拆分**：
- 从简单流水记录升级为专业复式记账

**交易核心（Apollo）**：
- 支持单笔/批量订单支付
- 交易类型：直接交易、担保交易、分账交易、预授权交易等
- 逆向流程：退款、撤销、预授权确认/撤销、二次分账

**支付核心（Minos）**：
- 支付方式：银行卡、支付宝、微信、拿去花、余额、积分、红包、代金券、立减等
- 支持单一使用和组合使用
- 支持同一批次多次组合扣款

**收银台**：
- 多端支持：APP Native、Touch、PC 标准/多单/预授权/英文版
- 直接影响订单转化率

**API 接入层读写分离**：
- 读服务（Ares）：包装 Dubbo 服务提供查询
- 写服务（Odin）：核心业务写操作（解冻、退款、撤销）

#### 2.2.3 DB 处理策略

- **分表**：按时间（月表/季表）或按 Key Hash，支持历史数据迁移
- **分库**：业务拆分后 DB 独立，保守方案（先建从库 → 切换 → 删除无关表）
- **读写分离 + 读负载均衡**：充分利用 MMM 结构资源
- **多动态数据源**：支持多库动态源读负载均衡

#### 2.2.4 异步化

- Servlet 3 异步：释放 HTTP 线程，隔离不同业务工作线程
- QMQ 消息队列：最广泛的异步方式
- Dubbo 异步：应对服务提供者响应慢的场景
- 组合扣款场景：HTTP 服务轮询 + QMQ 驱动扣款 + Cache 存储中间结果

#### 2.2.5 监控与报警

- Java 监控模块嵌入应用，支持多种数据发送方式
- Python 离线监控框架：metric_manager + graphite_sender + dbpool + scheduler
- 数据流系统：xflume → Kafka → Storm → HDFS/HBase/Redis/Hive
- 报警分类：明确失败报警（NPE/ERROR/慢查询）+ 计算类报警（期望值偏差）

## 三、创业公司架构变迁：贝聊 5 年实战

### 3.1 V1.0 诞生期：LNMP 单体架构

**背景**：创业初期，1 名 PHP 开发，需要快速上线。

**架构**：经典 LNMP（Linux + Nginx + MySQL + PHP + Memcached），3 台 ECS 服务器。

**特点**：
- 单体架构，简单清晰
- 快速研发，满足快速迭代
- 技术学习成本低，运维成本低

**支撑时间**：约 1.5 年

### 3.2 V2.0 成长期：Java 分布式架构

**重构契机**：
- 数据库设计混乱（两个团队维护遗留问题）
- APP 拆分为家长端和老师端，新旧接口逻辑混杂
- 单体代码臃肿，迭代慢，BUG 多
- 所有数据在一个 RDS 主库，无从库，慢查询导致雪崩
- 公共服务耦合严重，散落在各系统

**系统拆分两阶段**：
1. **第一阶段**：系统层面拆分，多系统共用一个 DB，按业务划分表的归属
2. **第二阶段**：DB 层面拆分，各子系统 DB 独立部署，实现主从分离

**关键改进**：
- 初步服务化：Hessian 实现 RPC
- 引入 MQ 消息队列：异步化消息推送，加快接口响应
- SLB 负载均衡：多台 Nginx 负载均衡，消除单点故障

### 3.3 V3.0 爆发期：微服务架构（Dubbo）

**驱动力**：用户规模短时间增长数百万，研发团队达几十人，业务逻辑耦合严重。

**选择 Dubbo 的原因**：
- 成熟的高性能分布式框架
- 与 Spring 无缝集成，代码无侵入
- 具备服务注册/发现/路由/负载均衡/降级/权重调节能力
- 开源可定制

**微服务拆分案例**：

**班级动态微服务**：
- 日动态量数十万，日回复/点赞量数百万
- 采用 DRDS 分库分表，性能提升数倍
- 对调用方透明，代码复用

**用户通行证微服务**：
- 解决用户逻辑代码散落各系统的问题
- 保证用户数据一致性
- 用户数据库独立，方便扩容和性能优化

**微服务治理体系**：

| 治理能力 | 实现方案 |
|---------|---------|
| 版本发布 | 自研发布系统：一键部署、灰度发布、快速回滚、上线审核 |
| 环境管理 | 四套环境：开发 → 测试 → 预发布（共用生产 DB）→ 生产 |
| 配置中心 | Disconf：配置云端管理、加密、无需重启 |
| 任务调度 | Elastic-Job：分片、弹性扩缩容、失效转移 |
| 全链路跟踪 | 美团 Cat + Dubbo 整合，全局链路 ID |
| 服务授权 | 自研授权认证中心，保证核心接口调用安全 |
| 服务监控 | Dubbo 监控中心 + 定制化统计（耗时/失败/异动排行） |
| 服务管理 | Dubbo 管理控制台：路由/访问控制/权重/降级/容错 |

### 3.4 V4.0 未来规划

- Docker 容器化部署：快速弹性扩展
- 统一 API 网关：身份认证、防篡改、流量控制
- 跨 IDC 机房部署：同地多机房 → 异地跨 IDC

### 3.5 核心经验总结

- 单体架构适合创业初期，但设计时要提前规划扩展性
- 微服务架构提高并行研发速度和模块复用性，但对服务治理要求高
- 架构来源于业务，演进生命周期必须匹配业务生命周期

## 四、全球分布式系统设计

### 4.1 核心原则（来自 NSONE 实战经验）

#### 4.1.1 架构优先，而非代码优化

> Don't optimize your code, optimize your architecture.

- 现代分布式系统的扩展瓶颈不在于单个角色的代码深度，而在于**系统间通信**和**各角色的水平扩展能力**
- 微服务架构解耦独立角色，使每个角色可独立扩展
- **优先考虑水平扩展**：服务器便宜且越来越快，开发者时间不是
- 在写代码之前先画架构图，思考通信和数据约束

#### 4.1.2 以角色（Role）思考系统

- 将应用拆分为解耦的、互相通信的角色
- 初期可以在单台服务器上以进程/VM/容器形式共存
- 随着增长，逐步将需要独立资源的角色剥离到专用服务器或集群
- NSONE 案例：从 AWS 单台 t2.small 实例起步，发展到横跨全球近 30 个数据中心的数百台服务器

#### 4.1.3 跨数据中心通信是核心难题

- 互联网连接脆弱且不可预测，尤其在非洲、巴西、印度等市场
- 设计时假设边缘设施会频繁失去与核心设施的连接
- **不同通信模式使用不同工具**：
  - 延迟敏感的关键控制消息 → 健壮的消息队列（如 RabbitMQ）
  - 非关键遥测数据 → 轻量但不太健壮的方案
  - 需要跨地域同步的持久化数据 → 支持 WAN 复制的现代数据库

#### 4.1.4 一致性哈希是同步与分片的利器

- 分布式系统无需实际通信即可达成一致（如请求到节点的映射）
- 实现简单，随基础设施增长优雅扩展，最小化重新分片
- NSONE 应用场景：
  - DNS 查询路由到特定 CPU 核心以最大化缓存局部性
  - 监控节点任务协商（无需显式通信）
  - 高流量数据 Feed 跨聚合节点分片

#### 4.1.5 测量和监控一切

**测量（Measure）**：
- 不仅收集系统指标，还要**深入应用层**：数据库响应时间、消息延迟、缓存命中率、内存碎片、磁盘 I/O
- 建立对"正常行为"的直觉理解
- 异常事件发生后回顾指标，理解事件"签名"

**监控（Monitor）**：
- 测量是为了理解行为，监控是为了检测异常
- **从多个维度监控**：物理维度（多地域多网络）+ 逻辑维度（内部遥测 + 第三方服务）
- **不仅监控"是否存活"**，还要监控服务质量指标（延迟、缓存命中率等）

### 4.2 广发证券分布式行情与交易系统

#### 4.2.1 业务特点

- 行情用户并发峰值高（牛市同时在线 50~80 万）
- 行情数据时效性要求高（交易所 3 秒一个快照）
- 推送流量大（约 100~200 万 QPS）
- 指标计算量大（每日 10 亿级别以上）
- 安全、稳定、快速

#### 4.2.2 架构设计

**横向拆分**（按功能）：实时行情、K 线、分时、分笔、板块、资金流向、代码链、鉴权、交易、期权期货、基金债券、Lv2 行情

**纵向拆分**（按层次）：
- 接入网关 → 读逻辑层 → 数据存储层 → 写逻辑层 → 行情转码层
- 交易系统：接入层 → 逻辑层 → 柜台接入层 → 柜台交易层

**接入层设计指标**：单进程（4 核 1G 内存）支持 5 万连接、10 万 QPS

#### 4.2.3 高并发高性能关键技术

- **收敛 Goroutine 数量**：用 Epoll 管理连接代替每连接两个协程，将 10 万协程收敛为一个整体
- **CGO 通信优化**：无锁队列 + 内存通信代替 CGO 调用
- **连接内存池**：一次性创建 5 万个 Hash 桶，用户上下线复用内存对象
- **Chan 隔离业务**：不同业务使用不同 Chan，互不影响，削峰填谷
- **自定义协议**：Handshake 压缩包头，接入协议与内部协议分离

#### 4.2.4 高可用与可扩展

- **服务发现**：Consul + Docker 容器化部署 + Registrator 自动注册
- **动态负载均衡**：根据调用成功次数、耗时、连接情况实时计分，毫秒级故障摘除，秒级新进程加入
- **解码容灾**：三套解码系统，Repeater 对两路行情数据冗余去重
- **监控告警**：业务进程 UDP 上报 → Monitor → InfluxDB → Grafana

#### 4.2.5 Go 语言挑战

- Goroutine 调度无优先级，10 万级别调度不均衡
- 大量 Goroutine 创建销毁导致 GC 影响响应速度
- CGO 通信机制低效，需自行优化

## 五、负载均衡原理

### 5.1 为什么需要负载均衡

当单台服务器的 CPU、内存、硬盘不堪重负时，有两种扩展方式：
- **垂直扩展（Scale Up）**：购买更强大的服务器，成本高昂
- **水平扩展（Scale Out）**：多台便宜服务器分担负载，性价比高

水平扩展的核心问题：**从客户角度看，最好只有一个服务器**。

### 5.2 DNS 负载均衡（最简方案）

将域名映射到多个服务器 IP，轮询返回不同 IP。

**致命缺陷**：DNS 分层系统和客户端都有缓存，某台机器故障后，缓存仍返回故障机器 IP，即使从 DNS 删除也无法立即生效。

### 5.3 NAT 模式（四层负载均衡）

**核心原理**：Load Balancer（LB）修改数据包的目的地 IP 和端口，转发给后端真实服务器（RS）；RS 响应经过 LB 时，LB 再将源地址改回自己的地址。

```
数据流向：客户端 → LB → RS → LB → 客户端
```

**LB 操作**：
1. 收到客户端数据包，将目的 IP/端口改为选中的 RS
2. RS 处理完毕，响应包经过 LB（LB 是网关）
3. LB 将源 IP/端口改回自己的，发给客户端

**瓶颈**：所有流量（请求 + 响应）都经过 LB，而响应报文通常远大于请求报文。

### 5.4 DR 模式（直接路由，高效方案）

**核心思想**：请求和响应分开处理，LB 只处理请求，RS 直接响应客户端。

**实现方式**：
1. 所有服务器（LB + RS）共享同一个 VIP（虚拟 IP）
2. 只让 LB 响应 VIP 的 ARP 请求，RS 抑制 ARP 响应
3. LB 收到数据包后，**不修改 IP 数据报**，只修改数据链路层的目的 MAC 地址为选中 RS 的 MAC
4. RS 收到数据包，目的 IP 是自己的 VIP，正常处理
5. RS 直接响应客户端（源地址就是 VIP）

```
数据流向：客户端 → LB → RS → 客户端（响应不经过 LB）
```

**优势**：LB 只负责转发请求，响应由 RS 直接返回，消除了 LB 的带宽瓶颈。

**约束**：RS 的端口必须与 LB 一致（因为 LB 不修改 IP 层数据）。

### 5.5 负载均衡策略

| 策略 | 说明 |
|-----|------|
| 轮询 | 依次分配，最简单 |
| 加权轮询 | 性能好的服务器权重高，被选中概率大 |
| 最少连接 | 当前连接数最少的服务器优先 |
| 加权最少连接 | 最少连接 + 权重 |
| 一致性哈希 | 相同请求始终路由到同一服务器，最大化缓存局部性 |

### 5.6 四层 vs 七层负载均衡

- **四层（传输层）**：基于 TCP 连接，维护连接映射表，确保同一连接的数据包发往同一 RS
- **七层（应用层）**：解析 HTTP 报文，根据 URL、浏览器、语言等信息智能分发

> 以上即为著名开源软件 **LVS**（Linux Virtual Server）的核心原理，由章文嵩博士于 1998 年创建，现已成为 Linux 内核的一部分。

## 六、大数据分布式计算

### 6.1 为什么需要分布式计算

当数据量远超单台计算机处理能力时（单位时间数据量大或数据总量大），需要多台计算机协作。

### 6.2 MapReduce 模型

MapReduce 是 Google 于 2004 年发布的分布式计算框架，与 GFS、BigTable 并称 Google 大数据"三宝"。开源实现为 Hadoop。

### 6.3 四字诀：分、变、洗、合

#### 第一步：切分（Split）

将输入数据切分成多份，每份由一个"Map 计算节点"处理。

- **指挥官**（Master）：负责数据切分和任务调度
- **Map 计算兵**：负责变换操作
- **Reduce 计算兵**：负责合并操作

#### 第二步：变换（Map）

每个 Map 节点对分到的每条数据按**相同规则**做映射变换。

例如：将每张扑克牌转换为 `<牌型, 1>` 的键值对。

**本质**：对每条数据执行同一种变换规则，严格流水化操作提升整体效率。

#### 第三步：洗牌（Shuffle）

将变换后的数据按规则分组，确保相同 Key 的数据汇聚到同一个 Reduce 节点。

- **阶段一**：Map 节点按 Reduce 节点的负责范围将数据分类
- **阶段二**：Reduce 节点从各 Map 节点获取属于自己的数据

#### 第四步：合并（Reduce）

Reduce 节点将同一 Key 的数据按计算规则合并（如累加），输出最终结果。

### 6.4 完整流程

```
输入数据 → 切分 → Map（变换）→ Shuffle（洗牌）→ Reduce（合并）→ 输出结果
```

### 6.5 核心要点

- 程序员只需定义两种规则：**Map 变换规则**和 **Reduce 合并规则**
- 分布式框架自动处理：任务调度、节点间通信、容错恢复等复杂过程
- MapReduce 是大数据技术的基石，理解它就理解了大数据分布式处理的核心

## 七、混沌工程（Chaos Engineering）

### 7.1 什么是混沌工程

混沌工程旨在**将故障扼杀在襁褓之中**——在故障造成中断之前将它们识别出来。通过主动制造故障，测试系统在各种压力下的行为，识别并修复问题。

> 混沌工程将预想的事情与实际发生的事情进行对比，通过"有意识地搞破坏"来提升系统的弹性。

### 7.2 发展历史

| 时间 | 里程碑 |
|-----|--------|
| 2010 | Netflix 开发 Chaos Monkey，测试 AWS 实例故障对用户体验的影响 |
| 2011 | Simian Army 诞生，增加更多故障注入模式 |
| 2012 | Netflix 在 GitHub 开源 Chaos Monkey |
| 2014 | "混沌工程师"角色诞生；FIT（故障注入测试）概念提出，可控制故障粒度 |

### 7.3 三步法

1. **假设**：假设系统在出现故障时的行为
2. **试验**：设计针对性试验测试系统
3. **评估**：评估故障产生的影响（正面/负面征兆）

### 7.4 分布式系统八大谬论（Peter Deutsch）

1. 网络是可靠的
2. 延迟是零
3. 带宽是无限的
4. 网络是安全的
5. 拓扑结构不会变
6. 存在管理员这样的角色
7. 传输成本是零
8. 网络是同质的

混沌工程正是针对这些谬论进行验证和测试。

### 7.5 测试执行顺序

按照认知矩阵递进：

| 类型 | 含义 | 示例 |
|-----|------|------|
| Known Known | 已了解并注意到的 | 副本宕机会被移除并重建 |
| Known Unknown | 注意到但不了解的 | 故障到恢复需要多长时间？ |
| Unknown Known | 了解但没注意到的 | 同时关闭两个副本时主库能否承受复制负载？ |
| Unknown Unknown | 既没注意也不了解的 | 关闭整个集群会发生什么？ |

### 7.6 规划混沌测试的步骤

1. **计划**：围绕"哪里可能出问题"展开讨论，在白板上画出服务、依赖和数据存储
2. **假设**：对预期结果建立假设（对客户/服务/依赖的影响）
3. **评估**：使用客户相关度量指标（如每分钟订单量），异常时立即停止测试
4. **回退计划**：制定回退方案，接受回退也可能失效的事实
5. **修复**：验证系统弹性或找到修复方式

### 7.7 混沌工程的价值

| 维度 | 价值 |
|-----|------|
| 客户 | 增强服务可用性和持久性 |
| 业务 | 避免利润损失，降低维护成本，改进事故处理流程 |
| 技术 | 减少事故和轮班待命负担，更好理解故障模式，更快检测事故 |

### 7.8 实践公司

Netflix、Twilio、LinkedIn、Facebook、Google、Microsoft、Amazon，以及澳大利亚国家银行等金融机构。

## 八、系统设计核心组件

### 8.1 Web 应用架构全景

```
客户端 → DNS → CDN → 负载均衡/反向代理 → 前端应用
                                            ↓
                                        消息队列 ← → 队列工作者
                                            ↓
                                        Web 服务（后端）
                                            ↓
                                    数据存储 / 搜索引擎 / 缓存
```

### 8.2 前端层组件

- **DNS**：域名到 IP 的映射
- **CDN**：缓存静态文件（图片/CSS/JS/视频），加速用户访问
- **负载均衡/反向代理**：托管服务（如 AWS ELB）、软负载（如 Nginx）、硬件负载
- **前端 Web 服务器**：表现层 + 后端结果聚合

**HTTP 会话管理**：Cookie / 外部数据存储 / 负载均衡器会话保持

### 8.3 后端层 / Web 服务

**实现方式**：
- 单体应用 + 按需增加 Web 服务
- API 优先设计：所有客户端使用同一套 API
- 两者结合

**Web 服务类型**：
- 以功能为中心：SOAP（XML + HTTP），更复杂但更安全
- 以资源为中心：REST + JSON，轻量，OAuth 2 认证

**扩展策略**：
- 按功能分片/分区（微服务，建议团队超过 10~20 人时采用 SOA）
- 增加克隆副本数
- HTTP 协议缓存

### 8.4 数据层

**关系型数据库扩展**：
- **复制同步**：主库写，多副本读；适合读密集型应用；副本可能返回过期数据
- **数据分区/分片**：按 Sharding Key 切分数据集；无法跨分片查询；添加服务器可能困难

**NoSQL 扩展**（Cassandra、Redis、MongoDB、Riak、CouchDB）：

**CAP 定理**（Eric Brewer）：分布式系统不可能同时满足一致性、可用性和分区容错性，最多满足其中两个。
- MongoDB：CP（一致性 + 分区容错，牺牲可用性）
- Cassandra：AP（可用性 + 分区容错，不保证强一致性）

**趋势**：按功能划分 Web 服务层，对应采用不同的数据存储（Purpose-built Database）。

### 8.5 图数据库与数据关系

**图数据模型**：关系是数据模型的核心部分，数据建模为节点（Vertices）和链接（Edges）。

**适用场景**：
- 社交网络：用户关系、社交 Feed 优先级
- 推荐引擎：基于兴趣/好友/购买历史的个性化推荐
- 欺诈检测：多人共享邮箱/IP 等关系模式识别
- 知识图谱：疾病与基因交互、Alexa 知识图谱

**两种图模型**：
- **属性图（Property Graph）**：类似传统数据结构，适合独立应用
- **RDF（资源描述框架）**：W3C 标准，支持互操作和数据交换，有标准化查询语言 SPARQL

**Amazon Neptune**：支持两种模型（Gremlin + SPARQL），全托管图数据库，6 副本跨 3 个可用区。

### 8.6 缓存体系

**基于 HTTP 的缓存**（读穿透）：
- 浏览器缓存
- 缓存代理（公司网络/ISP）
- 反向代理（如 Nginx）
- CDN

**自定义对象缓存**：
- 应用内存/RAM
- 共享内存（同机器多进程共享）
- 本地缓存服务器
- 分布式对象缓存（Redis、Memcached）

**缓存要点**：
- 追求更高的缓存命中率
- 适合读频繁的应用，写频繁时效果有限
- 可在后续阶段按需添加

### 8.7 异步处理与消息队列

**同步 vs 异步**：
- 同步：调用方等待响应，无法构建现代响应式应用
- 异步：即发即忘，客户端无需等待处理完成

**消息队列组件**：
- 消息生产者 → 消息队列 → 消息消费者（拉取模式/推送模式）

**消息平台选择**：
| 平台 | 特点 |
|-----|------|
| Amazon SQS | 简单实用，创业早期首选 |
| RabbitMQ | 特性丰富，复杂路由，简单灵活 |
| ActiveMQ | 基于 Java，低延迟，路由不够灵活 |

**事件驱动架构**：
- 组件声明已发生的事件（而非请求完成工作）
- 发布者和消费者彼此解耦，只知道事件消息的格式和含义

### 8.8 搜索引擎

- 全表扫描效率低，索引加速搜索
- 复杂搜索查询建议使用专用搜索引擎
- 搜索引擎通常使用**倒排索引**
- 可选方案：Elasticsearch、Solr、Sphinx、Amazon CloudSearch、Azure Search

### 8.9 可扩展性总结

**三种扩展方式**：
1. 增加克隆副本/服务器（最简单、最廉价）
2. 按功能划分（服务器专项化，SOA）
3. 按数据划分（分库分表、分片）

**不仅是架构**，还包括：
- 自动化（测试、构建、部署、监控、告警、日志聚合）
- 团队管理（小型跨职能自主团队、知识分享、避免过度加班）
- 使用第三方服务，不要重复造轮子
- 小范围发布，收集反馈，不要想当然写代码
