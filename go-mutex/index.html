1 什么是 Mutex？/ What is a Mutex?

🚻

生活类比 / Real-life Analogy

想象一个单人洗手间：门上有一把锁。进去的人锁上门（Lock），出来时打开锁（Unlock）。其他人必须在外面等待。这就是 Mutex（互斥锁）的核心思想——同一时刻只有一个线程能进入临界区。

Think of a single-occupancy restroom: one person enters and locks the door (Lock), others wait outside until it's unlocked (Unlock). That's mutual exclusion.

Mutex 是 Mutual Exclusion（互斥）的缩写。在多线程编程中，它保证同一时刻只有一个线程能执行"临界区"代码。

Mutex 保护临界区：Lock() → 执行 → Unlock()

各语言的用法非常简洁：

Go
var mu sync.Mutex

mu.Lock()
// critical section（临界区）
mu.Unlock()

接口也很简单——只需要两个方法：

type locker interface {
    Lock()    // 获取锁，如果已被持有则阻塞等待
    Unlock()  // 释放锁
}
💡 关键概念 / Key Concept

临界区 (Critical Section)：需要互斥访问的代码段。比如多个线程同时修改同一个计数器，如果不加锁，结果就会出错。

2 天真的尝试 / A Naive Attempt

最直觉的想法：用一个 bool 变量记录锁的状态。获取锁时检查是否为 false，是就设为 true；释放时设回 false。

⚠️ 这个实现有 Bug！/ This implementation is broken!
type NaiveSpinLock struct {
    state bool
}

func (l *NaiveSpinLock) Lock() {
    for l.state != false {
        // busy-wait（忙等待）
    }
    l.state = true
}

func (l *NaiveSpinLock) Unlock() {
    l.state = false
}

为什么不行？/ Why doesn't this work?

因为存在数据竞争 (Data Race)。两个线程可能同时通过检查，都认为自己拿到了锁：

⏱ Step 1
Thread A 检查 l.state == false ✅ 通过循环
⏱ Step 2
Thread B 检查 l.state == false ✅ 也通过了！
⏱ Step 3
Thread A 设置 l.state = true（以为自己拿到了锁）
⏱ Step 4
Thread B 设置 l.state = true（也以为自己拿到了锁！💥）
🚪

类比 / Analogy

就像两个人同时看到洗手间门是开的，都冲了进去——因为"看"和"锁门"是两个独立的动作，中间可能被打断。

问题的根源："检查"和"设置"不是一个原子操作。

数据竞争：两个线程的"读取"和"写入"交错执行

3 自旋锁 / Working Spin Lock

解决方案：使用 CPU 提供的原子指令，让"读取 + 写入"变成一个不可分割的操作。

🔄

类比 / Analogy

atomic.SwapUint32(&state, 1) 就像一个"旋转门"——你推门进去的同时，门会告诉你之前是开着还是关着的。这个"推门 + 获取状态"是一个原子动作，不可能被打断。

import "sync/atomic"

type SpinLock struct {
    state uint32 // 0 = unlocked, 1 = locked
}

func (l *SpinLock) Lock() {
    for atomic.SwapUint32(&l.state, 1) == 1 {
        // busy-wait（忙等待）
    }
}

func (l *SpinLock) Unlock() {
    atomic.StoreUint32(&l.state, 0)
}
原子 Swap：返回旧值 0 表示获取成功，返回 1 表示需要重试

优化 1：让出 CPU / Yielding

纯忙等待会浪费 CPU。Go 运行时可能让一个 goroutine 空转长达 10ms 才切换。加上 runtime.Gosched() 主动让出：

func (l *SpinLock) Lock() {
    for atomic.SwapUint32(&l.state, 1) == 1 {
        runtime.Gosched() // 让出 CPU，让其他 goroutine 运行
    }
}
📊 效果 / Result

测试耗时从 6~10 秒降到 ~3.34 秒！因为等待的 goroutine 不再空转，而是主动让出 CPU。

优化 2：Test-and-Test-and-Set

原子 Swap 指令需要刷新 CPU 缓存，比较昂贵。优化思路：先用便宜的 Load 检查，只在看到 0 时才尝试 Swap：

func (l *SpinLock) Lock() {
    for {
        for atomic.LoadUint32(&l.state) == 1 {
            runtime.Gosched() // 便宜的读取 + 让出
        }
        if atomic.SwapUint32(&l.state, 1) == 0 {
            return // 获取成功！
        }
    }
}

4 性能测量 / Measuring Performance

SpinLock 的墙钟时间 (wall time) 看起来和 sync.Mutex 差不多，但用 /usr/bin/time 看 CPU 时间就暴露了问题：

⏱ 三种时间 / Three Types of Time
  • Real time（墙钟时间）：你实际等了多久
  • User time（用户态 CPU 时间）：程序自己的代码用了多少 CPU
  • System time（内核态 CPU 时间）：操作系统内核为程序花了多少 CPU

多核 CPU 下，User + System 可以大于 Real time！比如 10 核 CPU，3 秒内可以完成 30 秒的 CPU 工作。

🎯 结论 / Takeaway

SpinLock 的墙钟时间差不多，但 CPU 消耗是 sync.Mutex 的 2 倍多！大量 CPU 时间浪费在 Go 调度器反复暂停/恢复忙等待的 goroutine 上。

5 Futex：操作系统来帮忙 / OS-Assisted Locking

自旋锁的问题是忙等待浪费 CPU。解决方案：让操作系统帮忙——用 Futex（Fast Userspace Mutex）让等待的线程真正"睡觉"。

🏨

类比 / Analogy

自旋锁：你站在洗手间门口不停敲门问"好了没？好了没？"（浪费精力）
Futex：你在旁边的椅子上坐下睡觉，里面的人出来时会叫醒你（节省精力）

Futex 工作流：获取失败 → 睡眠等待 → 被唤醒 → 重试

基础 FutexLock 实现

type FutexLock struct {
    state uint32
}

func (m *FutexLock) Lock() {
    for {
        for atomic.LoadUint32(&m.state) == 1 {
            futexWait(&m.state, 1) // 睡眠，而不是忙等待
        }
        if atomic.SwapUint32(&m.state, 1) == 0 {
            return
        }
    }
}

func (m *FutexLock) Unlock() {
    atomic.StoreUint32(&m.state, 0)
    futexWake(&m.state, 1) // 唤醒一个等待者
}
📝 注意 / Note

Futex 是 Linux 特有的系统调用。macOS 有类似的 os_sync_wait_on_address，Windows 有 WaitOnAddress。

6 优化 Futex / Optimized Futex Lock

基础版的 Unlock() 每次都调用 futexWake()，即使没有人在等待。系统调用是昂贵的！

优化思路：用三个状态来追踪是否有等待者：

0
未锁定
Unlocked
→
1
已锁定，无等待者
Locked, no waiters
→
2
已锁定，有等待者
Locked, has waiters
三状态机：状态 1 → 0 时无需系统调用，节省开销
func (m *FutexLock) Lock() {
    // 快速路径：锁未被持有
    if atomic.CompareAndSwapUint32(&m.state, 0, 1) {
        return
    }
    // 慢速路径
    for {
        if atomic.SwapUint32(&m.state, 2) == 0 {
            return // 获取成功，标记为有竞争状态
        }
        _ = futexWait(&m.state, 2) // 睡眠等待
    }
}

func (m *FutexLock) Unlock() {
    prev := atomic.SwapUint32(&m.state, 0)
    if prev == 0 {
        panic("unlock of unlocked FutexLock")
    }
    if prev == 1 {
        return // 无等待者，不需要 Wake 系统调用！
    }
    _ = futexWake(&m.state, 1) // 有等待者，唤醒一个
}
✅ 关键优化 / Key Optimization

当 prev == 1（无竞争）时，Unlock() 直接返回，省掉了一次系统调用。在锁竞争不激烈时，这个优化效果显著。

7 自适应锁 / Adaptive Spinning

自旋锁快但费 CPU，Futex 省 CPU 但系统调用有开销。最佳方案：先自旋一小会儿，不行再睡觉。

⏳

类比 / Analogy

等电梯时：先等 30 秒看看电梯来不来（自旋），如果还没来就去坐椅子上等（Futex 睡眠）。大多数时候电梯很快就来了，你不用坐下又站起来。

func (m *FutexLock) Lock() {
    // 快速路径
    if atomic.CompareAndSwapUint32(&m.state, 0, 1) {
        return
    }

    // 中速路径：自旋一小会儿
    for i := 0; i < 30; i++ {
        if atomic.LoadUint32(&m.state) == 0 {
            if atomic.CompareAndSwapUint32(&m.state, 0, 1) {
                return
            }
        }
        runtime.Gosched()
    }

    // 慢速路径：放弃自旋，用 Futex 睡眠
    for {
        if atomic.SwapUint32(&m.state, 2) == 0 {
            return
        }
        _ = futexWait(&m.state, 2)
    }
}
三级策略：快速 CAS → 短暂自旋 → Futex 睡眠

概念说明
数据竞争多线程同时读写共享变量，导致不可预测的结果
