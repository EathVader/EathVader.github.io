## 一、缓存更新策略

缓存更新是缓存系统中最核心也最容易出错的环节。业界有四种经典的 Design Pattern：

### 1.1 Cache Aside Pattern（旁路缓存，最常用）

这是 **Facebook 论文《Scaling Memcache at Facebook》** 采用的策略，也是业界最广泛使用的模式。

**核心流程：**

- **读（失效）**：先读 cache → 未命中 → 读数据库 → 写入 cache
- **读（命中）**：直接从 cache 返回
- **写（更新）**：**先更新数据库，成功后让缓存失效**（而非更新缓存）

**为什么是"失效"而非"更新"缓存？**

因为两个并发写操作可能导致缓存中的数据是脏的。Facebook 选择删除缓存而非更新缓存，就是为了避免这种并发写导致的脏数据。

**为什么不能"先删缓存，再更新数据库"？**

这是一个常见错误！考虑并发场景：
1. 更新操作删除缓存
2. 查询操作未命中缓存，从数据库读到**旧数据**放入缓存
3. 更新操作更新数据库

结果：缓存中永远是旧数据，且一直脏下去。

**Cache Aside 的极端 case：**

读操作未命中缓存 → 读数据库 → 此时写操作更新数据库并失效缓存 → 读操作把旧数据放入缓存。理论上会出现，但实际概率极低（需要读操作在写操作前进入数据库，又晚于写操作更新缓存）。

> **Facebook 的选择**：要么用 2PC/Paxos 保证强一致性（太慢/太复杂），要么拼命降低并发脏数据的概率。Facebook 选择了后者，并为缓存设置过期时间作为兜底。

### 1.2 Read/Write Through Pattern（读写穿透）

应用层认为后端就是一个单一存储，缓存层自己维护与数据库的同步：

- **Read Through**：缓存失效时，由缓存服务自己加载数据（对应用透明）
- **Write Through**：写数据时，命中缓存则更新缓存，由缓存同步更新数据库（同步操作）；未命中则直接更新数据库

### 1.3 Write Behind Caching Pattern（写回）

也叫 Write Back，与 **Linux 文件系统的 Page Cache 算法** 原理相同：

- 更新数据时**只更新缓存，不更新数据库**
- 缓存**异步批量**更新数据库
- 可以合并对同一数据的多次操作

**优点**：I/O 性能极高（直接操作内存），可合并写操作
**缺点**：数据不是强一致性，可能丢失（类似 Unix 非正常关机丢数据）；实现复杂，需要 track 哪些数据被更新（dirty bit）

### 1.4 四种模式总结

| 模式 | 一致性 | 性能 | 复杂度 | 适用场景 |
|------|--------|------|--------|----------|
| **Cache Aside** | 最终一致 | 好 | 低 | 通用场景，最推荐 |
| **Read/Write Through** | 较强 | 好 | 中 | 缓存层有代理能力 |
| **Write Behind** | 弱 | 极高 | 高 | 写密集、可容忍丢失 |

> **陈皓观点**：这些 Design Pattern 都是计算机体系结构里的老古董设计（CPU 缓存、硬盘缓存、数据库缓存），历经长时间考验。**如果你要做好架构，首先得把计算机体系结构以及很多老古董的基础技术吃透了。**

## 二、缓存问题（穿透/击穿/雪崩/惊群）

### 2.1 缓存穿透（Cache Penetration）

**定义**：查询一个一定不存在的数据，缓存和数据库都没有，每次请求都打到数据库。

**解决方案**：
- 布隆过滤器（Bloom Filter）：在缓存前加一层过滤
- 空值缓存：对不存在的 key 也缓存空结果，设置较短过期时间

### 2.2 缓存击穿（Cache Breakdown）

**定义**：某个热点 key 过期的瞬间，大量并发请求同时打到数据库。

**解决方案**：
- 互斥锁（Mutex）：只允许一个线程去加载数据
- 热点数据永不过期 + 异步更新
- Promise 缓存（见下文 Instagram 方案）

### 2.3 缓存雪崩（Cache Avalanche）

**定义**：大量缓存同时过期，或缓存服务宕机，导致请求全部打到数据库。

**解决方案**：
- 过期时间加随机值，避免同时失效
- 多级缓存架构（本地缓存 + 分布式缓存）
- 缓存高可用（Redis Cluster / Sentinel）
- 限流降级兜底

### 2.4 缓存惊群（Thundering Herd）

**定义**：缓存为空时（冷启动/重启/新请求），大量并发请求同时 cache miss，全部涌向后端。

这是 Instagram 在生产环境中遇到的核心问题。

## 三、Instagram 惊群方案（Promise-based Cache）

### 3.1 问题场景

Instagram 的服务是无状态的，依赖后端获取数据，通过缓存减少 90% 的后端请求。但在以下场景缓存失效：

1. **服务重启**：缓存被清空
2. **全新请求**：从未缓存过的数据
3. **新集群上线**：缓存完全为空

当服务有大量并发 worker 时，缓存为空意味着 **100 个并发请求全部 cache miss，产生 100 个独立的后端请求**——这就是惊群问题。

### 3.2 传统缓解方案及其不足

| 方案 | 缺点 |
|------|------|
| 外部缓存（memcache） | 增加复杂性和故障模式 |
| 本地磁盘持久化 | 增加复杂性 |
| 共享内存段 | 实现复杂 |
| 缓慢发布新客户端 | 限制业务敏捷性 |
| 预热（fake requests） | 不能覆盖所有场景 |

### 3.3 Promise 缓存方案（核心创新）

**关键思想：不缓存实际的值，而是缓存一个 Promise（将来会提供值的对象）。**

流程：
1. 请求到来，**原子地**检查缓存
2. 如果 cache miss，**创建一个 Promise 并立即插入缓存**
3. 这个 Promise 开始异步请求后端
4. 后续的并发请求会命中这个 Promise，**所有 worker 等待同一个后端请求**
5. 后端返回后，Promise 被 fulfill，所有等待的 worker 同时获得结果

**效果**：即使缓存为空，也能维持 90% 的缓存命中率假设——因为相同请求只会产生一次后端调用。

### 3.4 Instagram 的实现

- 使用 C++ 服务，基于 `folly::Future` 框架
- 使用 `folly::SharedPromise` 抽象简化实现
- 在新集群上线时显著减少了后端流量冲击

> **核心洞察**：Promise 缓存将"缓存值"升级为"缓存计算过程"，从根本上解决了冷缓存下的惊群问题。

## 四、内存一致性模型

### 4.1 为什么需要内存一致性模型

在多核/多线程环境下，**不同线程如何观察共享内存状态** 是一个根本性问题。内存一致性模型是硬件和软件之间的**契约**：硬件承诺只按模型允许的方式重排操作，软件承认所有允许的重排都可能发生。

### 4.2 顺序一致性（Sequential Consistency, SC）

由 Leslie Lamport 定义（2013 年图灵奖），两条规则：

1. **单一主存**：所有操作访问同一个主存，必须有全局顺序
2. **程序顺序**：单个线程内的事件按编写顺序发生

经典示例（A、B 初始为 0）：

```
Thread 1:        Thread 2:
(1) A = 1        (3) B = 1
(2) print B      (4) print A
```

在 SC 下，`00` 是**不可能**的输出（可通过 happens-before 关系的环路证明矛盾）。

**问题**：SC 极其缓慢——每次只能运行一条指令，必须等待写操作对所有线程可见。

### 4.3 缓存一致性（Coherence）

保证**对同一内存位置的所有写操作**被所有线程以相同顺序观察到。这是比 SC 更弱但必要的保证。

### 4.4 TSO（Total Store Ordering）

引入 **Store Buffer**（写缓冲区）来隐藏写延迟：

- 写操作先放入 store buffer（极快，on-core），不等待传播到 L3 cache（~90 cycles）
- 读操作先检查本地 store buffer，再读主存
- 保持单线程行为正确

**代价**：TSO 允许 SC 禁止的行为。上述示例在 TSO 下**可以**输出 `00`（两个写都在 store buffer 中，读操作看到的是旧值）。

**x86 架构**采用接近 TSO 的内存模型。

### 4.5 更弱的模型

- **ARM 架构**：弱排序（Weak Ordering），几乎允许任何重排
- **SPARC**：允许程序员在运行时选择不同模型

### 4.6 内存屏障（Memory Barrier / Fence）

所有现代架构都提供同步操作来恢复顺序保证：

- **Barrier/Fence**：强制屏障前的所有内存操作完成后，才能开始屏障后的操作
- **原子 CAS**（Compare-and-Swap）：更实用的同步原语
- 代价高昂（数百 cycles），应谨慎使用

### 4.7 编程语言的内存模型

编译器也会重排内存操作（循环不变量外提、死存储消除等），在并行环境下可能产生非等价行为。

**SC for DRF（Sequential Consistency for Data-Race-Free programs）**：
- C++ 和 Java 提供的保证
- 如果程序没有数据竞争，编译器会插入必要的 fence 保证 SC 的表现
- 如果程序有数据竞争，则不提供任何保证

### 4.8 对缓存系统的启示

| 概念 | 缓存系统中的映射 |
|------|-----------------|
| SC | 强一致性缓存（2PC/Paxos） |
| TSO | Cache Aside（最终一致，极端 case 可能读到旧值） |
| Weak Ordering | Write Behind（异步写回，弱一致） |
| Barrier | 缓存失效操作（强制同步点） |
| Data Race | 并发读写缓存和数据库 |

> **核心教训**：使用同步库！它会处理丑陋的重排问题。正如陈皓所说：**宏观系统架构中的很多设计都来源于微观的计算机体系结构。**

## 五、缓存分类与工程实践

### 5.1 缓存特征三要素

| 特征 | 说明 |
|------|------|
| **命中率** | 命中率 = 正确结果数 / 请求缓存次数，核心指标 |
| **最大容量** | 超过后触发清空策略 |
| **清空策略** | FIFO（先进先出）、LFU（最少使用）、LRU（最近最少使用） |

### 5.2 本地缓存 vs 分布式缓存

| 维度 | 本地缓存 | 分布式缓存 |
|------|---------|-----------|
| **代表** | Ehcache、Guava Cache、HashMap | Redis、Memcached |
| **优点** | 无网络开销，极快 | 多应用共享，独立扩展 |
| **缺点** | 进程内，无法跨应用共享 | 网络开销，序列化成本 |
| **适用** | 单机小数据量、配置数据 | 大规模分布式系统 |

### 5.3 Memcached vs Redis

| 维度 | Memcached | Redis |
|------|-----------|-------|
| **数据结构** | 仅 key-value | String/List/Set/Hash/ZSet |
| **持久化** | ❌ | ✅ RDB + AOF |
| **内存管理** | Slab 预分配（chunk 最大 1MB） | 现场申请，有碎片 |
| **分布式** | 客户端一致性 Hash | Redis Cluster（服务端） |
| **线程模型** | 多线程 | 单线程 I/O 多路复用 |
| **事务** | CAS | MULTI/EXEC + WATCH |
| **淘汰策略** | LRU | 6 种策略 |
| **适用** | 简单 KV 高并发 | 复杂数据结构、持久化需求 |

### 5.4 注解式缓存（Spring Cache）

Spring 3.1+ 提供注解缓存，基于 AOP 动态代理：

| 注解 | 作用 |
|------|------|
| `@Cacheable` | 方法结果缓存，命中则不执行方法 |
| `@CachePut` | 每次执行方法并更新缓存 |
| `@CacheEvict` | 清除缓存 |

**注意事项**：
- 类内部调用（this 引用）会导致代理失效，缓存不生效
- 非 public 方法无法使用注解缓存
- key 应使用简单可区分元素（ID/名称），不要用 list 或整体 model
