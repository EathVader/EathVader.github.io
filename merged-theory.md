# 分布式系统理论与实践全面总结

> 本文综合整理自 CAP 理论、分布式八大谬误、分布式锁、延时任务、不可变性原则等多篇经典文献与实践笔记，旨在提供一份去重、深度、系统化的中文参考。

## 一、CAP / BASE 理论

### 1.1 CAP 定理

CAP 定理（Brewer 定理）指出，在一个分布式系统中，以下三个属性最多只能同时满足两个：

| 属性 | 含义 |
|------|------|
| **C（Consistency，一致性）** | 所有节点在同一时刻看到相同的数据，即每次读取都能获得最新写入的值 |
| **A（Availability，可用性）** | 每个请求都能在合理时间内收到非错误的响应（不保证是最新数据） |
| **P（Partition Tolerance，分区容错性）** | 系统在网络分区（节点间通信中断）的情况下仍能继续运作 |

**关键认知纠偏：**

1. **P 是基础，不是可选项。** 在广域网环境下，网络分区是客观事实。Google 通过建设私有网络来最大程度减少分区发生，但不能消除。CAP 之父 Brewer 本人也强调：P 不能和 CA 做 trade-off，真正的 trade-off 在 C 和 A 之间。
2. **"三选二"是误导。** C、A、P 三者并不对等。牺牲 C 或 A 并不能提高 P；P 的提升依赖基础设施的稳定性。
3. **CA 系统并非不可能，但极难实现。** Google Spanner 和阿里 OceanBase 宣称实现了 CA 系统，核心依赖是极高可靠性的网络基础设施。
4. **即使不发生分区，CA 也不是免费午餐。** FLP 不可能性定理表明：在不可靠的分布式系统中无法同时实现安全性（Safety）和活性（Liveness）。CAP 是这一 trade-off 的具体体现。

**CAP 的分解思维：**

- **P 分解：** 延迟 ≠ 故障 ≠ 分区。需要区分物理故障（节点宕机、网络断开）和协议导致的不可用（Paxos 少数派）。故障检测依赖超时机制，而超时设置本身就是 C 和 A 的 trade-off。
- **C 分解：** 一致性不是非黑即白。从强到弱包括：强一致性 → 因果一致性 → 读己所写 → 会话一致性 → 单调读 → 单调写 → 最终一致性。不同业务场景应选择不同一致性模型（如用户名密码需强一致，用户属性可最终一致）。
- **A 分解：** 可用性是连续的（90% ~ 99.9999%），不是二元的。可用性还包含延迟维度——PACELC 理论指出：即使没有分区，也需要在延迟（L）和一致性（C）之间取舍。

### 1.2 PACELC 理论

PACELC 是对 CAP 的扩展：

- **发生分区时（PAC）：** 在 A 和 C 之间取舍
- **未发生分区时（ELC）：** 在 L（延迟）和 C 之间取舍

这更贴近实际系统设计——大多数时候系统并未发生分区，此时核心矛盾是一致性与延迟的平衡。

### 1.3 BASE 理论

BASE 是对 CAP 中 AP 方向的实践总结，与 ACID 形成对比：

| ACID | BASE |
|------|------|
| Atomicity（原子性） | Basically Available（基本可用） |
| Consistency（一致性） | Soft State（软状态） |
| Isolation（隔离性） | Eventually Consistent（最终一致性） |
| Durability（持久性） | — |

**核心思想：** 通过牺牲强一致性来获得可用性，允许数据在一段时间内不一致，但最终达到一致状态。

**补偿机制（C = A + Compensation）：**
- 无论事先还是事后，系统总要保证一致性
- 补偿依赖完善的日志（数据不变性），记录所有分区历史
- 内部补偿（系统自动修复）+ 外部补偿（通知用户、优惠券等）
- 结合社会信用和法律因素，一致性最终都是可以保证的

### 1.4 典型系统分类

| 类型 | 特征 | 代表系统 |
|------|------|----------|
| CP | 牺牲可用性，使用严格法定数协议（Paxos/Raft/ZAB/2PC） | MongoDB, HBase, ZooKeeper |
| AP | 牺牲一致性，使用非严格法定数协议 | CouchDB, Cassandra, Dynamo |
| CA（极难） | 依赖极高可靠性网络基础设施 | Google Spanner, OceanBase |

## 二、分布式八大谬误（Fallacies of Distributed Computing）

由 L. Peter Deutsch 等人在 Sun Microsystems 提出，James Gosling 补充第八条。这是每个分布式系统工程师必须内化的认知：

### 八大谬误列表

| # | 谬误 | 现实 |
|---|------|------|
| 1 | **网络是可靠的** | 网络随时可能中断、丢包、重复。必须为网络错误编写充分的错误处理代码，否则应用会在网络中断时挂起或永久消耗资源 |
| 2 | **延迟为零** | 光速限制了信息传输速度，跨数据中心延迟在 150ms 级别。忽略延迟会导致无限制的流量和大量丢包 |
| 3 | **带宽是无限的** | 带宽有限且共享，忽略带宽限制会导致频分复用介质上的瓶颈 |
| 4 | **网络是安全的** | 恶意用户和程序持续适应安全措施，对网络安全的自满会导致被攻击 |
| 5 | **拓扑不会变化** | 网络拓扑变化影响带宽和延迟，服务器上下线、网络重组是常态 |
| 6 | **只有一个管理员** | 多个管理员（如不同子网的不同公司）可能实施冲突的策略 |
| 7 | **传输成本为零** | 构建和维护网络的隐性成本不可忽略，必须纳入预算 |
| 8 | **网络是同构的** | 假设同构网络会导致前三个谬误的相同问题，现实中网络设备、协议、性能各异 |

### 谬误的实际影响

- **谬误 1 的后果：** 应用在网络中断时可能无限等待，永久消耗内存等资源；恢复后可能无法重试已暂停的操作
- **谬误 2 的后果：** 开发者允许无限制流量，大量丢包浪费带宽
- **谬误 4 的后果：** 被恶意用户和不断进化的恶意软件攻击
- **谬误 7 的后果：** 预算严重不足

### 工程启示

- 设计时必须假设网络不可靠、有延迟、不安全
- 所有远程调用都需要超时、重试、幂等性设计
- 不能把远程交互当作本地对象调用（《A Note on Distributed Computing》的核心论点）
- 分布式系统的故障是部分的（partial failure），比全局故障更难处理

## 三、分布式系统核心挑战

### 3.1 一致性（Consistency）

#### 一致性模型谱系

从强到弱排列：

| 模型 | 描述 |
|------|------|
| **强一致性（Linearizability）** | 更新完成后，所有后续读操作都返回最新值。等价于单副本语义 |
| **顺序一致性（Sequential Consistency）** | 所有进程看到的操作顺序一致，但不要求实时性 |
| **因果一致性（Causal Consistency）** | 有因果关系的操作保序，无因果关系的操作可以乱序 |
| **读己所写（Read Your Writes）** | 进程永远能读到自己最近写入的值 |
| **会话一致性（Session Consistency）** | 在同一会话内保证读己所写 |
| **单调读（Monotonic Reads）** | 一旦读到某个值，后续不会读到更旧的值 |
| **单调写（Monotonic Writes）** | 同一进程的写操作在所有副本上以相同顺序执行 |
| **最终一致性（Eventual Consistency）** | 如果没有新的更新，最终所有副本会收敛到相同的值 |

#### 服务器端一致性实现

- **强一致性：** 所有副本同步更新后才返回（同步复制）
- **为读优化：** 允许从任意副本读取，可能读到旧数据
- **为写优化：** 写入多数派即返回，异步复制到其余节点
- **时间线一致性（PNUTS）：** 副本可能不一致，但保证所有副本以相同顺序应用更新

#### 一致性与性能的 Trade-off

- 强一致性 → 高延迟、低吞吐
- 弱一致性 → 低延迟、高吞吐、但应用需处理不一致
- 实际系统应按业务分解：关键数据用强一致，非关键数据用弱一致

### 3.2 可用性（Availability）

#### 可用性指标

| 可用性 | 每年允许宕机时间 |
|--------|------------------|
| 90%（一个 9） | > 1 个月 |
| 99%（两个 9） | < 4 天 |
| 99.9%（三个 9） | < 9 小时 |
| 99.99%（四个 9） | < 1 小时 |
| 99.999%（五个 9） | ≈ 5 分钟 |
| 99.9999%（六个 9） | ≈ 31 秒 |

#### 容错（Fault Tolerance）

容错不是系统在所有故障下都能工作，而是系统发生故障时能以明确的预定方式运行。

**故障模型层次（从简单到复杂）：**

1. **Crash-Stop（崩溃停止）：** 进程崩溃后永不恢复
2. **Crash-Recovery（崩溃恢复）：** 进程可以崩溃后恢复，通过持久化存储或与其他进程通信恢复状态
3. **Omission（遗漏）：** 进程可能丢失发送或接收的消息
4. **Byzantine（拜占庭）：** 进程可能发送错误信息、冒充其他进程、任意行为

**故障处理策略：**

| 策略 | 描述 | 适用场景 |
|------|------|----------|
| Fail-Fast | 快速失败，立即报错 | 非幂等写操作 |
| Fail-Over | 失效转移到备份节点 | 读操作（推荐） |
| Fail-Safe | 失效安全，直接忽略 | 审计日志等非关键操作 |
| Fail-Back | 故障恢复后自动切回原节点 | 消息通知等 |

#### 部分可用性（Partial Availability）

- 即使部分系统故障，也应返回部分结果
- 搜索系统：设置时间限制，超时返回已有结果
- 私信系统：宁可部分用户不可用，也不丢失消息
- 不同功能的故障域应隔离

### 3.3 分区容错性（Partition Tolerance）

#### 分区的本质

- 分区是网络故障的一种极端形式
- 由于延迟的存在，节点无法获得对方的实时状态
- 故障检测依赖超时，而超时设置本身就是 C 和 A 的 trade-off

#### 分区处理三步骤

1. **探知分区发生：** 通过故障检测算法（心跳 + 超时）
2. **进入分区模式：** 限制某些操作，明确哪些不变性约束可能被违反
3. **启动恢复过程：** 恢复数据一致性，补偿分区期间的错误

#### 分布式系统的核心问题

> "分区两侧是否能够在无通信的情况下继续其操作？" —— Brewer

### 3.4 时间与顺序

#### 为什么时间是难题

- 分布式系统中没有全局时钟
- 光速限制了信息传输速度，"同时性"的概念需要放弃
- 不同节点的本地时钟存在漂移

#### 逻辑时钟

- **Lamport 时钟：** 为事件分配逻辑时间戳，保证因果序（如果 a → b，则 C(a) < C(b)），但反之不成立
- **向量时钟（Vector Clocks）：** 每个进程维护一个向量，可以准确判断因果关系和并发关系
- **区间树时钟（Interval Tree Clocks）：** 向量时钟的改进，支持动态进程加入和退出

#### FLP 不可能性定理

在异步系统中，即使只有一个进程可能崩溃（crash-stop），也不可能实现既总是正确（safety）又总是终止（liveness）的共识算法。

**前提条件：** 异步系统 + crash-stop 故障 + 可靠消息系统 + 无法检测进程死亡

**绕过方式：** 引入故障检测器（如最终完美故障检测器）、引入随机化、引入时钟（如 Paxos 引入时钟保证 liveness）

### 3.5 共识（Consensus）

#### 共识的四个属性

- **终止性（Termination）：** 每个正确的进程最终会做出决定
- **有效性（Validity）：** 决定的值必须是某个进程提出的
- **完整性（Integrity）：** 没有进程会决定两次
- **一致性（Agreement）：** 没有两个正确的进程做出不同的决定

#### 主要共识算法

| 算法 | 特点 |
|------|------|
| **2PC（两阶段提交）** | 简单但阻塞，协调者单点故障 |
| **3PC（三阶段提交）** | 减少阻塞但更复杂 |
| **Paxos** | 理论完备但实现极难 |
| **Raft** | 可理解性优先，等价于 Paxos |
| **ZAB** | ZooKeeper 使用，依赖 Leader + FIFO 消息序 |

### 3.6 Quorum（法定数）

- **多数派 Quorum：** N 个进程中，操作需要 N/2 + 1 个进程确认
- **读写 Quorum：** 写 Quorum（W）+ 读 Quorum（R）> N，保证读写交集非空
- **拜占庭 Quorum：** 需要 > (N + f) / 2 个进程，其中 f 是可容忍的拜占庭故障数

### 3.7 复制（Replication）

#### 复制类型

| 维度 | 类型 | 特点 |
|------|------|------|
| 处理方式 | 主动复制（状态机复制） | 所有副本处理所有请求，要求确定性 |
| | 被动复制（主从复制） | 主节点处理请求后同步到从节点 |
| 同步方式 | 同步复制 | 零数据丢失，但延迟高 |
| | 异步复制 | 性能好，但可能丢失最近数据 |
| 组织关系 | Master/Slave | 最常见 |
| | Paxos 复制 | 可呈现外部一致性 |
| | 链式复制 | 高吞吐 |

#### 复制是问题的根源也是解决方案

> "To replication! The cause of, and solution to all of life's problems." —— 改编自 Homer Simpson

- 复制提高性能（更多计算能力和带宽）和可用性（更多副本）
- 但复制引入一致性问题——独立副本必须保持同步

## 四、分布式锁（Redis / ZooKeeper / DB）

### 4.1 锁的基础

在多线程世界里，对共享资源的并发访问保护最直接的办法就是引入锁。锁本质上是将并发请求串行化。

#### 单机锁类型

| 锁类型 | 特点 |
|--------|------|
| **自旋锁（Spin Lock）** | 非阻塞，循环等待，适合短期持锁、多核场景。基于 CAS 原子操作实现 |
| **互斥锁（Mutex Lock）** | 阻塞锁，获取失败则线程挂起。竞争激烈时性能优于自旋锁 |
| **可重入锁（Reentrant Lock）** | 同一线程可多次获取，不会死锁。Java 的 synchronized 和 ReentrantLock 都是可重入锁 |
| **轻量级锁 / 偏向锁** | Java 6 引入的锁优化，减少获取/释放锁的性能消耗。锁可升级（偏向 → 轻量级 → 重量级）但不能降级 |

#### 自旋锁 vs 互斥锁选择

- 多核 + 预计等待时间短（< 两次上下文切换）→ 自旋锁
- 多核 + 预计等待时间长 → 互斥锁
- 单核 → 不要使用自旋锁

#### 锁的后遗症

死锁、饥饿、活锁、Lock Convoying（多个同优先级线程重复竞争同一把锁导致频繁调度切换）、优先级反转、不公平和低效率。

### 4.2 分布式锁的需求

分布式锁是为分布式应用各节点对共享资源的排他式访问而设定的锁。典型场景：全局发号器、分布式事务、库存扣减等。

### 4.3 分布式锁设计要点

1. **锁的时效：** 避免单点故障造成死锁，但也要保证持锁客户端可用时不被其他客户端解锁
2. **持锁期间的 check：** 在关键节点检查锁状态，设计为可重入锁
3. **减少获取锁的操作：** 让客户端有等待时间，而非不断循环申请
4. **锁粒度尽量小：** 减少其他客户端等待时间
5. **解锁通知：** 持锁客户端解锁后通知其他等待节点（类似 notifyAll）
6. **异常处理：** 不能因为一个节点解锁失败影响整个等待队列
7. **降级方案：** Redis 宕机时用单机锁限流 + 数据库持久化锁做最终一致性

### 4.4 基于 Redis 的分布式锁

#### 基本实现

- **加锁：** `SET key value NX PX timeout`（原子操作，设置键值 + 过期时间）
- **解锁：** 通过 Lua 脚本原子地检查 value 并 DEL（防止误删其他客户端的锁）

#### 核心问题

| 问题 | 解决方案 |
|------|----------|
| 客户端宕机导致死锁 | 设置过期时间（TTL） |
| 时钟漂移导致误删 | value 中包含唯一标识（如 UUID），解锁时校验 |
| 业务执行超过 TTL | 看门狗机制自动续期（Redisson 实现） |
| 主从切换导致锁丢失 | Redlock 算法（向 N 个独立 Redis 实例申请锁，多数成功才算获取） |

#### Redlock 算法

Redis 作者 Antirez 提出，向 N（通常 5）个独立 Redis 实例申请锁，多数（N/2+1）成功且总耗时 < TTL 才算获取成功。

> **争议：** Martin Kleppmann 在《How to do distributed locking》中质疑 Redlock 的正确性（GC 暂停、时钟跳跃等场景），Antirez 在《Is Redlock safe?》中回应。核心分歧在于是否需要 fencing token 来保证安全性。

### 4.5 基于 ZooKeeper 的分布式锁

#### 实现原理

1. 在锁路径下创建临时顺序节点（EPHEMERAL_SEQUENTIAL）
2. 获取锁路径下所有子节点，判断自己是否是最小节点
3. 如果是最小节点，获取锁成功
4. 如果不是，监听（watch）前一个节点的删除事件
5. 前一个节点删除后，重新判断自己是否是最小节点

#### 优势

- 临时节点：客户端断开连接后自动删除，天然避免死锁
- 顺序节点：天然实现公平锁
- Watch 机制：避免惊群效应（只通知下一个等待者）

#### 劣势

- 性能低于 Redis（ZK 的写操作需要多数派确认）
- 依赖 ZK 集群的可用性

### 4.6 基于数据库的分布式锁

#### 实现方式

1. **唯一索引/唯一约束：** INSERT 成功即获取锁，DELETE 释放锁
2. **排他锁（FOR UPDATE）：** SELECT ... FOR UPDATE 获取行锁
3. **乐观锁（version）：** UPDATE ... WHERE version = ? 通过版本号控制

#### 优劣

- 优点：实现简单，不依赖额外组件
- 缺点：性能差，不适合高并发；需要自行处理超时和死锁；数据库成为瓶颈

### 4.7 三种方案对比

| 维度 | Redis | ZooKeeper | 数据库 |
|------|-------|-----------|--------|
| 性能 | 高 | 中 | 低 |
| 可靠性 | 中（主从切换可能丢锁） | 高（ZAB 协议保证） | 高（事务保证） |
| 实现复杂度 | 中 | 中 | 低 |
| 公平性 | 默认不公平 | 天然公平（顺序节点） | 取决于实现 |
| 死锁处理 | TTL 过期 | 临时节点自动删除 | 需要额外机制 |

### 4.8 CAP 视角下的锁设计

在 CAP 范围内，分布式锁需要满足：

1. 锁对象信息写入多副本以应对故障
2. 不同对象的锁信息需要分布化和负载均衡
3. 锁信息写入持久化存储系统

## 五、延时任务方案

延时任务与定时任务的区别：延时任务没有明确触发时间和执行周期，在某事件触发后一段时间内执行，通常是单个任务。

### 5.1 方案对比

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **数据库轮询** | 定时线程扫描数据库，按时间判断超时 | 简单易行，支持集群 | 内存消耗大，存在延迟（最坏=扫描间隔），大数据量时数据库损耗极大 |
| **JDK DelayQueue** | 无界阻塞队列，延迟期满才能取出元素 | 效率高，触发延迟低 | 数据在内存中（宕机丢失），集群扩展难，大量任务易 OOM |
| **时间轮算法** | 类比时钟，指针按固定频率轮动，任务放在对应槽位 | 效率高，延迟比 DelayQueue 更低，代码复杂度较低 | 同 DelayQueue（内存、宕机、OOM） |
| **Redis ZSet** | 用 score 存超时时间戳，member 存订单号，扫描第一个元素判断超时 | 消息持久化在 Redis，集群扩展方便，时间准确度高 | 需要额外维护 Redis |
| **Redis Keyspace Notifications** | 利用 key 过期回调机制 | 实现简单 | pub/sub 是 fire-and-forget，客户端断连期间事件丢失，不可靠 |
| **RabbitMQ 延时队列** | 利用 TTL + Dead Letter Exchange 实现 | 高效，支持持久化，易横向扩展 | 依赖 RabbitMQ 运维，复杂度和成本高 |

### 5.2 Redis ZSet 方案详解

```
# 生产者：将订单放入 ZSet，score 为超时时间戳
ZADD OrderId <timeout_timestamp> <order_id>

# 消费者：取第一个元素，判断是否超时
ZRANGEBYSCORE OrderId 0 <current_timestamp> LIMIT 0 1

# 超时则移除并处理
ZREM OrderId <order_id>
```

**并发安全：** 多消费者场景下，通过 ZREM 的返回值判断（> 0 才消费），避免重复消费。

### 5.3 时间轮算法详解

- **核心参数：** ticksPerWheel（一轮的 tick 数）、tickDuration（一个 tick 的持续时间）、timeUnit（时间单位）
- **任务放置：** 当前指针在 1，任务需要 4 秒后执行 → 放在槽位 5
- **超过一轮：** 需要 20 秒，槽数为 8 → 放在 2 圈后的槽位 5（20 % 8 + 1）
- **实现：** Netty 的 HashedWheelTimer、Kafka 的时间轮

## 六、不可变性原则（Immutability Changes Everything）

### 6.1 核心思想

> "会计不用橡皮擦，否则他们会坐牢。"

不可变数据一旦写入就永不改变。修正只能通过追加新记录来实现。这是分布式系统中协调远距离节点的关键手段。

### 6.2 为什么现在可以拥抱不可变性

- **存储越来越便宜：** 每 TB 磁盘成本持续下降，可以长期保存大量数据
- **计算越来越便宜：** 多核处理器使计算成本降低
- **协调越来越昂贵：** 锁的延迟在多核环境下损失大量指令执行机会

### 6.3 不可变性的应用层次

不可变性贯穿计算栈的每一层：

| 层次 | 应用 |
|------|------|
| **应用层** | Append-only 计算、事件溯源、账本式记录 |
| **数据库层** | 事务日志（WAL）是真相，数据库是日志的缓存；MVCC 多版本并发控制 |
| **文件系统层** | LSM Tree（Log-Structured Merge Tree）、COW（Copy-on-Write）、日志结构文件系统 |
| **分布式文件系统** | GFS/HDFS 的不可变文件和不可变块，简化复制和一致性 |
| **大数据层** | MapReduce/Dryad 基于不可变输入的函数式计算，天然支持失败重试（幂等性） |
| **硬件层** | SSD 磨损均衡（Wear Leveling）、叠瓦式磁盘（Shingled Disk） |

### 6.4 内部数据 vs 外部数据

| 维度 | 内部数据（Data on the Inside） | 外部数据（Data on the Outside） |
|------|------|------|
| 管理方式 | 关系数据库 + 事务 | 消息、文件、文档、网页 |
| 可变性 | 可变（事务更新） | 不可变（一旦发出不再改变） |
| 锁定 | 数据库锁保护 | 无锁（已提取的副本） |
| 标识 | 数据库内部 ID | 唯一标识（如 URL） |
| 版本 | 事务版本 | 新版本 = 新唯一标识 |

### 6.5 不可变数据集的特性

- **语义不可变：** 内容不变，但可以物理优化（加索引、反规范化、列存储、分区）
- **描述性元数据：** 创建时描述数据的形态，不能事后修改语义（区别于 SQL DDL 的规定性元数据）
- **反规范化是合理的：** 不可变数据不需要规范化（规范化是为了消除更新异常，不可变数据没有更新）
- **可与关系数据库 JOIN：** 不可变数据集可以和数据库快照进行关系运算

### 6.6 不可变性如何打败 CAP 定理

> "如果你选择可用性而不是一致性，系统可以达到最终一致性并规避所有最终一致性带来的复杂问题。"

- 不可变数据没有更新操作，不存在副本不一致的问题
- 不需要版本化数据、向量时钟或读取修复
- 数据只有"存在"或"不存在"两种状态
- 在一致性哈希环中存储不可变数据，可以避免最终一致性的应用层复杂性

### 6.7 LSM Tree 与 COW

- **LSM Tree：** 在不可变文件之上呈现可变的假象。写入先进入内存（MemTable），然后刷写为不可变的 Level 0 文件，逐级合并。读取需要搜索每一级
- **COW（Copy-on-Write）：** 更新时不修改原数据，而是写入新的不可变副本。日志和 LSM 树都是 COW 的体现
- **写放大 vs 读放大：** 更积极的合并减少读成本但增加写成本，需要根据读写比例权衡

## 七、分布式系统设计原则

### 7.1 基础设计原则

1. **为故障而设计（Design for Failure）：** 分布式系统的本质区别在于故障概率高且是部分故障。部分故障比全局故障更难推理
2. **通过复制提高可用性：** 这是分布式系统设计的首要原则
3. **分区和复制是两大基本技术：** 分区提高并行处理能力，复制提高可用性和降低延迟
4. **协调是非常困难的：** 尽可能避免机器间的协调（水平扩展的真正诀窍是独立性）
5. **用 CAP 定理批判系统设计：** CAP 不是用来构建系统的，而是用来审视和改进设计的

### 7.2 分布式程序设计实践原则

| 原则 | 说明 |
|------|------|
| **幂等性** | 一次和多次请求应具有相同副作用。实现方式：天然幂等（查询）、关键字段去重（订单号）、全局 requestId |
| **接口安全** | 分配 appCode + 非对称加密签名，确保调用方身份可识别 |
| **统一返回格式** | status（调用状态）+ errorCode（业务错误码）+ message（提示）+ data（业务数据） |
| **日期格式统一** | 统一使用可读格式或 long 型时间戳，统一精度（秒/毫秒） |
| **小数传输** | 浮点数以字符串传输，避免精度丢失 |
| **数据字典** | 统一属性命名和含义，避免同义不同名或同名不同义 |

### 7.3 工程实践建议

#### 指标与监控

- **指标是唯一的工作方式：** 暴露延迟百分位、计数器、变化率等指标，是跨越"你认为系统在做什么"和"系统实际在做什么"之间鸿沟的唯一方式
- **使用百分位而非平均值：** 分布式系统的延迟几乎不遵循正态分布，平均值会误导决策。默认使用 P50、P99、P99.9、P99.99
- **日志会说谎：** 少数错误类型可能占据日志大部分空间但实际发生率很低。优先看指标，再用日志验证

#### 背压（Backpressure）

- 在整个系统中实现背压机制
- 过载或故障时丢弃新消息或返回错误（并记录指标）
- 对其他系统的连接和请求使用超时和指数退避
- 没有背压机制，级联故障或意外消息丢失将不可避免

#### Feature Flags

- 用 Feature Flag 逐步发布基础设施变更，而非"大爆炸"式切换
- 分步骤：并行写入 → 后台读取验证 → 对比检查 → 逐步切流
- 接受多版本基础设施和数据并存是常态，用局部复杂性换取全局简单性和弹性

#### 容量估算

- 学会信封背面计算：一台机器 24GB 内存，减去 OS 和请求处理开销，tweet ID 8 字节，能存多少？
- 熟记 Jeff Dean 的"每个人都应该知道的数字"

#### 服务提取

- 当代码变更在独立服务中更容易时，提取服务
- 服务提供封装、独立部署、更低的协调成本
- 典型场景：隐藏即将变更的存储层

#### ID 空间设计

- 获取数据所需的 ID 越多，分区选择越多
- 获取数据所需的 ID 越少，消费越容易
- 注意 ID 中隐含的信息（自增 ID 可被爬取、去匿名化）

#### 数据局部性

- 处理和缓存尽量靠近持久化存储
- 局部性包括空间局部性和时间局部性（相近时间的相同请求可以合并）

#### 不要将缓存数据写回持久化存储

- 这是一个常见但严重的错误，会导致数据神秘回退到旧值

### 7.4 分布式系统的真正难题

1. **是否允许任意节点并发可写：** Google F1、OceanBase、Aurora 都指定写节点
2. **读写并发：** 多版本数据存储 + 分布式事务，B 进程可无锁快照读取
3. **元数据保存：** 如何获取数据版本和时间戳？Google TrueTime 用物理时间代替逻辑时钟
4. **读写事务期间节点故障：** 包括 leader、参与者、用户节点的任意故障
5. **嵌套/链式事务：** 电子商务的基础，如 Percolator 的两阶段提交实现
