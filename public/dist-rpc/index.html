1 RPC 核心原理

什么是 RPC

RPC（Remote Procedure Call）即远程过程调用，核心目标：让调用方感觉就像调用本地函数一样调用远端函数。在 OSI 模型中，RPC 跨越了传输层和应用层，基于请求-应答模式，是跨语言的通信标准。

RPC 的分层架构

层次职责示例
服务层（Service）RPC 接口定义与实现IDL 定义、业务逻辑
协议层（Protocol）RPC 报文格式和数据编码PB、Thrift Binary/Compact
传输层（Transport）实现底层通信TCP、HTTP/2

可靠性保障

  • 网络闪断重连、重传：自动恢复通信
  • 连接超时控制：避免无限等待
  • Netty 等框架封装了这些复杂性

2 I/O 模型与线程模型

四种 I/O 模型

模型特点适用场景
阻塞 I/O（BIO）线程被 I/O 阻塞，CPU 空闲低并发
非阻塞 I/O（NIO）轮询检查 I/O 状态中等并发
I/O 多路复用一个线程监听多个 Socket（epoll/kqueue）高并发，工业界主流
异步 I/O（AIO）内核完成 I/O 后回调通知Windows IOCP
💡 Reactor vs Proactor

I/O 多路复用 → Reactor 模式：epoll + 事件驱动，Redis、Nginx、Node.js、Netty 均采用此模式。
异步 I/O → Proactor 模式：在 Windows 上大放异彩（IOCP）。

🔍 Protocol Buffers 为何高效

使用字段序号作为标识（而非包名类名），极其紧凑；使用 varint 和 zigzag 对整型做特殊编码。同一个 Person 对象：PB 序列化仅 33 字节，远小于 JSON/XML。

场景化建议

场景推荐方案理由
企业内部 Java 微服务Dubbox服务治理完善，可集成多种序列化协议
跨语言微服务gRPCHTTP/2 友好，PB 高效，生态活跃
极致性能（大规模内部通信）Thrift序列化性能最优，TNonblockingServer 高效
新项目、云原生gRPC与 K8s/Istio 等云原生生态契合

关键原则

📌 五大选型原则
  • 接口先行，语言中立：服务提供者和消费者解耦，并行开发
  • 契约驱动开发：基于 IDL 定义接口，Mock 测试桩并行验证
  • API 版本兼容性优先：使用字段编号机制保证前向兼容
  • 循序渐进：微服务拆分粒度随业务演进逐步细化
  • 综合考量：不仅看性能，还要看团队熟悉度、社区活跃度、运维成本
