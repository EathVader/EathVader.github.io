数据定位方式

方式原理优点缺点代表
Hash 计算按数据特征计算 hash无需元信息，任何节点可计算增减节点引起数据迁移一致性 Hash
元数据查询存储映射关系，查询定位灵活，可精确控制元数据服务器可能成为瓶颈GFS/HDFS

一致性模型

模型定义适用
强一致性更新成功后，所有读操作立即获得最新数据金融、交易
弱一致性更新后需要一段时间才能读到最新数据一般业务
最终一致性经过一个时间窗口后保证一致对象存储、CDN

2 两种核心架构模式

模式一：有中心节点 + 存储节点有主从

代表：GFS、HDFS、小米 Pegasus

Client
→
Master（调度）
→
StorageNode（主）
→
StorageNode（从）

Master 节点职责

  • 存储全局视图：集群拓扑（存储池→分区→服务器→磁盘→目录）
  • 读写调度：根据节点状态和权重选择合适节点
  • 存储节点选主：为每个备份组选择主从节点
  • 数据恢复调度：维持副本数量和内容正确性
⚠️ 性能瓶颈

大文件场景：主节点出口网卡（3 副本时出口流量 = 入口 × 2）→ 解决方案：以 Partition 为单位做副本，每个节点都有主从角色，流量均衡。

模式二：无中心节点 + 存储节点无主从

代表：Swift、Ceph

Client/Proxy
→
Ring（一致性 Hash）
→
StorageNode（对等）

采用 Quorum 协议（R+W>N）。Swift 默认 N=3, W=2, R=1（弱一致）。

Swift StorageNode 组件

组件职责
Object Server单机存储引擎，文件内容存为文件，元数据存为文件属性
Object Replicator保证副本数量和位置正确性
Object Auditor检测比特级错误，隔离损坏文件
Object Updater异步更新元数据（高负载时延迟处理）

存储引擎：合并 vs 不合并

方式优点缺点代表
不合并实现简单小文件读写受磁盘随机 I/O 制约Swift
合并随机写→顺序写，小文件性能大幅提升需要二次映射，实现复杂GFS、自研系统

4 Ceph 运维实战

扩容的"坑"

⚠️ 坑一：扩容粒度受容错域制约

Ceph 一次只能扩一个"容错域"。同时扩 2 台机器时，某些 PG 可能把 3 副本中的 2 个映射到新机器（无完整数据），导致 PG 停止服务。
解决方案：初始规划时设定更大层次的容错域（如逻辑 Rack），扩容时扩一整个 Rack。

⚠️ 坑二：Crushmap 变更引发连锁反应

扩容进行中如果有硬盘损坏 → crushmap 改变 → PG 重新 re-hash → 大量 PG 位置重新计算。1000 台规模集群一天坏 2-3 块盘很正常，crushmap 频繁变动 → 整体 I/O 下降。

⚠️ 坑三：PG 数量需随规模调整

社区建议 ~200 PG/OSD。集群从 10 台扩到 1000 台时，PG 数量不足会放大不平衡。调 PG 代价极大：约 50% 的对象需要重新放置物理位置。

集群可靠利用率问题

Ceph 集群整体使用率达到 70% 后就可能进入不稳定状态。

磁盘变满
→
管理员调 reweight
→
crushmap 变更
→
更多磁盘被写满
→
恶性循环
📊 利用率对比

HDFS：95% 仍可服务 | EMC Isilon：98-99% 仍可服务 | Ceph：50% 就要开始准备扩容

预估 50PB 存储需要 300 台机器，但必须提前采购 200-300 台备用机器。

运维建议

  • 积累运维文档，踩过的坑不要踩第二次
  • 新手容易犯错导致事故，需要技术积累管理
  • Ceph 对象没有 last_access_time，冷热数据区分需要二次开发
  • Red Hat 版本的 Ceph 文档逻辑感更强，推荐参考

5 选型建议

中心化 vs 去中心化

维度中心化（GFS/HDFS）去中心化（Ceph/Swift）
扩容✅ 几乎无限制，平滑⚠️ 受容错域制约
利用率✅ 95% 仍可服务❌ 50-70% 需扩容
单点❌ NameNode 单点（可 HA）✅ 无单点
数据放置✅ 精确控制⚠️ 伪随机，方差不可控
运维✅ 相对简单❌ 复杂，需强团队
💡 没有银弹，只有 tradeoff

去中心化消除了单点问题，但带来了扩容不平滑、利用率低、运维复杂等代价。

场景化选型

场景推荐理由
大数据分析HDFS生态完善，大文件顺序读写优秀
云平台块存储Ceph RBD统一存储，与 OpenStack 深度集成
对象存储（海量小文件）Ceph RGW / SwiftS3 兼容接口，水平扩展
文件+块+对象统一Ceph唯一支持三种存储类型的开源方案
团队运维能力有限HDFS运维相对简单，社区成熟
