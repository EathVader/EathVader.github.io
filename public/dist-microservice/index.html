1 微服务核心概念

单体 vs 微服务

维度单体架构微服务架构
开发效率所有人改同一代码库，冲突多各团队独立开发，互不干扰
部署任何修改需重新构建整个项目单个服务独立部署
扩展性整体扩展，资源浪费按需扩展单个服务
稳定性一个小问题可能导致整体崩溃故障隔离，影响范围可控
技术栈统一技术栈可异构，按需选择

解决复杂性

CI/CD 友好

微服务的挑战

  • 服务划分难度：按什么规则划分是经验工程，过早拆分代价高
  • 分布式复杂性：服务发现、调用链跟踪、质量监控变得棘手
  • 分布式事务：CAP 约束下需放弃强一致性，转向最终一致性
  • 测试复杂度：需启动所有依赖服务
  • 运维复杂度：需要完善的基础设施支撑

2 服务拆分策略

核心原则

高内聚
相关问题放在同一服务
→
松耦合
修改一个不影响另一个

拆分建议

⚠️ 不要过早拆分

将已有代码库划分成微服务比从头建设微服务要简单得多。先构建单体应用理解业务域，再逐步拆分。微服务是达到目的的手段，而不是目标。

  • 使用 DDD 中的限定上下文（Bounded Context）方法拆分
  • 识别业务域：财务、仓库、用户、订单等
  • 找到接缝：理清代码库中的模块边界，沿接缝拆分
  • 服务大小无统一标准

3 服务治理

服务治理技术演进

阶段特点
第一代：SOA Governance面向企业 IT 异构服务，聚焦全生命周期管理，缺少运行时动态治理
第二代：分布式服务框架面向互联网业务，聚焦线上运行态细粒度治理（如 Dubbo）
第三代：云端微服务治理服务微自治，利用云调度弹性，逐渐消除人工治理

策略说明

一致性 Hash相同参数总发到同一提供者
粘滞连接有状态服务，尽可能连同一提供者

流量控制

静态流控
基于 SLA 的 QPS
动态流控
基于 CPU/内存负载
并发控制
限制并发执行数
连接控制

集群容错策略

策略说明
Failover失败自动切换，重新选路查找下一个可用提供者
Failback消费者获取失败详情，自行决定后续策略
Failcache失败后缓存消息，等待周期 T 后重试
Failfast只调用一次，失败不重试（高峰期非核心服务）
🔍 链路追踪与监控

通过 Flume 等框架实时收集服务调用链日志和性能 KPI 数据，实现故障的自动发现、自动分析。常用工具：Zipkin、Spring Cloud Sleuth、Prometheus、ELK。

选型建议

📌 选型指南
  • 需要多数据中心支持 → Consul
  • 需要快速支持多语言 → Consul（HTTP/DNS 接口）
  • 已有 Dubbo 生态 → ZooKeeper
  • 使用 Kubernetes → etcd（K8s 原生集成）

5 配置中心

为什么需要配置中心

⚠️ "配置私藏"问题

当下游服务的地址配置分散在各个上游服务的配置文件中时，每份配置都是冗余数据。一旦下游扩容/缩容，所有上游都需要修改配置并重启。

工作原理

配置中心统一维护
→
上游订阅配置
→
下游变更时统一修改
→
通知所有订阅方

开源方案对比

方案特点
Spring Cloud ConfigGit 存储库支持，直接映射到 Spring Environment
Apollo（携程）功能完善，支持多环境、灰度发布、权限管理
Nacos（阿里）集服务发现与配置管理于一体
Consul KVConsul 自带 KV 存储
etcd轻量级 KV 存储，Kubernetes 原生使用

配置中心 vs MQ 解耦

维度配置中心MQ
逻辑解耦✅✅
物理解耦❌（上下游仍建立物理连接）✅
适用场景需要关注下游处理结果（RPC）不关注下游处理结果（异步）

6 事件驱动与遗留改造

编排 vs 协作

模式说明优点缺点
编排中心服务协调整个流程流程清晰可见中心服务可能成为瓶颈
协作各服务通过事件自主响应高度解耦流程分散，难以调试
✅ 最佳实践

在微服务架构中需要平衡编排和协作，不能一味追求事件驱动而忽视流程可见性。将复杂业务逻辑提取到专用服务中，使用轻量级状态机简化长时间运行服务的实现。

遗留系统改造步骤

识别业务域
→
评估依赖关系
→
建立基础设施
→
逐步迁移
→
灰度验证
  • 不要一步到位：渐进式改造，先识别业务域再逐步拆分
  • 先单体后微服务：先在单体中理解业务
  • 新功能以微服务方式开发，旧模块逐步抽取
  • 使用防腐层（Anti-Corruption Layer）隔离新旧系统

下一代：Service Mesh

🌐 核心理念

微服务间的 TCP/IP，负责服务间网络调用、限流、熔断和监控，应用程序无感知。架构分为数据面板（Data Plane）和控制面板（Control Plane）。

组件定位
Linkerd面向服务通信的网络代理，负载均衡、熔断、服务发现
EnvoyL7 代理和通信总线，支持 HTTP/2、gRPC、动态配置
Istio更高层抽象，流量管理、可观察性、策略执行、安全
