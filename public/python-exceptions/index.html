1为什么需要异常处理？

Python 使用 EAFP（Easier to Ask for Forgiveness than Permission）哲学，即"先做，出错再处理"，而不是 LBYL（Look Before You Leap）。

Python
# LBYL 风格（不推荐）
if os.path.exists(filename):
    with open(filename) as f:
        data = f.read()

# EAFP 风格（推荐）
try:
    with open(filename) as f:
        data = f.read()
except FileNotFoundError:
    print(f"文件 {filename} 不存在")
💡 EAFP vs LBYL
  • EAFP：直接尝试操作，捕获异常处理错误（Python 风格）
  • LBYL：先检查条件，再执行操作（防御性编程）
  • Python 官方推荐 EAFP，因为更简洁、更高效（避免重复检查）

2方案一：巨型 Try-Except

把所有代码放在一个巨大的 try 块中，用一个 except 捕获所有异常：

Python
def process_data(filepath):
    try:
        # 读取文件
        with open(filepath) as f:
            data = f.read()
        # 解析 JSON
        parsed = json.loads(data)
        # 处理数据
        result = transform(parsed)
        # 写入结果
        with open('output.json', 'w') as f:
            json.dump(result, f)
        print("处理成功")
    except Exception as e:
        print(f"出错了: {e}")
优点缺点
代码简洁，一个 try 块搞定无法区分具体是哪一步出错
不会遗漏任何异常可能掩盖不应该捕获的异常（如 KeyboardInterrupt）
适合快速原型调试困难，错误信息不明确
🚨 不推荐

捕获 Exception 过于宽泛，会掩盖所有异常，包括 KeyboardInterrupt 和 SystemExit。无法针对不同错误采取不同处理策略。

3方案二：细粒度异常处理

为每个可能出错的操作分别添加 try-except：

Python
def process_data(filepath):
    try:
        with open(filepath) as f:
            data = f.read()
    except FileNotFoundError:
        print(f"文件不存在: {filepath}")
        return
    except PermissionError:
        print(f"无权限读取: {filepath}")
        return

    try:
        parsed = json.loads(data)
    except json.JSONDecodeError as e:
        print(f"JSON 解析失败: {e}")
        return

    try:
        result = transform(parsed)
    except ValueError as e:
        print(f"数据转换失败: {e}")
        return

    try:
        with open('output.json', 'w') as f:
            json.dump(result, f)
    except IOError as e:
        print(f"写入文件失败: {e}")
        return

    print("处理成功")
优点缺点
精确捕获每种异常代码冗长，重复的 return 语句
针对不同错误有不同处理正常逻辑被异常处理代码打断
调试友好难以维护，添加新步骤需要加新的 try-except

4方案三：嵌套异常处理

将 try-except 嵌套，只有上一步成功才执行下一步：

Python
def process_data(filepath):
    try:
        with open(filepath) as f:
            data = f.read()
        try:
            parsed = json.loads(data)
            try:
                result = transform(parsed)
                try:
                    with open('output.json', 'w') as f:
                        json.dump(result, f)
                    print("处理成功")
                except IOError as e:
                    print(f"写入失败: {e}")
            except ValueError as e:
                print(f"转换失败: {e}")
        except json.JSONDecodeError as e:
            print(f"解析失败: {e}")
    except FileNotFoundError:
        print(f"文件不存在: {filepath}")
优点缺点
逻辑清晰，依赖关系明确缩进层级过深（箭头反模式）
精确控制错误处理代码可读性差，难以维护
不会执行后续步骤超过 3 层嵌套后几乎无法阅读
⚠️ 箭头反模式（Arrow Anti-Pattern）

嵌套过深导致代码形成"箭头"形状，可读性极差。一般建议嵌套不超过 3 层。

5方案四：While True 模式

使用 while True 循环，成功时 break，失败时 continue 或 return：

Python
def process_data(filepath):
    while True:
        try:
            with open(filepath) as f:
                data = f.read()
            parsed = json.loads(data)
            result = transform(parsed)
            with open('output.json', 'w') as f:
                json.dump(result, f)
            print("处理成功")
            break
        except FileNotFoundError:
            print(f"文件不存在: {filepath}")
            return
        except json.JSONDecodeError as e:
            print(f"解析失败: {e}")
            return
        except ValueError as e:
            print(f"转换失败: {e}")
            return
        except IOError as e:
            print(f"写入失败: {e}")
            return
优点缺点
避免嵌套，扁平化代码while True 看起来像无限循环，容易误解
所有异常处理集中在一起break 和 return 混用，控制流不清晰
适合需要重试的场景不适合不需要重试的场景

6方案五：自定义异常

定义业务异常类，将底层异常转换为业务异常：

Python
class DataProcessingError(Exception):
    """数据处理异常基类"""
    pass

class FileReadError(DataProcessingError):
    pass

class ParseError(DataProcessingError):
    pass

class TransformError(DataProcessingError):
    pass

class WriteError(DataProcessingError):
    pass

def process_data(filepath):
    try:
        try:
            with open(filepath) as f:
                data = f.read()
        except FileNotFoundError as e:
            raise FileReadError(f"文件不存在: {filepath}") from e
        except PermissionError as e:
            raise FileReadError(f"无权限: {filepath}") from e

        try:
            parsed = json.loads(data)
        except json.JSONDecodeError as e:
            raise ParseError(f"JSON 解析失败") from e

        try:
            result = transform(parsed)
        except ValueError as e:
            raise TransformError(f"数据转换失败") from e

        try:
            with open('output.json', 'w') as f:
                json.dump(result, f)
        except IOError as e:
            raise WriteError("写入文件失败") from e

    except DataProcessingError as e:
        print(f"处理失败 [{type(e).__name__}]: {e}")
        raise
优点缺点
异常层次清晰，便于分类处理需要定义大量异常类
保留原始异常链（from e）代码量增加
调用方可以针对性捕获过度设计，小项目不适用

7方案六：封装函数（推荐）

将每个可能出错的操作封装成独立函数，每个函数只处理自己的异常：

Python
def read_file(filepath):
    """读取文件内容"""
    try:
        with open(filepath) as f:
            return f.read()
    except FileNotFoundError:
        raise ValueError(f"文件不存在: {filepath}")
    except PermissionError:
        raise PermissionError(f"无权限读取: {filepath}")

def parse_json(data):
    """解析 JSON 数据"""
    try:
        return json.loads(data)
    except json.JSONDecodeError as e:
        raise ValueError(f"JSON 解析失败: {e}")

def write_json(filepath, data):
    """写入 JSON 文件"""
    try:
        with open(filepath, 'w') as f:
            json.dump(data, f)
    except IOError as e:
        raise IOError(f"写入文件失败: {filepath}") from e

def process_data(filepath):
    """主处理函数：清晰、简洁、易读"""
    data = read_file(filepath)
    parsed = parse_json(data)
    result = transform(parsed)
    write_json('output.json', result)
    print("处理成功")
✅ 为什么推荐？
  • 单一职责：每个函数只做一件事，异常处理也集中在一处
  • 可读性强：主函数 process_data 像伪代码一样清晰
  • 可测试：每个函数可以独立测试
  • 可复用：read_file、parse_json 可在其他地方复用
  • 易于扩展：添加新步骤只需添加新函数
  • 异常链完整：使用 raise ... from e 保留原始异常信息

9总结

✅ 最佳实践要点
  • 使用 EAFP 风格：先尝试，再捕获异常
  • 捕获具体异常，不要裸用 except: 或 except Exception:
  • 使用 raise ... from e 保留异常链
  • 将异常处理封装到独立函数中，保持主逻辑清晰
  • 自定义异常用于大型项目，小项目用内置异常即可
  • 使用 contextlib 管理资源（如 with 语句）
❌ 巨型 try
过于宽泛

⚠️ 嵌套/细粒度
代码冗长
