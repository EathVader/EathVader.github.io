iOS/Android 网络抓包教程：tcpdump

来源：MrPeak 技术分享

为什么需要抓包？

现在的移动端应用几乎都会通过网络请求来和服务器交互，通过抓包来诊断和网络相关的 bug 是程序员的重要技能之一。

抓包工具对比

工具 协议 特点
Charles HTTP/HTTPS 设置代理，GUI 友好
Wireshark 所有协议 GUI 分析，深入全面
tcpdump 所有协议 命令行，快速高效

iOS 上启动 tcpdump

iOS 设备上启动 tcpdump 比较方便。Apple 在 Mac 上有个叫 rvictl 的程序，可以通过 iOS 设备的 UDID 创建一个虚拟网卡，然后通过这个虚拟网卡监听设备上所有的网络流量。

1

获取设备 UDID

通过 iTunes 或 Finder 获取 iOS 设备的 UDID

2

创建虚拟网卡

rvictl -s <udid>

在终端输入命令，创建虚拟网卡（rvi0）

3

启动 tcpdump 监控

sudo tcpdump -i rvi0 -AAl

启动 tcpdump 监控设备所有网络流量

Android 上启动 tcpdump

Android 设备没办法通过 rvictl 创建虚拟网卡，但是可以把 tcpdump 的可执行文件上传到 Android 设备上，然后通过 Mac 远程登录 Android 设备运行 tcpdump，前提是这台 Android 设备必须已经 root 过。

1

下载 Android 版 tcpdump

从 androidtcpdump.com 下载专为 Android 编译的版本

2

上传到设备

adb push tcpdump /sdcard/data/

通过 adb 将 tcpdump 文件上传到设备

3

运行 tcpdump

adb shell
cd /sdcard/data/
chmod 755 tcpdump
./tcpdump

通过 adb shell 登录设备并执行 tcpdump

分析 tcpdump 输出

成功运行 tcpdump 之后，接下来就可以分析输出的网络包内容了。iOS 设备和 Android 设备的输出是一致的。

tcpdump 输出示例

14:37:41.615018 IP 17.143.164.37.5223 > 10.29.44.140.58036: Flags [P.], seq 1:54, ack 101, win 255, options [nop,nop,TS val 2381386761 ecr 427050796], length 53

字段详解

字段 含义
14:37:41.615018 包接收到的时间
17.143.164.37.5223 发送方的 IP 地址及端口号
10.29.44.140.58036 接收方的 IP 地址及端口号
Flags [P.] TCP 包的 flag 位（见下方详解）
seq 1:54 TCP 包的 seq 号，1 是起始值，54 结束值，共 53 字节
ack 101 确认收到 seq 100，期望下一个从 101 开始
win 255 接收方还可以接受的字节数（窗口大小）
TS val 2381386761 TCP 包的时间戳，用于确认包的顺序
ecr 427050796 用于计算 RTT（往返时间）
length 53 应用层传过来的数据大小（不包括 TCP header）

TCP Flags 详解

Flags 位于 TCP header 的第十四个字节，包含 8 个比特位。每个 flag 都有特定的功能用途：

SYN

建立连接请求。三次握手的第一个包设置此位：SYN → SYN+ACK → ACK

ACK

确认收到。告诉对方上一个数据包已成功收到。一般不会单独发送，而是在下一个要发送的 packet 里设置。

PSH

Push。接收方需要马上将包 push 到应用层处理。HTTP request 的最后一个包通常设置此位。

FIN

Finish。发送方没有更多数据要发送了，之后要单向关闭连接。接收方回 ACK，再发送自己的 FIN 即可双向关闭。

RST

Reset。表明发送方马上就要断开当前连接了。HTTP 请求结束时一般可以看到设置了 RST 位。

URG

Urgent。表明包的优先级高，需要优先传送对方并处理。如 terminal 中 Ctrl+C 产生的网络数据包。

TCP Header 结构

理解 tcpdump 输出的关键是理解 TCP header 的结构。常看常新的 TCP header 包含：

  • 源端口 + 目标端口：标识通信的两端
  • Sequence Number (Seq)：包的序号，解决乱序问题
  • Acknowledgement Number (Ack)：确认号，解决丢包问题
  • Window：接收窗口，用于流量控制
  • Flags (第14字节)：8个比特位，操控 TCP 状态机
  • Options：可选字段，如时间戳、MSS 等

RTT 计算原理

tcpdump 输出中的 TS val 和 ecr 可以用来计算 RTT（Round Trip Time）：

  1. A 发送一个 TCP 包给 B，A 会在包里带上 TS val
  2. B 收到之后在 ACK 包里把这个值原样返回（作为 ecr）
  3. A 收到 B 的 ACK 包之后，根据本地时钟就可以计算出 RTT

这个时间戳和设备的系统时间没关系，刚开始是随机值，后面随着系统时钟自增长。ecr 值只在 ACK 包里有效，非 ACK 包 ecr 的值为 0。

常用 tcpdump 命令

# 监控特定端口
tcpdump -i any port 80

监控 80 端口的所有流量

# 监控特定主机
tcpdump -i any host 192.168.1.100

监控与特定主机的通信

# 只捕获 SYN 包
tcpdump -i any 'tcp[tcpflags] & tcp-syn != 0'

只捕获建立连接的 SYN 包

# 保存为 pcap 文件
tcpdump -i any -w capture.pcap

保存抓包结果，可用 Wireshark 分析

# 查看 HTTP 请求
tcpdump -i any -A 'tcp port 80'

以 ASCII 格式查看 HTTP 流量

# 限制抓包数量
tcpdump -i any -c 100

只抓 100 个包后自动退出

总结

  • iOS 抓包：使用 rvictl -s <udid> 创建虚拟网卡，再通过 tcpdump -i rvi0 抓包
  • Android 抓包：通过 adb 上传 tcpdump 到设备，在 root 环境下运行
  • Seq/Ack：Seq 是包的序号，Ack 是确认号（期望收到的下一个序号）
  • TCP Flags：SYN/ACK/FIN/RST/PSH/URG 6 个常用 flag，操控 TCP 状态机
  • RTT 计算：通过 TS val 和 ecr 计算往返时间
  • Wireshark：tcpdump 保存的 pcap 文件可用 Wireshark 进行 GUI 分析
