HTTP幂等性概念与应用

来源：酷壳 CoolShell · 作者：Todd

什么是幂等性？

在HTTP/1.1规范中，幂等性的定义是：

Methods can also have the property of "idempotence" in that (aside from error or expiration issues) the side-effects of N > 0 identical requests is the same as for a single request.

简单来说，HTTP方法的幂等性是指一次和多次请求某一个资源应该具有同样的副作用。幂等性属于语义范畴，正如编译器只能帮助检查语法错误一样，HTTP规范也没有办法通过消息格式等语法手段来定义它。

幂等性是分布式系统设计中十分重要的概念，而HTTP的分布式本质也决定了它在HTTP中具有重要地位。

分布式事务 vs 幂等设计

为什么需要幂等性？我们先从一个例子说起：

取款场景： 假设有一个从账户取钱的API withdraw(account_id, amount)

取钱的语义是从account_id对应的账户中扣除amount数额的钱。如果扣除成功返回true，账户余额减少；如果扣除失败返回false，账户余额不变。

问题： 在分布式环境中，无法轻易假设网络的可靠性。withdraw请求可能已经被服务器端正确处理，但服务器端的返回结果由于网络等原因被丢失了，导致客户端无法得知处理结果。

如果用户刷新页面重试，就会导致withdraw被调用两次，账户被多扣了一次钱。

解决方案一：分布式事务

通过引入支持分布式事务的中间件来保证withdraw功能的事务性。

优点

  • 对调用者很简单
  • 复杂性交给中间件管理

缺点

  • 架构太重量级
  • 容易被绑在特定的中间件上
  • 无法提供性能和可用性的保证

解决方案二：幂等设计

通过业务设计将操作变成幂等的：

// 1. 先获取唯一的处理号
ticket_id = create_ticket();

// 2. 带ticket_id的取款操作是幂等的
idempotent_withdraw(ticket_id, account_id, amount);

一个ticket_id表示的操作至多只会被处理一次，每次调用都将返回第一次调用时的处理结果。这样客户端就可以放心地多次调用。

优势： 轻量级、容易适应异构环境、性能和可用性好

HTTP方法的幂等性分析

HTTP协议本身是一种面向资源的应用层协议，但使用方式有两种：

  • RESTful：把HTTP当成应用层协议，忠实遵守HTTP协议的各种规定
  • SOA：把HTTP协议作为传输层协议，在HTTP之上建立自己的应用层协议
方法 语义 幂等性 说明
GET 获取资源 ✓ 幂等 不应有副作用，多次GET结果相同
DELETE 删除资源 ✓ 幂等 删除同一个资源，副作用相同
PUT 创建/更新资源 ✓ 幂等 同一URI多次PUT的副作用相同
POST 创建资源 ✗ 非幂等 多次POST会创建多个资源
PATCH 部分更新 ? 不一定 取决于具体实现

GET - 幂等

GET方法用于获取资源，不应有副作用，所以是幂等的。比如：

GET http://www.bank.com/account/123456

不会改变资源的状态，不论调用一次还是N次都没有副作用。

注意：强调的是副作用相同，而不是每次GET的结果相同。GET http://www.news.com/latest-news 可能会每次得到不同的结果，但它本身并没有产生任何副作用。

DELETE - 幂等

HTTP DELETE方法用于删除资源，有副作用但应该满足幂等性。比如：

DELETE http://www.forum.com/article/4231

调用一次和N次对系统产生的副作用是相同的，即删掉id为4231的帖子。

PUT - 幂等

PUT所对应的URI是要创建或更新的资源本身。比如：

PUT http://www.forum.com/articles/4231

对同一URI进行多次PUT的副作用和一次PUT是相同的。

POST - 非幂等

POST所对应的URI并非创建的资源本身，而是资源的接收者。比如：

POST http://www.forum.com/articles

两次相同的POST请求会在服务器端创建两份资源，它们具有不同的URI，所以POST方法不具备幂等性。

交互式：幂等性判断

点击判断以下操作是否是幂等的：

GET /users/123 点击判断

幂等

GET用于获取资源，多次获取不会改变资源状态。

POST /users 点击判断

非幂等

POST创建新资源，每次调用都会创建新的用户。

PUT /users/123 点击判断

幂等

PUT更新指定资源，多次更新同一资源结果相同。

DELETE /users/123 点击判断

幂等

删除已不存在的资源，返回结果可能不同，但副作用相同。

实际应用：实现取款功能

用Web API实现前面提到的取款功能：

// 1. 创建ticket（POST非幂等）
POST /tickets
Response: { "ticket_id": "abc123" }

// 2. 幂等取款
PUT /accounts/{account_id}/withdraw?ticket_id=abc123&amount=100
Response: { "success": true, "balance": 900 }

这种模式可以应用于很多场合，比如：论坛网站中防止意外的重复发帖。

总结

  • 幂等性是分布式系统设计中重要的概念，可以取代重量级的分布式事务
  • GET、DELETE、PUT是幂等的，POST是非幂等的
  • 幂等设计的关键是引入唯一标识（如ticket_id）来保证操作只被执行一次
  • RESTful API设计应该遵循HTTP协议的语义，考虑幂等性
