1 架构目标与评判标准

1.1 目标先行

架构设计的首要任务是明确目标。不同系统的目标不同：

系统类型核心目标
导弹发射系统稳定、迅速反应
互联网应用成本可控、快速迭代
金融系统安全、准确
电商系统高可用、可扩展
📐 评判标准

以是否实现架构师的设计目标为标准

  • 实现了设计目标 = 好架构
  • 未实现设计目标 = 不好的架构

2 架构设计原则

2.1 分层架构

最常见的架构模式，分为：

层级职责
表现层用户交互、UI 渲染
业务层业务规则处理
数据访问层数据库操作
✅ 核心原则：关注点分离

每层只负责其职责，避免跨层直接访问，防止"污水池反模式"。

3 SNAKE 分析法

SNAKE：场景 → 需求 → 应用 → 数据 → 进化
  • Scenario 场景：确定系统要解决什么问题，明确用户群体
  • Needs 需求：功能需求 + 非功能需求（性能、安全、可扩展性）
  • Application 应用：技术选型、架构设计、接口设计
  • Kilobyte 数据：数据量评估、存储方案、数据访问模式
  • Evolve 进化：未来扩展性、技术演进、架构迭代

  1. 单体架构：所有功能在一个应用中
  2. 垂直拆分：按业务模块拆分
  3. 服务化：通过 RPC 调用
  4. 微服务：独立部署、独立演进

7 架构决策框架

7.1 需要考虑的问题

  1. 什么最重要？ 性能？可用性？一致性？成本？
  2. 团队能力如何？
  3. 时间窗口？
  4. 技术约束？

7.2 决策原则

  • 最小化前期决策：只做必要的决策，保持灵活性
  • 提供多个方案：Sell Architecture Options
  • 验证决策：通过反馈循环验证架构有效性
💡 性能 vs 可维护性

可维护性通常比性能更重要。性能问题可以通过代码以外的方法解决（缓存、CDN、扩容），可维护性基本只能靠代码。
