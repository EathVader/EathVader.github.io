JVM 堆外内存回收机制详解

来源：占小狼的博客 · 2026-04-02

什么是堆外内存

JVM 启动时分配的内存称为堆内存，由 JVM 管理，GC 算法负责回收。但在代码中还可以使用堆外内存（Off-Heap Memory），这部分内存不归 JVM 管理，GC 算法不会对它们进行回收。

堆外内存的使用场景：

  • Netty 等高性能网络框架广泛使用堆外内存，减少内存拷贝
  • DirectByteBuffer 进行 NIO 操作时避免 Java 堆到 Native 堆的拷贝
  • 大内存缓存，不受 JVM 堆大小限制
注意：堆外内存不归 JVM 管理，GC 不会自动回收。使用时要格外小心，防止内存泄漏导致线上故障。

堆外内存的申请和释放

JDK 的 ByteBuffer 类提供了 allocateDirect(int capacity) 方法进行堆外内存申请，底层通过 unsafe.allocateMemory(size) 实现。

Java — 申请堆外内存
// 高层 API
ByteBuffer buffer = ByteBuffer.allocateDirect(1024);

// 底层调用链
// ByteBuffer.allocateDirect()
//   → Bits.reserveMemory()
//     → Unsafe.allocateMemory()
//       → malloc()  ← 最终是 C 语言的 malloc

最底层是通过 malloc 方法申请的，这块内存需要手动释放。Unsafe 提供了 freeMemory 接口进行释放：

Java — 释放堆外内存
unsafe.freeMemory(address);

堆外内存的回收机制

如果每次申请堆外内存都需要在代码中显式释放，对于 Java 来说显然不够优雅。JVM 虽然不会直接管理堆外内存，但通过巧妙的机制实现了间接回收。

DirectByteBuffer 与 Cleaner

JDK 使用 DirectByteBuffer 对象来表示堆外内存。每个 DirectByteBuffer 在初始化时，都会创建一个对应的 Cleaner 对象。这个 Cleaner 对象会在合适的时机执行 unsafe.freeMemory(address)，从而回收堆外内存。

Java — DirectByteBuffer 构造器关键代码
DirectByteBuffer(int cap) {
    super(-1, 0, cap, cap);
    // 1. 申请堆外内存
    long base = unsafe.allocateMemory(cap);
    // 2. 创建 Cleaner，关联 this（DirectByteBuffer）和 Deallocator
    cleaner = Cleaner.create(this, new Deallocator(base, cap, cap));
    // ...
}

// Deallocator 的 run 方法负责释放内存
private static class Deallocator implements Runnable {
    public void run() {
        if (address == 0) return;
        unsafe.freeMemory(address);  // 释放堆外内存
        address = 0;
        Bits.unreserveMemory(size, capacity);
    }
}

引用链与回收过程

Cleaner 的 clean 方法

Cleaner 对象的 clean 方法有两个作用：

  1. 把自身从 Cleaner 链表删除，从而在下次 GC 时能够被回收
  2. 释放堆外内存
Java — Cleaner.clean() 核心逻辑
public void run() {
    if (address == 0) {
        // Paranoia - 防止重复释放
        return;
    }
    unsafe.freeMemory(address);    // 释放堆外内存
    address = 0;
    Bits.unreserveMemory(size, capacity);
}

System.gc() 的触发机制

如果 JVM 一直没有执行 Full GC，无效的 Cleaner 对象就无法放入 ReferenceQueue，堆外内存也一直得不到释放——内存岂不会爆？

实际上，在初始化 DirectByteBuffer 时，如果当前堆外内存的条件很苛刻（剩余空间不足），会主动调用 System.gc() 强制触发 Full GC：

Java — Bits.reserveMemory()
static synchronized boolean reserveMemory(long size, int cap) {
    // ... 检查剩余堆外内存是否足够
    if (/* 内存不足 */) {
        System.gc();  // 主动触发 Full GC
        try { Thread.sleep(100); } catch (InterruptedException e) {}
        // 再次尝试申请
    }
    // ...
}
⚠️ 线上大坑：很多线上环境的 JVM 参数配置了 -XX:+DisableExplicitGC，导致 System.gc() 变成一个空函数，根本不会触发 Full GC！使用 Netty 等框架时要特别注意这个问题。

如何避免堆外内存泄漏

方案说明
不要使用 -XX:+DisableExplicitGC让 System.gc() 能正常触发，保证 Cleaner 有机会执行
主动调用 DirectByteBuffer.cleaner().clean()手动触发清理（不推荐，依赖内部 API）
使用 Netty 的 ReferenceCountUtil.release()Netty 自己管理堆外内存的生命周期
控制 DirectByteBuffer 的使用量通过 -XX:MaxDirectMemorySize 限制最大值

总结

核心要点

  1. 堆外内存申请：通过 ByteBuffer.allocateDirect() → Unsafe.allocateMemory() → malloc()
  2. Cleaner 机制：每个 DirectByteBuffer 关联一个 Cleaner，GC 回收 DirectByteBuffer 后，Cleaner 被放入 ReferenceQueue 并触发 clean()
  3. clean() 两件事：从 Cleaner 链表删除自身 + 调用 unsafe.freeMemory() 释放堆外内存
  4. System.gc()：堆外内存不足时会主动调用，但 -XX:+DisableExplicitGC 会让它失效
  5. 线上注意：使用 Netty 等框架时，要确认 GC 策略是否会影响堆外内存回收
