Java 内存模型详解

来源：HollisChuang's Blog · 2026-04-02

为什么要有内存模型

要理解 Java 内存模型，首先要了解计算机硬件层面的内存模型。这要从一段"古老的历史"说起。

CPU 和缓存一致性

计算机执行程序时，每条指令在 CPU 中执行，执行过程中需要和数据打交道，而数据存放在主存（物理内存）中。随着 CPU 技术发展，CPU 执行速度越来越快，而内存技术变化不大，导致 CPU 每次操作内存都要耗费大量等待时间。

🏢

公司类比

就像一家创业公司，创始人的能力越来越强，但基层员工跟不上节奏。老板的每一个命令传达到基层，由于理解能力和执行能力的欠缺，拖慢了整体效率。

于是公司设立了中层管理人员——CEO 只需告诉管理人员，管理人员负责协调底层员工。大多数时候，CEO 只要和管理人员沟通就够了。

人们想到的解决办法是：在 CPU 和内存之间增加高速缓存。缓存速度快、容量小、价格昂贵。

程序执行过程变成了：

  1. 将运算需要的数据从主存复制到 CPU 高速缓存
  2. CPU 直接从高速缓存读取和写入数据
  3. 运算结束后，将缓存中的数据刷新回主存

随着 CPU 能力提升，又衍生出多级缓存（L1 → L2 → L3），每一级都是下一级的一部分。

多核 CPU 的缓存一致性问题

多核 CPU 中，每个核心都有自己的 L1 缓存，共享 L3 缓存。当多个线程在不同核心上执行，且访问同一共享内存时，每个核心都会在各自的 cache 中保留一份副本。如果多个线程同时写各自的缓存，数据就可能不一致。

👥

多合伙人类比

如果公司有多个联席 CEO，每个 CEO 只把自己的命令下达给自己的管理人员。合伙人1要辞退员工a，合伙人2要给员工a升职。两个管理人员在没得到对方消息之前，各自执行了命令——这就产生了冲突。

处理器优化和指令重排

除了缓存一致性问题，处理器为了充分利用运算单元，可能会对代码进行乱序执行（处理器优化）。编译器（如 Java 的 JIT）也会做指令重排。

并发编程的三大问题

硬件层面的问题抽象为软件层面的三个并发问题：

问题硬件根源描述
原子性处理器优化操作要么全部执行，要么全部不执行，不会被中断
可见性缓存一致性一个线程修改了变量，其他线程能立即看到
有序性指令重排程序按照代码的先后顺序执行

什么是内存模型

最简单的方法是废除 CPU 缓存、废除处理器优化——但这会因噎废食，性能大幅下降。

内存模型（Memory Model）就是为了解决这个问题而诞生的规范：

内存模型定义了共享内存系统中多线程程序读写操作行为的规范。通过规则来规范对内存的读写操作，保证指令执行的正确性。它解决了 CPU 多级缓存、处理器优化、指令重排等导致的内存访问问题。

内存模型解决并发问题主要采用两种方式：限制处理器优化和使用内存屏障。

什么是 Java 内存模型（JMM）

Java 内存模型（Java Memory Model, JMM）是符合内存模型规范的机制，屏蔽了各种硬件和操作系统的访问差异，保证 Java 程序在各种平台下对内存的访问效果一致。JDK 5 开始使用的新的内存模型由 JSR-133 规范描述。

主内存与工作内存

JMM 规定：

  • 所有变量都存储在主内存中
  • 每条线程有自己的工作内存，保存了该线程使用到的变量的主内存副本
  • 线程对变量的所有操作都必须在工作内存中进行，不能直接读写主内存
  • 不同线程之间无法直接访问对方工作内存中的变量
  • 线程间变量的传递需要通过工作内存和主存之间的数据同步
注意：主内存和工作内存与 JVM 内存结构中的 Java 堆、栈、方法区等不是同一个层次的内存划分。主内存主要对应 Java 堆中的对象实例数据部分，工作内存对应于虚拟机栈中的部分区域。

JMM 的实现：关键字

JMM 除了定义规范，还提供了一系列原语（关键字），封装了底层实现供开发者直接使用。

volatile — 保证可见性和有序性

被 volatile 修饰的变量：

  • 被修改后立即同步到主内存
  • 每次使用前都从主内存刷新
  • 禁止指令重排
Java
// volatile 保证可见性
volatile boolean flag = false;

// 线程 A
flag = true;  // 修改后立即同步到主内存

// 线程 B
while (!flag) {
    // 每次读取都从主内存刷新，能立即看到线程 A 的修改
}

经典的 DCL（双重检查锁定）单例模式：

Java
public class Singleton {
    // volatile 禁止指令重排，防止其他线程拿到未初始化完的对象
    private static volatile Singleton instance;

    public static Singleton getInstance() {
        if (instance == null) {              // 第一次检查
            synchronized (Singleton.class) {
                if (instance == null) {      // 第二次检查
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}

synchronized — 保证原子性、可见性、有序性

synchronized 是 JMM 中最强大的关键字，同时满足三大特性：

  • 原子性：同一时刻只允许一条线程执行
  • 可见性：解锁前将工作内存数据刷新到主内存
  • 有序性：保证同一时刻只有一条线程操作
Java
// 同步方法
public synchronized void increment() {
    count++;
}

// 同步代码块（更细粒度）
public void increment() {
    synchronized (this) {
        count++;
    }
}

final — 保证可见性

被 final 修饰的字段在构造器中初始化后，其他线程就能看到正确的值（前提是对象引用没有逃逸）。

Java
public class ImmutableConfig {
    private final String url;
    private final int timeout;

    public ImmutableConfig(String url, int timeout) {
        this.url = url;
        this.timeout = timeout;
    }
    // 构造完成后，其他线程看到 url 和 timeout 一定是正确的值
}

关键字能力对比

关键字原子性可见性有序性
volatile❌✅✅
synchronized✅✅✅
final—✅—
性能提示：虽然 synchronized 是万能的，但它会影响性能。虽然编译器提供了很多锁优化技术（锁消除、锁粗化、轻量级锁、偏向锁），但也不建议过度使用。能使用 volatile 的场景优先使用 volatile。

总结

核心要点

  1. 为什么有内存模型：CPU 缓存和多核并行导致缓存一致性问题，处理器优化和编译器优化导致指令重排
  2. 并发编程三大问题：原子性、可见性、有序性
  3. JMM 是什么：一种规范，屏蔽硬件和 OS 差异，保证 Java 程序在各平台内存访问一致
  4. JMM 怎么做：定义主内存和工作内存模型，规定数据同步的时机和方式
  5. JMM 的实现：提供 volatile、synchronized、final 等关键字封装底层实现
