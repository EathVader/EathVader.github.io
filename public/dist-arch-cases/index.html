架构演进的驱动力

驱动因素具体表现
业务复杂度增长业务线增多，代码臃肿，迭代速度下降
用户规模爆发高并发压力，单机性能瓶颈
团队规模扩大多人协作困难，缺少并行研发能力
数据量膨胀单库单表瓶颈，需要分库分表
可用性要求提升单点故障风险，需要高可用架构

架构选型原则

📌 四大原则
  • 选择合适的技术而非最好的技术：权衡研发效率和产品目标
  • 架构围绕业务：不同业务时期需要不同架构
  • 提前规划扩展性：单体阶段就规划好未来解耦路径
  • 渐进式演进：在维护现有系统的基础上同时进行架构重构

机票搜索系统

高峰期承载日均十亿级 PV、月均上亿 UV。

核心指标数值

搜索结果 Average-Time< 3 秒

整体架构（水平分层 + 垂直按引擎划分）

终端渠道层
APP/WWW/低价
→
报价聚合层
全局缓存 + Cache Manager
→
MQ + Redis
数据交换区，解耦
→
报价计算引擎层
多渠道并行计算
→
基础数据层
运价/调价/航班
✅ 关键设计点
  • MQ + Redis 交互：解耦聚合层与引擎层，新增/减少引擎不影响聚合层
  • 缓存管理：所有影响报价的变更收拢到 Cache Manager
  • 多引擎并行：各渠道报价引擎并行计算，形成互补优势
  • 自定义 txt 格式：压缩到原来 30%~40%，极大节约带宽

4 负载均衡原理

为什么需要负载均衡

水平扩展的核心问题：从客户角度看，最好只有一个服务器。

NAT 模式（四层负载均衡）

客户端
→
LB
修改目的 IP/端口
→
RS
真实服务器
→
LB
改回源地址
→
客户端

瓶颈：所有流量（请求 + 响应）都经过 LB，响应报文通常远大于请求报文。

DR 模式（直接路由，高效方案）

客户端
→
LB
只修改 MAC 地址
→
RS
直接响应客户端
→
客户端
✅ DR 模式优势

LB 只负责转发请求，响应由 RS 直接返回，消除了 LB 的带宽瓶颈。这就是著名开源软件 LVS（Linux Virtual Server）的核心原理。

负载均衡策略

策略说明
轮询依次分配，最简单
加权轮询性能好的服务器权重高
最少连接当前连接数最少的服务器优先
一致性哈希相同请求始终路由到同一服务器

四层 vs 七层

  • 四层（传输层）：基于 TCP 连接，维护连接映射表
  • 七层（应用层）：解析 HTTP 报文，根据 URL、浏览器等信息智能分发

5 大数据分布式计算

MapReduce 四字诀：分、变、洗、合

① 切分（Split）
输入数据切分多份
→
② 变换（Map）
每条数据按相同规则映射
→
③ 洗牌（Shuffle）
相同 Key 汇聚到同一节点
→
④ 合并（Reduce）
同一 Key 数据合并输出

角色分工

角色职责
指挥官（Master）负责数据切分和任务调度
Map 计算兵负责变换操作
Reduce 计算兵负责合并操作
💡 核心要点

程序员只需定义两种规则：Map 变换规则和 Reduce 合并规则。分布式框架自动处理任务调度、节点间通信、容错恢复等复杂过程。

6 混沌工程

混沌工程旨在将故障扼杀在襁褓之中——在故障造成中断之前将它们识别出来。

发展历史

时间里程碑
2010Netflix 开发 Chaos Monkey，测试 AWS 实例故障影响
2011Simian Army 诞生，增加更多故障注入模式
2012Netflix 在 GitHub 开源 Chaos Monkey
2014"混沌工程师"角色诞生；FIT 概念提出

三步法

① 假设
假设系统故障时的行为
→
② 试验
设计针对性试验
→
③ 评估
评估故障影响

分布式系统八大谬论（Peter Deutsch）

  1. 网络是可靠的
  2. 延迟是零
  3. 带宽是无限的
  4. 网络是安全的
  5. 拓扑结构不会变
  6. 存在管理员这样的角色
  7. 传输成本是零
  8. 网络是同质的
💡 混沌工程的价值

客户：增强服务可用性 | 业务：避免利润损失，降低维护成本 | 技术：减少事故和轮班待命负担

CAP 定理

分布式系统不可能同时满足一致性、可用性和分区容错性，最多满足其中两个。

数据库CAP 选择说明
MongoDBCP一致性 + 分区容错，牺牲可用性
CassandraAP可用性 + 分区容错，不保证强一致性

消息队列选择

平台特点
Amazon SQS简单实用，创业早期首选
RabbitMQ特性丰富，复杂路由，简单灵活
ActiveMQ基于 Java，低延迟

三种扩展方式

增加克隆副本
最简单、最廉价
按功能划分
服务器专项化，SOA
按数据划分
分库分表、分片
✅ 全球分布式系统设计原则（NSONE）
  • 架构优先，而非代码优化：优先考虑水平扩展
  • 以角色（Role）思考系统：将应用拆分为解耦的、互相通信的角色
  • 跨数据中心通信是核心难题：不同通信模式使用不同工具
  • 一致性哈希是同步与分片的利器：无需实际通信即可达成一致
  • 测量和监控一切：深入应用层，建立对"正常行为"的直觉理解
