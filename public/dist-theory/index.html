1 CAP / BASE 理论

CAP 定理

属性含义
C（一致性）所有节点在同一时刻看到相同的数据
A（可用性）每个请求都能在合理时间内收到非错误的响应
P（分区容错性）系统在网络分区的情况下仍能继续运作
⚠️ 关键认知纠偏
  • P 是基础，不是可选项。网络分区是客观事实，真正的 trade-off 在 C 和 A 之间
  • "三选二"是误导。C、A、P 三者并不对等
  • CA 系统极难实现。Google Spanner 依赖极高可靠性的网络基础设施

PACELC 理论

发生分区时（PAC）
A 和 C 之间取舍
未发生分区时（ELC）
L（延迟）和 C 之间取舍

BASE 理论

ACIDBASE
Atomicity（原子性）Basically Available（基本可用）
Consistency（一致性）Soft State（软状态）
Isolation（隔离性）Eventually Consistent（最终一致性）

典型系统分类

类型特征代表系统
CP牺牲可用性，严格法定数协议MongoDB, HBase, ZooKeeper
AP牺牲一致性，非严格法定数协议CouchDB, Cassandra, Dynamo
CA（极难）依赖极高可靠性网络Google Spanner, OceanBase

2 分布式八大谬误

由 L. Peter Deutsch 等人在 Sun Microsystems 提出，每个分布式系统工程师必须内化的认知：

#谬误现实
1网络是可靠的网络随时可能中断、丢包、重复
2延迟为零跨数据中心延迟在 150ms 级别
3带宽是无限的带宽有限且共享
4网络是安全的恶意用户和程序持续适应安全措施
5拓扑不会变化服务器上下线、网络重组是常态
6只有一个管理员多个管理员可能实施冲突的策略
7传输成本为零构建和维护网络的隐性成本不可忽略
8网络是同构的网络设备、协议、性能各异
🔧 工程启示
  • 设计时必须假设网络不可靠、有延迟、不安全
  • 所有远程调用都需要超时、重试、幂等性设计
  • 不能把远程交互当作本地对象调用
  • 分布式系统的故障是部分的（partial failure），比全局故障更难处理

3 核心挑战

一致性模型谱系

模型描述
强一致性更新完成后，所有后续读操作都返回最新值
顺序一致性所有进程看到的操作顺序一致，但不要求实时性
因果一致性有因果关系的操作保序，无因果关系的可以乱序
读己所写进程永远能读到自己最近写入的值
单调读一旦读到某个值，后续不会读到更旧的值
最终一致性如果没有新的更新，最终所有副本会收敛到相同的值

策略描述适用场景
Fail-Fast快速失败，立即报错非幂等写操作
Fail-Over失效转移到备份节点读操作（推荐）

📌 FLP 不可能性定理

在异步系统中，即使只有一个进程可能崩溃，也不可能实现既总是正确（safety）又总是终止（liveness）的共识算法。绕过方式：引入故障检测器、随机化或时钟。

6 不可变性原则

💡 核心思想

"会计不用橡皮擦，否则他们会坐牢。" 不可变数据一旦写入就永不改变。修正只能通过追加新记录来实现。

为什么现在可以拥抱不可变性

存储越来越便宜
计算越来越便宜
协调越来越昂贵

不可变性的应用层次

层次应用
应用层Append-only 计算、事件溯源、账本式记录
数据库层事务日志（WAL）是真相，数据库是日志的缓存；MVCC
文件系统层LSM Tree、COW（Copy-on-Write）
分布式文件系统GFS/HDFS 的不可变文件和不可变块
大数据层MapReduce 基于不可变输入的函数式计算

不可变性如何打败 CAP

✅ 关键洞察

不可变数据没有更新操作，不存在副本不一致的问题。不需要版本化数据、向量时钟或读取修复。数据只有"存在"或"不存在"两种状态。在一致性哈希环中存储不可变数据，可以避免最终一致性的应用层复杂性。

7 设计原则

基础设计原则

原则说明
为故障而设计分布式系统的本质区别在于故障概率高且是部分故障
通过复制提高可用性分布式系统设计的首要原则
分区和复制分区提高并行处理能力，复制提高可用性和降低延迟
避免协调水平扩展的真正诀窍是独立性
用 CAP 批判设计CAP 不是用来构建系统的，而是用来审视和改进设计的

程序设计实践原则

原则说明
幂等性一次和多次请求应具有相同副作用
接口安全appCode + 非对称加密签名
统一返回格式status + errorCode + message + data
日期格式统一统一使用可读格式或 long 型时间戳
小数传输浮点数以字符串传输，避免精度丢失

工程实践建议

📊 指标与监控
  • 指标是唯一的工作方式：暴露延迟百分位、计数器、变化率
  • 使用百分位而非平均值：默认使用 P50、P99、P99.9、P99.99
  • 日志会说谎：优先看指标，再用日志验证
🔄 背压（Backpressure）

在整个系统中实现背压机制。过载或故障时丢弃新消息或返回错误。对其他系统的连接和请求使用超时和指数退避。没有背压机制，级联故障将不可避免。

🚩 Feature Flags

用 Feature Flag 逐步发布基础设施变更：并行写入 → 后台读取验证 → 对比检查 → 逐步切流。接受多版本基础设施和数据并存是常态。

分布式系统的真正难题

  • 是否允许任意节点并发可写：Google F1、OceanBase 都指定写节点
  • 读写并发：多版本数据存储 + 分布式事务
  • 元数据保存：Google TrueTime 用物理时间代替逻辑时钟
  • 读写事务期间节点故障：包括 leader、参与者、用户节点的任意故障
  • 嵌套/链式事务：电子商务的基础
