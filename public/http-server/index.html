Http Server 的演进：从单进程到epoll

来源：码农翻身 · 作者：刘欣

Http Server 1.0：单进程阻塞

HTTP的工作原理其实很简单：浏览器向服务器发送一个预先定义好的文本（HTTP Request），服务器处理一下（通常是从硬盘上取一个HTML文件，然后再把这个文件通过文本方式发回去（HTTP Response）。

唯一麻烦的是，需要请操作系统建立 HTTP 层下面的 TCP 连接通道，因为所有的文本数据都得通过这些 TCP 通道接收和发送，这个通道是用 socket 建立的。

// Http Server 1.0 伪代码
socket = socket()
bind(socket, 80)
listen(socket)

while True:
    conn = accept(socket)     // 阻塞等待连接
    data = receive(conn)      // 阻塞接收数据
    response = process(data)  // 处理请求
    send(conn, response)      // 发送响应
    close(conn)
问题： receive 调用很慢，足足等了 100 毫秒还没有响应！程序被阻塞住了。单进程一旦阻塞就什么事都干不了了。

Http Server 2.0：多进程并发

多进程的思路非常简单：当 accept 连接以后，对于这个新的 socket，不在主进程里处理，而是新创建子进程来接管。这样主进程就不会阻塞在 receive 上，可以继续接受新的连接了。

// Http Server 2.0 伪代码
while True:
    conn = accept(socket)     // 主进程只负责accept
    pid = fork()              // 创建子进程
    
    if pid == 0:              // 子进程处理
        data = receive(conn)
        response = process(data)
        send(conn, response)
        close(conn)
        exit()

问题： 当并发连接越来越多，成百上千的进程耗费大量系统资源。光是做进程切换就把操作系统累死了。

Http Server 3.0：Select模型

操作系统老大说："一个 Socket 连接就是一个文件描述符（fd，是个整数），这个 fd 背后是一个简单的数据结构，但是我们用了一个非常重量级的东西——进程——来表示对它的读写操作，有点浪费啊。"

解决方案是改变工作方式：

  1. Http Server 把要等待的 socket fd 编号告诉操作系统
  2. Http Server 进入阻塞状态
  3. 操作系统在后台检查这些编号的 socket，发现可以读写的做标记
  4. 唤醒 Http Server，让它去遍历所有 socket fd，看谁有标记，有标记的做相应处理
  5. 处理完了，再把 socket fd 告诉操作系统，再次进入阻塞，如此循环往复
// Http Server 3.0 select 伪代码
fd_set readfds;

while True:
    FD_ZERO(&readfds)
    for fd in all_sockets:
        FD_SET(fd, &readfds)
    
    select(max_fd + 1, &readfds, NULL, NULL, NULL)  // 阻塞等待
    
    for fd in all_sockets:                          // 遍历所有fd
        if FD_ISSET(fd, &readfds):                  // 检查标记
            data = receive(fd)
            response = process(data)
            send(fd, response)
核心思想： 单进程就可以处理所有的 socket！不再需要为每个连接创建进程。
问题：
  • select 最多只能监控 1024 个 socket fd
  • 每次唤醒后需要遍历所有 1000+ 个 socket fd，但实际可能只有几十个需要处理
  • 大量不活跃的 socket 也被检查，效率低

Http Server 4.0：epoll

epoll 和 select 类似，但关键改进在于：操作系统只会告诉 Http Server 那些可以读写的 socket，Http Server 只需要处理这些 'ready' 的 socket 就可以了。

// Http Server 4.0 epoll 伪代码
epfd = epoll_create()
epoll_ctl(epfd, EPOLL_CTL_ADD, socket, &event)

while True:
    nfds = epoll_wait(epfd, events, MAX_EVENTS, -1)  // 阻塞等待
    
    for i in 0..nfds:                                // 只遍历就绪的
        fd = events[i].data.fd
        data = receive(fd)
        response = process(data)
        send(fd, response)
特性 select epoll
最大连接数 1024 无限制
就绪检测 遍历所有fd 只返回就绪的fd
时间复杂度 O(n) O(1)
适用场景 连接少且活跃 大量连接，少量活跃
平台 跨平台 Linux 特有

演进总结

1.0 单进程阻塞

一个 receive 阻塞就什么事都干不了

2.0 多进程并发

每个连接一个进程，进程切换开销大

3.0 select模型

单进程处理所有连接，但需要遍历所有fd

4.0 epoll

只处理就绪的socket，性能飞跃提升

总结

  • select 和 epoll 都是 I/O 多路复用技术，让单进程可以处理多个连接
  • select 需要遍历所有 fd，epoll 只返回就绪的 fd，性能差距巨大
  • epoll 是 Linux 特有的，其他平台有类似机制（kqueue、IOCP）
  • 现代高性能服务器（Nginx、Redis）都使用 epoll 模型
