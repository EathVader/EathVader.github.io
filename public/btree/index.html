1 为什么需要 B 树 / Why B-Tree

📚

类比 / Analogy

想象你在一个巨大的图书馆里找一本书。二叉树就像每个路口只有"左/右"两个方向，你要走很多层楼梯。B 树就像每个路口有几十个方向标，几步就能到达目标楼层。楼梯越少 = 磁盘 IO 越少 = 越快。

二叉搜索树（BST/AVL/红黑树）的问题：树的深度 = O(log₂ n)。当数据量达到百万级别时，树高约 20 层，意味着 20 次磁盘 IO。

💡 核心思想

降低树的深度 → 减少磁盘 IO → 用多叉树代替二叉树。一个节点存多个键值，有多个子节点，树高从 log₂n 降到 logmn（m 为阶数，通常几百到几千）。

2 磁盘 IO 基础 / Disk IO Basics

理解 B 树之前，必须理解磁盘读写的代价：

寻道 Seek
~10ms
→
旋转 Rotate
~4ms
→
传输 Transfer
~0.02μs/byte
💾 关键认知
  • 磁盘以块 (block) 为单位读写，通常 4KB~16KB
  • 一次随机 IO ≈ 10ms，而内存访问 ≈ 100ns，差 10 万倍
  • B 树的设计目标：让一个节点 = 一个磁盘块，最大化每次 IO 的信息量
  • 树高每降低 1 层 = 少一次磁盘 IO = 节省 ~10ms

3 B 树 / B-Tree

一棵 m 阶 B 树的性质：

性质说明
每个节点最多 m 个孩子m 阶 = 最多 m 路分支
非根节点至少 ⌈m/2⌉ 个孩子保证节点至少半满
根节点至少 2 个孩子除非只有根节点
有 k 个孩子的节点有 k-1 个键键分隔子树范围
所有叶节点在同一层完美平衡
B 树：每个节点存多个键，数据分布在所有节点中
🔍 B 树查找过程

① 读取根节点（1 次 IO）→ ② 二分查找确定子树 → ③ 读取子节点（1 次 IO）→ 重复直到找到。总 IO 次数 = 树高 = O(logm n)。

4 B+ 树 / B+ Tree

B+ 树是 B 树最重要的变体，也是数据库索引的事实标准（MySQL InnoDB、PostgreSQL 都用 B+ 树）。

🔑 B+ 树 vs B 树的三个关键区别
  1. 数据只在叶节点：内部节点只存索引键，不存数据
  2. 叶节点链表相连：支持高效范围查询
  3. 内部节点有 n 个键就有 n 个子树（B 树是 n 个键 n+1 个子树）
B+ 树：内部节点仅索引，数据全在叶节点，叶节点链表相连

B+ 树的优势

优势原因
磁盘 IO 更少内部节点不存数据 → 能容纳更多键 → 树更矮
查询效率稳定所有查询都要到叶节点 → 路径长度一致
范围查询高效叶节点链表 → 顺序扫描，不用反复遍历树
缓存友好内部节点小 → 更多节点能放进内存缓存

4b B+ 树分裂过程 / B+ Tree Node Split

当一个节点满了（键数 = m-1），插入时触发分裂：将节点一分为二，中间键上提到父节点。

✂️

类比 / Analogy

就像一个书架放满了，把书架从中间切开，左半边留原位，右半边放到新书架，然后把分界点的书名告诉上层目录。

B+ 树分裂：节点一分为二，分界键复制到父节点（B 树是移动，B+ 树是复制）

4c MySQL InnoDB 中的 B+ 树 / B+ Tree in MySQL

MySQL InnoDB 存储引擎使用 B+ 树实现索引，分为两种：

聚簇索引叶节点存完整行；二级索引叶节点只存主键，查询需"回表"
💡 覆盖索引 / Covering Index

如果查询的字段都在二级索引中（包括主键），就不需要回表，称为覆盖索引。例如：SELECT id, name FROM t WHERE name='Alice' 只需扫描 name 索引，无需回表。

5 B* 树 / B* Tree

B* 树是 B+ 树的进一步优化：

🔧 B* 树的改进
  • 非叶节点增加兄弟指针
  • 节点最低使用率从 1/2 提高到 2/3
  • 分裂策略：先尝试向兄弟节点转移数据，兄弟也满了才分裂
节点满了
→
兄弟有空间？
转移数据
→ 没有
两节点各出 1/3
创建新节点

效果：分裂概率更低 → 空间利用率更高（~2/3 vs B+ 树的 ~1/2）。
