1 缓存更新策略

Cache Aside Pattern（旁路缓存，最常用）

Facebook 论文《Scaling Memcache at Facebook》采用的策略，业界最广泛使用。

读（失效）
读 cache → 未命中 → 读 DB → 写 cache
读（命中）
直接从 cache 返回
写（更新）
先更新 DB → 让缓存失效
⚠️ 为什么是"失效"而非"更新"缓存？

两个并发写操作可能导致缓存中的数据是脏的。Facebook 选择删除缓存而非更新缓存，避免并发写导致的脏数据。

❌ 常见错误：先删缓存，再更新数据库

① 更新操作删除缓存 → ② 查询操作未命中，从 DB 读到旧数据放入缓存 → ③ 更新操作更新 DB。结果：缓存中永远是旧数据。

Read/Write Through Pattern（读写穿透）

应用层认为后端就是一个单一存储，缓存层自己维护与数据库的同步。

  • Read Through：缓存失效时，由缓存服务自己加载数据（对应用透明）
  • Write Through：写数据时，命中缓存则更新缓存，由缓存同步更新数据库

Write Behind Caching Pattern（写回）

与 Linux 文件系统的 Page Cache 算法原理相同：更新数据时只更新缓存，不更新数据库，缓存异步批量更新数据库。

四种模式总结

模式一致性性能复杂度适用场景
Cache Aside最终一致好低通用场景，最推荐
Read/Write Through较强好中缓存层有代理能力
Write Behind弱极高高写密集、可容忍丢失
💬 陈皓观点

这些 Design Pattern 都是计算机体系结构里的老古董设计（CPU 缓存、硬盘缓存、数据库缓存），历经长时间考验。如果你要做好架构，首先得把计算机体系结构以及很多老古董的基础技术吃透了。

3 Instagram Promise 缓存方案

问题场景

Instagram 的服务是无状态的，通过缓存减少 90% 的后端请求。但在服务重启、全新请求、新集群上线时，缓存为空意味着 100 个并发请求全部 cache miss，产生 100 个独立的后端请求。

传统方案及其不足

方案缺点
外部缓存（memcache）增加复杂性和故障模式
本地磁盘持久化增加复杂性
预热（fake requests）不能覆盖所有场景
缓慢发布新客户端限制业务敏捷性

Promise 缓存方案（核心创新）

💡 关键思想

不缓存实际的值，而是缓存一个 Promise（将来会提供值的对象）。

① 请求到来
原子检查缓存
→
② Cache Miss
创建 Promise 插入缓存
→
③ 异步请求后端
Promise 开始加载
→
④ 后续请求命中 Promise
所有 worker 等待同一请求
→
⑤ Promise Fulfill
所有 worker 同时获得结果
✅ 效果

即使缓存为空，也能维持 90% 的缓存命中率假设——因为相同请求只会产生一次后端调用。Instagram 使用 C++ 服务，基于 folly::SharedPromise 实现。

🔑 核心洞察

Promise 缓存将"缓存值"升级为"缓存计算过程"，从根本上解决了冷缓存下的惊群问题。

4 内存一致性模型

为什么需要内存一致性模型

在多核/多线程环境下，不同线程如何观察共享内存状态是一个根本性问题。内存一致性模型是硬件和软件之间的契约。

一致性模型层次

模型说明代价
顺序一致性（SC）所有操作有全局顺序，单线程按编写顺序极慢，每次只能运行一条指令
TSO引入 Store Buffer 隐藏写延迟允许 SC 禁止的行为（x86 采用）
弱排序（ARM）几乎允许任何重排需要显式内存屏障

内存屏障（Memory Barrier / Fence）

  • Barrier/Fence：强制屏障前的所有内存操作完成后，才能开始屏障后的操作
  • 原子 CAS（Compare-and-Swap）：更实用的同步原语
  • 代价高昂（数百 cycles），应谨慎使用

对缓存系统的启示

硬件概念缓存系统中的映射
SC（顺序一致性）强一致性缓存（2PC/Paxos）
TSOCache Aside（最终一致，极端 case 可能读到旧值）
Weak OrderingWrite Behind（异步写回，弱一致）
Barrier缓存失效操作（强制同步点）
Data Race并发读写缓存和数据库
💡 核心教训

使用同步库！它会处理丑陋的重排问题。正如陈皓所说：宏观系统架构中的很多设计都来源于微观的计算机体系结构。

5 缓存分类与工程实践

缓存特征三要素

特征说明
命中率命中率 = 正确结果数 / 请求缓存次数，核心指标
最大容量超过后触发清空策略
清空策略FIFO（先进先出）、LFU（最少使用）、LRU（最近最少使用）
