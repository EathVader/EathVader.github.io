Guava Cache 源码分析 — 引用回收与移除通知

来源：crossoverJie · 2026-04-02

前言

Guava Cache 是 Google 开源的本地缓存实现。在分析其原理之前，需要先了解 Java 自带的两个特性——引用类型和事件回调，Guava 中都有具体的应用。

Java 中的四种引用

JVM 根据可达性分析算法找出需要回收的对象，判断对象存活状态都和"引用"有关。JDK 1.2 之后新增了四种引用状态，用于更细粒度地划分引用关系：

引用类型强度GC 行为典型场景
强引用 (Strong)最强不会被 GC 回收A a = new A()，最常见
软引用 (Soft)较强内存不足时才回收缓存：有用但非必需的对象
弱引用 (Weak)较弱GC 时无论内存是否足够都回收WeakHashMap、Guava weakKeys
虚引用 (Phantom)最弱被回收时获得通知DirectByteBuffer 的 Cleaner
Java — 四种引用示例
// 强引用 — 不会被 GC 回收
A a = new A();

// 软引用 — 内存不足时才被回收
SoftReference softRef = new SoftReference<>("soft");

// 弱引用 — GC 时一定被回收
WeakReference weakRef = new WeakReference<>("weak");

// 虚引用 — 无法通过引用获取对象，仅用于回收通知
ReferenceQueue queue = new ReferenceQueue<>();
PhantomReference phantomRef = new PhantomReference<>("phantom", queue);

事件回调机制

事件回调是一种常见的设计模式。在 Java 中利用接口来实现回调：

Java — 回调接口定义
public interface CallBackListener {
    /** 回调通知函数 */
    void callBackNotify(String msg);
}
Java — Caller（调用方）
public class Caller {
    private CallBackListener callBackListener;
    private Notifier notifier;
    private String question;

    public void call() {
        // 新建线程，达到异步效果
        new Thread(() -> {
            notifier.execute(Caller.this, question);
        }).start();
        // 提问完毕，继续做其他事
    }
}
Java — Notifier（响应方）
public class Notifier {
    public void execute(Caller caller, String msg) throws InterruptedException {
        // 执行计算（耗时操作）
        TimeUnit.SECONDS.sleep(2);
        // 回调 Caller 告知结果
        caller.getCallBackListener().callBackNotify("我在北京！");
    }
}
Java — 使用示例
Notifier notifier = new Notifier();
Caller caller = new Caller();
caller.setNotifier(notifier);
caller.setQuestion("你在哪儿！");
caller.setCallBackListener(new CallBackListener() {
    @Override
    public void callBackNotify(String msg) {
        System.out.println("回复=【" + msg + "】");
    }
});
caller.call();
输出结果
[main] Caller - 开始提问
[main] Caller - 提问完毕，我去干其他事了
[Thread-0] Notifier - 收到消息=【你在哪儿！】
[Thread-0] Notifier - 等待响应中。。。。。
[Thread-0] Main - 回复=【我在北京！】

Guava Cache 的引用回收

Guava 利用 Java 的引用类型，可以在初始化缓存时自定义键和值的引用关系：

Java
LoadingCache loadingCache = CacheBuilder.newBuilder()
    .expireAfterWrite(2, TimeUnit.SECONDS)
    .weakKeys()      // 键使用弱引用
    .weakValues()    // 值使用弱引用
    // .softValues()  // 或者使用软引用
    .removalListener(new RemovalListener() {
        @Override
        public void onRemoval(RemovalNotification notification) {
            System.out.println("删除原因=" + notification.getCause()
                + "，删除 key=" + notification.getKey()
                + "，删除 value=" + notification.getValue());
        }
    })
    .build(new CacheLoader() {
        @Override
        public AtomicLong load(Integer key) throws Exception {
            return new AtomicLong(0);
        }
    });

ReferenceEntry 的定义

Guava Cache 中的 ReferenceEntry 类似于 HashMap 的 Entry，用来存放数据：

Java — ReferenceEntry 接口
interface ReferenceEntry {
    ValueReference getValueReference();
    void setValueReference(ValueReference valueReference);
    ReferenceEntry getNext();
    int getHash();
    K getKey();
    long getAccessTime();
    void setAccessTime(long time);
}

根据 ValueReference 的实现，具有强引用和弱引用的不同实现，key 也是同样的道理。当使用弱引用构造方式时，弱引用的 key 和 value 都会被垃圾回收。

显式回收

Java
// 回收单个 key
cache.invalidate(key);

// 回收多个 key
cache.invalidateAll(keys);

// 回收所有条目
cache.invalidateAll();

Guava Cache 的回调机制

当缓存被删除时，Guava 会回调我们自定义的函数，并告知删除原因。

Java — 使用 RemovalListener
LoadingCache loadingCache = CacheBuilder.newBuilder()
    .expireAfterWrite(2, TimeUnit.SECONDS)
    .removalListener(new RemovalListener() {
        @Override
        public void onRemoval(RemovalNotification notification) {
            LOGGER.info("删除原因={}，删除 key={}, 删除 value={}",
                notification.getCause(),
                notification.getKey(),
                notification.getValue());
        }
    })
    .build(new CacheLoader() {
        @Override
        public AtomicLong load(Integer key) throws Exception {
            return new AtomicLong(0);
        }
    });

回调实现原理

Guava 是如何实现回调的？跟着源码走一遍：

  1. LocalCache.getLiveValue() 中判断缓存过期
  2. 跟着调用关系一直跟到 removeValueFromChain()
  3. enqueueNotification() 方法将回收的缓存（包含 key、value）以及回收原因包装成事件通知，加入到一个本地队列中

这样一看，似乎也没有立即回调我们初始化时候的事件。但用过队列的同学应该能猜到：既然写入队列，那就肯定有消费。

回到获取缓存的地方，在 finally 中执行了 postReadCleanup() 方法——这里面就是对刚才的队列进行了消费。一直跟进来就会发现，这里消费了队列，将之前包装好的移除消息调用了我们自定义的事件，这样就完成了一次事件回调。

关键设计：回调不是在移除时立即执行，而是将通知加入队列，在下次 get() 时消费队列并触发回调。这种异步设计避免了在 GC 线程或移除操作的上下文中执行用户代码，保证了线程安全性。

总结

核心要点

  1. 四种引用：强引用（不回收）、软引用（内存不足回收）、弱引用（GC 时回收）、虚引用（回收通知）
  2. Guava 引用配置：weakKeys()、weakValues()、softValues() 自定义键值引用关系
  3. 事件回调：通过 RemovalListener 接口，缓存被删除时回调并告知原因
  4. 回调实现：移除时将通知加入本地队列，在 get() 的 postReadCleanup() 中消费队列触发回调
  5. 异步设计：回调延迟执行，避免在 GC 线程中执行用户代码
