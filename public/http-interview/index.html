HTTP 协议详解

来源：SegmentFault · 作者：cxuan

什么是 HTTP？

HTTP（Hypertext Transfer Protocol）是一个在计算机世界里专门在两点之间传输文字、图片、音频、视频等超文本数据的约定和规范。

📄
超文本
语义扩大的文本，包括图片、音频、视频、超链接
→
🔄
传输
二进制数据包由计算机终端传输到另一个终端
→
📋
协议
计算机之间相互通信需要共同遵守的规则

HTTP 请求响应过程

当你在浏览器中输入网址并点击回车时，发生以下过程：

1

DNS 解析

DNS服务器将域名映射为IP地址

2

建立TCP连接

客户端在80端口发起TCP连接（三次握手）

3

发送HTTP请求

客户端通过套接字发送HTTP请求报文

4

服务器处理

服务器解析请求，从存储器检索对象

5

返回HTTP响应

服务器封装对象到HTTP响应报文并发送

6

关闭连接/渲染

TCP连接关闭，浏览器解析HTML并渲染页面

HTTP 报文格式

HTTP报文由三大部分组成：

起始行（Start Line）

描述请求或响应的基本信息

GET /path/to/resource HTTP/1.1

头部字段（Headers）

使用key-value形式更详细地说明报文

Host: example.com
User-Agent: Mozilla/5.0
Accept: text/html
空行（CRLF）

消息正文（Body）

实际传输的数据，可以是图片、视频等二进制数据

Header是必须的，Body可以没有。Header和Body之间必须有一个空行（CRLF）。

HTTP 请求方法

HTTP请求方法共有8种，最常用的是GET和POST：

GET

获取资源

请求访问已被URI识别的资源，服务器解析后返回响应内容

幂等 安全
POST

传输实体

用于传输实体主体，一般用于提交数据

非幂等
PUT

传输文件

上传文件到请求URI指定的位置

幂等
DELETE

删除文件

删除请求URI指定的资源

幂等
HEAD

获得响应首部

与GET一样，但不返回报文主体部分

幂等 安全
OPTIONS

询问支持的方法

查询针对请求URI指定的资源支持的方法

幂等
TRACE

追踪路径

让Web服务器端将之前的请求通信环回给客户端

幂等
CONNECT

隧道协议连接代理

建立隧道，实现用隧道协议进行TCP通信

非幂等

HTTP 状态码

HTTP状态码表示请求的处理结果，分为5类：

2xx 成功

200成功响应
204请求处理成功，但无资源返回
206对资源某一部分进行响应（断点续传）

3xx 重定向

301永久性重定向
302临时性重定向
303使用GET方法定向获取资源
304未修改（缓存命中）
307临时重定向（与302相同含义）

4xx 客户端错误

400请求报文存在语法错误
401需要认证信息
403访问被服务器拒绝
404服务器上无法找到请求的资源

5xx 服务器错误

500服务器执行请求时发生错误
503服务器暂时超负载或停机维护

URL 结构解析

以完整URL为例：

http://www.example.com:80/path/to/myfile.html?key1=value1&key2=value2#SomewhereInTheDocument
协议（Scheme） http:// 或 https://
主机（Host） 域名或IP地址
端口（Port） HTTP默认80，HTTPS默认443
路径（Path） Web服务器上资源的路径
查询参数（Query） 键/值对列表，用&分隔
锚点（Hash） 资源内的"书签"位置，不发送到服务器

重要请求头

Host

指明服务器的域名和端口

Host: developer.mozilla.org

User-Agent

告诉服务器客户端的类型和版本

User-Agent: Mozilla/5.0 ...

Accept

客户端能够理解的MIME类型

Accept: text/html,application/xml;q=0.9

Accept-Language

客户端希望的自然语言

Accept-Language: en-US,en;q=0.5

Accept-Encoding

客户端支持的编码方式

Accept-Encoding: gzip, deflate, br

Referer

告诉服务器请求是从哪个页面链接过来的

Referer: https://example.com/page

If-Modified-Since

条件请求：资源在指定日期后是否修改

If-Modified-Since: Mon, 18 Jul 2016...

If-None-Match

条件请求：ETag是否匹配

If-None-Match: "c561c68d0ba92bbe..."

重要响应头

Content-Type

响应主体的MIME类型

Content-Type: text/html; charset=utf-8

Content-Length

实体主体的大小（字节）

Content-Length: 348

Content-Encoding

响应主体的编码方式

Content-Encoding: gzip

ETag

资源的唯一标识，用于缓存验证

ETag: "c561c68d0ba92bbe..."

Last-Modified

资源最后修改时间

Last-Modified: Mon, 18 Jul 2016...

Set-Cookie

设置Cookie

Set-Cookie: mykey=myvalue; Path=/

Access-Control-Allow-Origin

CORS跨域控制

Access-Control-Allow-Origin: *

Cache-Control

缓存控制指令

Cache-Control: max-age=3600

内容协商

内容协商机制是指客户端和服务器端就响应的资源内容进行交涉，然后提供给客户端最为适合的资源。

服务器驱动协商

由服务器端根据请求首部字段进行自动处理

客户端驱动协商

由客户端来进行内容协商

透明协商

服务器端和客户端各自进行内容协商

内容协商相关头部

  • Accept - 客户端接受的MIME类型
  • Accept-Charset - 客户端接受的字符集
  • Accept-Encoding - 客户端接受的编码方式
  • Accept-Language - 客户端接受的自然语言
  • Content-Language - 资源使用的自然语言

HTTP 的优缺点

优点

  • 简单灵活易扩展：header + body 结构简单，允许开发者任意定制
  • 应用广泛、环境成熟：跨语言、跨平台，几乎所有编程语言都有HTTP库
  • 无状态：不需要额外资源记录状态，减轻服务器负担

缺点

  • 无状态：无法支持连续多个步骤的事务操作，需要Cookie技术
  • 明文传输：不安全，可以被监听和窥探
  • 性能：不完全适应现在的互联网，还有很大提升空间

总结

  • HTTP = 超文本 + 传输 + 协议，是Web通信的基础
  • 报文格式：起始行 + 头部 + 空行 + 主体
  • 8种请求方法：GET/POST/PUT/DELETE/HEAD/OPTIONS/TRACE/CONNECT
  • 状态码：2xx成功、3xx重定向、4xx客户端错误、5xx服务器错误
  • URL由协议、主机、端口、路径、查询参数、锚点组成
  • 内容协商机制让客户端和服务器协商最适合的资源格式
