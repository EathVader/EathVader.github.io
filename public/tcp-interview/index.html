Q1 进程 A 发送字符串"hello"，TCP 返回成功确认，进程 A 能否确定进程 B 收到了？

❌ 答案：不能！

TCP 确认收到，只表示数据到达了进程 B 的 TCP 缓冲区，并不表示进程 B 已经读取。

📬

类比：邮局

邮局告诉你"信已送到"，但收件人可能还没打开邮箱。TCP 的"确认"就是邮局的送达通知，不是收件人的阅读确认。

💡 风险

如果此时机器突然掉电，缓冲区数据丢失，进程 B 将无法收到"hello"。

Q2 如何尽量避免上述情况？

✅ 答案：设置 PSH 标志

在 TCP 报文段首部设置 PSH (Push) 标志，告诉接收方 TCP 尽快将数据交给进程，不要缓存。

// 示例：socket 设置
socket.setOption(TCP_NODELAY, true);  // 或者使用 PSH 标志

Q3 为什么关闭连接需要 4 次挥手，而建立连接只需要 3 次握手？

📝 答案

TCP 是全双工，两个方向可以独立关闭。

因为 TCP 允许半关闭状态：

  • A 发送 FIN："我要发完数据了"
  • B 确认："好的，但你可能还没发完"
  • B 继续发送数据...
  • B 发完 → 发送 FIN
  • A 确认 → 完全关闭

Q4 服务器突然掉电重启，客户端不知情，二者之间的 TCP 连接处于什么状态？

⚠️ 答案：半打开状态

客户端还认为连接正常，但服务器已经没有连接信息了。

🚪

类比

你打电话对方手机突然关机，你不知道，还以为通话中。这就是"半打开"——一边还保持着会话，另一边已经重置了。

Q5 客户端通过半打开连接向服务器请求数据，服务器会怎么做？

❌ 答案：发送 RST

服务器收到请求后：

  1. 进行 ARP 查询获取客户端 MAC
  2. 发现没有连接信息（丢失了）
  3. 发送 RST (Reset) 给客户端
💡 RST 的含义

"哥们，我不认识你，想跟我说话请先发送 SYN！"

Q6 客户端重新建立连接但端口号错误，会发生什么？

⚠️ 答案：两种情况
  1. 有进程监听该端口：TCP 交给进程，进程可能拒绝并发送 RST
  2. 无进程监听：TCP 直接发送 RST（大多数情况）

Q7 多进程服务器，进程 A 接受连接后新建进程 B 处理，进程 ABC 是否监听同一端口？

✅ 答案：是！

所有连接请求首先到达监听进程 A，通过 <sourceIP:port, targetIP:port> 四元组区分不同连接，交给进程 B 或 C。

Q8 TCP 如何决定把报文段发给哪个进程？

📝 答案
  • SYN 包：全部交给监听进程 A
  • 其他包：根据四元组 (src_ip, src_port, dst_ip, dst_port) 决定

Q9 服务器正在创建进程时又有新连接请求，TCP 如何处理？

📝 答案

服务器进程 A 会让 TCP 维护一个连接队列。三次握手后，连接放入队列，服务器进程在合适时取走。

Q10 连接队列会影响并发处理能力吗？

✅ 答案：不会！

队列只影响"等待处理"的连接数量，不影响实际处理能力。并发能力取决于服务器进程设计和硬件。

Q11 MSS 和 MTU 是什么？二者是什么关系？

📝 答案
  • MSS：TCP 最大报文段长度（数据部分）
  • MTU：最大传输单元（包含头部）

例如 MTU=1500 字节，MSS 只能是 1460 字节。

Q12 机器 A (MSS=1400) 向机器 B (MSS=1600) 发送 1600 字节数据，可以吗？

❌ 答案：不可以！

虽然没突破 B 的 MSS，但突破了 A 自己的 MSS。数据会被切分为 2 个 IP 片，每个 840 字节，带重复的头部，降低传输效率。

Q13 机器 A 想尽快断开连接，应该怎么办？

❌ 答案：发送 RST

直接发送 RST (Reset) 标志，表示异常关闭："我关闭连接了，你看着办吧。"

Q14 收到 RST 后，TCP 会怎么做？

⚠️ 答案

TCP 告诉上层进程"连接已断开"，直接回收连接资源，不发送任何确认给对端。

Q15 收到 FIN 并回复 ACK，算完成 4 次挥手了吗？

❌ 答案：不算！

还需要等待 2MSL：

  • 防止 ACK 没送到对方，对方重传 FIN
  • 确保连接不会跟后续连接混在一起
📝 MSL

MSL (Maximum Segment Lifetime)：报文在网络中的最大存活时间，通常 2 分钟。Linux 默认 30 秒。
