1 核心需求

需求说明
全局唯一不同节点、不同时间生成的 ID 不能重复
趋势有序ID 基本按时间递增，有利于数据库索引

2 常见方案

UUID

✅ 本地生成，无中心化
❌ 无序，36字符
📌 适合不依赖有序的业务

数据库自增

✅ 有序递增，简单
❌ 单点瓶颈，扩展难
📌 适合单机单库

雪花算法（Snowflake）

❄️ Twitter 开源

64 位 long 型 ID，单机每秒可生成约 400 万个 ID。

号段模式

批量获取 ID（如 1000 个）
→
内存中分配
→
用完再获取下一段

Redis INCR

✅ 性能高、有序
❌ 依赖外部 Redis
📌 适合已有 Redis 基础设施

3 雪花算法详解

64 位结构

位段长度说明
符号位1 bit永远是 0
时间戳41 bit毫秒时间戳，支持 69 年
数据中心 ID5 bit0-31
机器 ID5 bit0-31
序列号12 bit每毫秒 4096 个 ID
⚠️ 时钟回拨问题

机器时钟回拨会导致 ID 重复。解决方案：检测回拨后等待追上，或使用更大的序列号。

优缺点

✅ 高性能
✅ 有序
✅ 分布式
❌ 依赖时钟

✅ 总结
  • 无序可接受：UUID
  • 高性能+有序：雪花算法（推荐）
  • 简单实现：数据库自增
  • 批量生成：号段模式
