1 问题场景

⚠️ 实际案例

某 PHP 服务通过 Nginx 封装后端存储：

  • 测试：小对象（几字节）→ 几毫秒，正常
  • 上线：大对象（300KB）→ 300ms+，异常！

单线程单次操作，无并发压力，却延迟300ms，无法接受。

🚚

类比：快递公司

想象快递公司有两个优化规则：

  • Delay Ack：收到快递后不立即回复，等有其他快递顺路一起回复
  • Nagle：小包裹不急着发，等前面包裹确认收到再一起发

如果两个规则同时启用，就会出现：两边互相等，都不发货！

2 什么是 Delay Ack？

TCP 是可靠传输，收到包后需要回复 ACK。但频繁回复 ACK 浪费带宽。

📝 Delay Ack 机制
  1. 收到包后不立即回复，等待一小段时间（如 40ms）
  2. 如果有其他数据发往对方，就搭顺风车（ACK 跟着数据一起发）
  3. 如果凑够 2 个 ACK，立即发送
  4. 如果超过 40ms 还是没有数据，也单独发送

3 什么是 Nagle 算法？

Nagle 算法用于优化小数据包的发送，减少网络开销。

📝 Nagle 算法逻辑
if 要发送的数据 >= MSS:
    立即发送
else:
    if 还有未确认的数据:
        等待确认后再发
    else:
        立即发送
📦

一句话解释

小包裹不急发，等前面的确认到了再一起发。

这样可以减少网络中小包的数量，节省带宽（早年带宽宝贵）。

⚠️ 默认开启

Nagle 算法默认是开启的，几乎所有操作系统都默认启用。

4 两者相遇的悲剧

当 Nagle 算法遇到 Delay Ack，就会产生200-500ms的延迟：

经典案例：99,900 bytes vs 100,000 bytes

📊 数据对比
  • 99,900 bytes → 速度 5.2MB/s（正常）
  • 100,000 bytes → 速度 2.7MB/s（慢一倍！）

原因：100,000 = 69×1448 + 88，最后 88 字节触发 Nagle 等待。

❶ 99,900 字节

68个整包 + 1436 字节 → 68是偶数，立即发送，Delay Ack 凑够2个也立即回复。

❷ 100,000 字节

69个整包 + 88 字节 → 最后一个包太小，Nagle 等待前面的 ACK；服务器 Delay Ack 只收到一个包，超时才回复。

5 解决方案

✅ 解决方案
  1. 禁用 Nagle 算法：设置 TCP_NODELAY
  2. 合并 HTTP 请求：禁用 Expect 头
  3. 调整超时参数：减少 Delay Ack 等待时间

代码示例

// Curl 示例：禁用 Expect 头
struct curl_slist *list = NULL;
list = curl_slist_append(list, "Expect:");

// 根据请求大小决定是否禁用 Nagle
if (request.size() < 1024) {
    curl_easy_setopt(curl, CURLOPT_TCP_NODELAY, 1);  // 小包：禁用
} else {
    curl_easy_setopt(curl, CURLOPT_TCP_NODELAY, 0);   // 大包：启用
}
🔧 其他语言
  • Go: conn.(*net.TCPConn).SetNoDelay(true)
  • Java: socket.setTcpNoDelay(true)
  • Python: socket.setsockopt(socket.IPPROTO_TCP, socket.TCP_NODELAY, 1)

核心知识点

概念说明
Delay Ack收到包后延迟回复ACK，等顺风车或超时
Nagle 算法小包等待前面的ACK确认后再发
悲剧场景Nagle 等 ACK，Delay Ack 等数据，互相等待
解决方案禁用 Nagle (TCP_NODELAY)
适用场景高频小请求（如 RPC、HTTP）、需要低延迟的场景
