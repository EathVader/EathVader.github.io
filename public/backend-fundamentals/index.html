1 网络基础 / Networking Basics

后端工程师的基本功由三大支柱构成：网络协议（数据如何传输）、数据库事务（数据如何保持一致）、操作系统（程序如何被调度和管理）。三者环环相扣。

1.1 OSI 七层模型与路由

计算机网络通信遵循 OSI 七层模型，将复杂的通信过程分层解耦。路由（Routing）发生在第三层（网络层），核心任务是将数据包从源节点以最优路径送达目标节点。

OSI 七层模型：每层各司其职，路由发生在第三层
📮

类比 / Analogy

路由就像邮局系统——IP 地址是邮编，路由器是邮局，每个邮局根据目的地邮编决定把包裹转发到哪个下一站。

1.2 IP 地址

  • IPv4：32 位地址，约 43 亿个，已经不够用
  • IPv6：128 位地址，地址空间几乎无限，是未来趋势

1.3 网络质量指标

指标含义影响场景
带宽 Bandwidth单位时间内能传输的数据量（bps）文件下载、视频流
延迟 Latency / RTT数据从发送到接收的时间差视频会议、在线游戏
💡 带宽 ≠ 延迟

类比一个路口：每秒通过 1 辆车（带宽不变），但突然来了 100 辆车排队，每辆车的等待时间就变长了（延迟增加）。两者没有必然联系。

2 TCP 可靠传输 / TCP Reliable Transport

TCP 是后端开发中最核心的传输协议，提供三大保障机制：

① 可靠传输

  • 发送端给每个数据包分配递增的 sequence number
  • 接收端收到后回复 ACK，告知期望的下一个序号
  • 如果某个包丢失，接收端会重复发送同一个 ACK，发送端据此重传
  • 最终保证数据完整、有序到达
发送端发数据带 Seq，接收端回 ACK 确认

② 流量控制（Flow Control）

  • 采用滑动窗口机制：接收端通过 ACK 告知自己还能接收多少数据
  • 发送端不能超发，防止接收端被淹没
  • 窗口为 0 时发送端暂停，通过定时器探测窗口是否重新打开

③ 拥塞控制（Congestion Control）

  • 目标：在不堵塞网络的前提下尽量多传数据
  • 核心算法：加性增、乘性减（AIMD）——收到 ACK 则线性增速，丢包则速率减半
  • 现代 TCP 包括 Slow-start、Congestion Avoidance、Fast Retransmit、Fast Recovery 四个阶段

TCP vs UDP

对比项TCPUDP
连接建立三次握手无需握手
可靠性保证有序、完整不保证
流量/拥塞控制有无

💡 选择原则

需要可靠性优先选 TCP；需要低延迟且能容忍少量丢包选 UDP。很多基于 UDP 的协议（如 QUIC）会自行实现可靠性机制。

3 数据库事务 / Database Transactions

3.1 ACID 四大特性

事务是数据库操作的基本单元，必须满足：

特性含义通俗解释
A — 原子性事务中的操作要么全部成功，要么全部回滚"要么全做，要么全不做"
C — 一致性事务完成后数据库从一个合法状态到另一个合法状态所有约束不被破坏
I — 隔离性并发事务之间互不干扰中间状态对其他事务不可见
D — 持久性事务提交后数据永久保存即使崩溃，已提交数据不丢失
🏦

类比 / Analogy

银行转账：A 给 B 转 100 元。原子性：A 扣钱和 B 加钱必须同时成功或同时失败。一致性：转账前后总金额不变。隔离性：转账过程中其他人查余额看不到中间状态。持久性：转账成功后即使银行系统崩溃，记录也不会丢。

3.2 并发异常问题

不加控制的并发访问会产生四类问题：

脏读
读到未提交数据
不可重复读
两次读结果不同
幻读
多出新行
丢失修改
修改被覆盖
⚠️ 脏读示例

事务 A 修改了一行数据但还没提交，事务 B 读到了这个修改。如果事务 A 随后回滚，B 读到的就是"脏数据"——一个从未真正存在过的值。

4 隔离级别与锁 / Isolation Levels & Locks

4.1 四种隔离级别

从低到高，隔离性越强，并发性能越低：

隔离级别脏读不可重复读幻读说明
Read Uncommitted✅允许✅允许✅允许最低隔离，几乎不用
Read Committed❌防止✅允许✅允许Oracle 默认
Repeatable Read❌防止❌防止✅允许MySQL InnoDB 默认
Serializable❌防止❌防止❌防止完全串行，性能最差

4.2 锁机制

隔离级别通过锁来实现：

  • 共享锁（S 锁）：读操作使用，允许多个事务同时读，但阻止写
  • 排他锁（X 锁）：写操作使用，独占资源，阻止其他一切读写
  • 更新锁（U 锁）：介于两者之间，用于"先读后写"场景，防止死锁
三种锁：共享锁允许并发读，排他锁完全独占

5 索引 / Indexing

索引是提升查询性能的关键手段，本质是一种有序的数据结构（B+树、哈希等）。

B+ 树：非叶节点做索引，叶子节点存数据并串联

应该建索引的列

  • 主键列、外键列（加速 JOIN）
  • 频繁出现在 WHERE、ORDER BY、GROUP BY 中的列
  • 范围查询的列（索引天然有序）

不应该建索引的列

  • 很少被查询的列
  • 区分度极低的列（如性别，只有两个值）
  • TEXT/BLOB 等大字段
  • 写远多于读的表（索引维护成本高）
💡 索引的代价

占用额外存储空间，且每次 INSERT/UPDATE/DELETE 都需要同步维护索引。索引不是越多越好，要在读写之间找平衡。

7 同步与死锁 / Synchronization & Deadlock

7.1 同步原语

当多个线程/进程访问共享资源时，必须通过同步机制避免数据竞争：

原语原理特点
信号量 Semaphore计数器，P 操作 -1（申请），V 操作 +1（释放）控制同时访问的线程数
互斥锁 Mutex信号量特例，计数器初值为 1同一时刻只允许一个线程
读写锁允许多读并发，写者独占读多写少场景性能好

7.2 死锁

死锁的四个必要条件（缺一不可）：

互斥
资源独占
持有并等待
占着不放还要更多
不可抢占
不能强制收回
循环等待
环形等待链
死锁：A 持有 R1 等待 R2，B 持有 R2 等待 R1
💡 预防死锁

打破任意一个条件即可预防死锁。实践中最常用的是破坏循环等待——规定所有线程按固定顺序获取锁。

8 虚拟内存与 I/O / Virtual Memory & I/O

8.1 虚拟内存

操作系统通过虚拟内存机制，让程序"以为"自己拥有连续的大内存空间：

  • 逻辑地址 → 程序看到的地址（连续的）
  • 物理地址 → 内存条上的真实位置（可能不连续）
  • MMU 负责逻辑地址到物理地址的映射
  • TLB 是地址转换的高速缓存

页面置换算法

算法原理特点

⚠️ 内存抖动 Thrashing

当频繁发生缺页中断时，OS 不断在磁盘和内存之间搬运数据，系统性能急剧下降。这就是内存抖动。

8.2 I/O 多路复用

传统模型中，每个连接需要一个线程处理，开销巨大。I/O 多路复用让一个线程同时监听多个 I/O 事件：

实现平台特点
select跨平台有文件描述符数量限制（通常 1024）
epollLinux高性能，事件驱动，无数量限制
kqueueBSD/macOS类似 epoll，macOS 原生支持
💡 谁在用 I/O 多路复用？

Nginx、Redis、Node.js 等高性能服务都依赖 I/O 多路复用实现高并发。

8.3 用户态与内核态

  • 用户态：程序正常执行，只能访问自己的内存空间
  • 内核态：通过系统调用进入，可以访问所有硬件和内存
  • 系统调用会触发软中断，从用户态切换到内核态，完成后再切回

中断优先级：机器错误 > 时钟 > 磁盘 > 网络设备 > 终端 > 软件中断

9.1 磁盘调度算法

磁盘访问延迟中，寻道时间是主要瓶颈：

算法原理

9.3 Linux 文件权限

权限由 10 位标志表示，如 -rwxr-xr--：

类型 | 所有者 | 同组 | 其他
 -   |  rwx   | r-x  | r--
     |  7     | 5    | 4     → chmod 754
  • r=4, w=2, x=1，相加得到数字权限
  • chmod 755 file — 所有者读写执行，其他人读和执行
  • chown user file — 改变文件所有者
  • chgrp group file — 改变文件所属组
