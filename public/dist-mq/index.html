1 核心概念

消息队列（MQ）本质上是一个基于磁盘持久化的 FIFO 队列数据结构，用于在分布式系统中实现进程间的异步通信。它充当"蓄水池"角色——当洪峰数据来临时，大量消息堆积在 MQ 中，后端按自身消费速度读取。

三大核心场景

解耦
下游自行订阅
异步
加快主链路响应
削峰填谷
平滑瞬时写压力

使用代价

⚠️ 引入 MQ 的代价
  • 系统可用性降低：MQ 本身成为新的故障点
  • 系统复杂性增加：需额外考虑一致性、幂等、可靠传输、顺序性等问题

4 可靠性保证

消息不丢失

消息丢失可能发生在三个环节：

生产者丢数据
MQ 丢数据
消费者丢数据

Kafka 方案

环节解决方案
生产者acks=all（所有 ISR 副本确认）+ retries=MAX
MQ 端replication.factor > 1 + min.insync.replicas > 1
消费者关闭自动提交 offset，改为手动提交

RabbitMQ 方案

环节解决方案
生产者使用 confirm 模式，Broker 返回 Ack/Nack
MQ 端开启持久化（durable=true + deliveryMode=2）
消费者关闭自动确认，采用手动确认（manual ack）

幂等性（消息不重复消费）

⚠️ 重复消费的根因

消费者处理完消息后的确认信息（ACK/offset 提交）因网络故障未送达 MQ，导致 MQ 重新投递。

场景解决方案
数据库 insert利用唯一主键约束，重复插入会主键冲突
Redis set天然幂等，set 多次结果一致
通用方案用 Redis 维护消费记录，消费前查询是否已存在

顺序性保证

📌 核心思路

保证需要有序的消息进入同一个队列（Partition/Queue），由同一个消费者顺序消费。Kafka 通过 Key 的 hash 将相关消息路由到同一 Partition。

5 Kafka 性能监控

Broker 核心指标

指标含义告警建议
UnderReplicatedPartitions未充分复制的分区数> 0 持续时应告警
ActiveControllerCount活跃 Controller 数量必须为 1，为 0 立即告警
OfflinePartitionsCount无 Leader 的离线分区数> 0 立即告警
UncleanLeaderElections不干净的 Leader 选举> 0 必须告警（数据丢失）
BytesIn/OutPerSec入站/出站字节速率监控网络瓶颈

Consumer 关键指标

指标含义
ConsumerLag / MaxLag消费者 offset 与生产者 offset 的差值，最关键的消费者指标
BytesPerSec消费字节速率
MinFetchRate最小拉取速率，趋近 0 可能是消费者故障
💡 ConsumerLag 处理

持续高 lag 说明消费者过载，可增加消费者数量 + 增加 Partition 数量来提升并行度。

主机级别指标

  • Page Cache 读取比率：< 80% 应考虑扩容 Broker
  • 磁盘使用率：磁盘满则服务失败，需监控增长趋势
  • JVM GC：GC 暂停过长会导致 ZooKeeper 会话超时

6 微信 MQ 实践

架构概述

微信 MQ 是自研的异步队列组件，分为 MQ（任务持久化和调度框架）和 Worker（任务处理框架）两部分。关注单机性能，任务单机持久化、本机消费。

MQ 2.0 三大优化

优化方向问题解决方案
跨机消费部分 IDC 网络质量差导致积压去中心化、自适应的弹性消费网络（拉模式）
类 MapReduce群聊批量投递代码极其复杂封装通用 MR 过程和并发调度，提供并发池隔离
过载保护后端处理不过来前向限速（CPU/成功率）+ 后向限速（RPC 访问量）

过载保护机制

前向限速
基于 CPU 使用率
前向限速
基于任务成功率
后向限速
基于后端 RPC 访问量
✅ 效果

实现 IDC 级别容灾，即使只剩一个 IDC 可用，推送质量不受影响。
