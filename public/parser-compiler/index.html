编译器原理与 Parser 实战

原创 · 2026-04-02 · 阅读约 25 分钟

为什么需要 Parser？

很多人第一次接触 Parser 时都会问：正则表达式不是能匹配一切文本吗？ 让我们从一个经典场景说起。

Regex 的局限：Nginx 配置文件的困境

假设你要解析一段 Nginx 配置文件：

nginx.conf
server {
    listen 80;
    server_name example.com;

    location /api {
        proxy_pass http://backend:8080;
        if ($request_method = POST) {
            return 405;
        }
    }

    location /static {
        root /var/www/html;
    }
}

这段配置有以下几个特点让正则表达式无能为力：

  • 嵌套结构：server 块内嵌套 location 块，location 内又嵌套 if 块
  • 上下文依赖：同一个指令在不同块中的含义可能不同
  • 递归定义：块可以无限嵌套
⚠️ 核心结论：正则表达式对应的是正则语言（Regular Language），而嵌套结构属于上下文无关语言（Context-Free Language）。正则表达式无法处理任意深度的嵌套——这是计算理论中的经典结论（泵引理可证）。

编译器架构：从源码到目标代码

一个完整的编译器通常包含以下阶段：

1. 词法分析（Lexical Analysis）

词法分析器（Lexer / Scanner）将源代码的字符流切分成一个个有意义的词法单元（Token）。每个 Token 包含类型和值：

输入: x = 42 + y
输出 Token 流:
[IDENTIFIER, "x"]
[ASSIGN, "="]
[NUMBER, "42"]
[PLUS, "+"]
[IDENTIFIER, "y"]

2. 语法分析（Syntax Analysis）

语法分析器（Parser）接收 Token 流，根据文法规则构建抽象语法树（Abstract Syntax Tree, AST）。AST 去除了语法细节（如分号、括号），只保留程序的逻辑结构：

3. 语义分析（Semantic Analysis）

语义分析阶段检查程序的语义正确性：

  • 类型检查：确保操作数类型匹配
  • 作用域分析：变量是否已声明
  • 常量折叠：编译期计算常量表达式

BNF / CFG 文法详解

Parser 的核心是文法（Grammar）。最常用的是 BNF（巴科斯范式）和 CFG（上下文无关文法）。

BNF 基本语法

符号含义示例
::=定义为<expr> ::= <term>
|或（选择）<digit> ::= 0 | 1 | 2
< >非终结符<expression>
引号内容终结符（字面量）"+", "if"
*零次或多次（EBNF）<stmt>*
[ ]可选（EBNF）["else" <stmt>]

四则运算的 BNF 定义

arithmetic.bnf
<expr>   ::= <expr> "+" <term>   /* 加法 */
          |  <expr> "-" <term>   /* 减法 */
          |  <term>               /* 递归终止 */

<term>   ::= <term> "*" <factor> /* 乘法 */
          |  <term> "/" <factor> /* 除法 */
          |  <factor>             /* 递归终止 */

<factor> ::= "(" <expr> ")"      /* 括号 */
          |  NUMBER               /* 数字 */
💡 为什么需要分层？ 通过 <expr> → <term> → <factor> 的分层，自然实现了运算符优先级：乘除（term）比加减（expr）绑定更紧，括号（factor）优先级最高。

左递归问题

上面的文法是左递归的（<expr> 的第一个产生式以 <expr> 开头）。LL 解析器（递归下降）无法处理左递归，需要改写为右递归或使用 LR 解析器。

Parser 生成器对比

手写 Parser 工作量大且容易出错，实际项目中通常使用 Parser 生成器（Parser Generator）。

Flex + BisonC/C++LR(1)工业标准，性能极高，Linux/Unix 生态标配
JisonJavaScriptLR(1)/LALRBison 的 JS 移植版，浏览器/Node.js 均可用
ANTLRJava/多语言ALL(*)支持左递归，目标语言丰富，生态完善

实战：用 Jison 写一个 Todo 文件 Parser

我们来设计一个 Todo 文件格式，并用 Jison 实现完整的 Parser。

Todo 文件格式设计

tasks.todo
# 我的任务列表
project "工作" {
    task "完成 Parser 教程" @high due 2026-04-10
    task "写单元测试" @medium due 2026-04-15
    task "部署到 GitHub Pages" @low
}

project "生活" {
    task "买菜" @low
    task "健身" @medium due 2026-04-03
}

第一步：定义词法规则（Lexer）

todo.jison — Lex 部分
%lex

%%
\s+                   /* 跳过空白 */
"//".*                /* 跳过单行注释 */
"project"             return 'PROJECT'
"task"                return 'TASK'
"due"                 return 'DUE'
"@"[a-z]+             return 'PRIORITY'
[0-9]{4}-[0-9]{2}-[0-9]{2}  return 'DATE'
\"[^\"]*\"            return 'STRING'
"{"                   return '{'
"}"                   return '}'
<<EOF>>               return 'EOF'
/lex

第二步：定义语法规则（BNF）

todo.jison — Grammar 部分
%start todo

%%

todo
    : projects EOF
        { return $1; }
    ;

projects
    : project
        { $$ = [$1]; }
    | projects project
        { $$ = $1.concat($2); }
    ;

project
    : PROJECT STRING '{' tasks '}'
        { $$ = { type: 'project', name: $2, tasks: $4 }; }
    ;

tasks
    : %empty
        { $$ = []; }
    | tasks task
        { $$ = $1.concat($2); }
    ;

task
    : TASK STRING PRIORITY
        { $$ = [{ type: 'task', title: $2, priority: $3.slice(1) }]; }
    | TASK STRING PRIORITY DUE DATE
        { $$ = [{ type: 'task', title: $2, priority: $3.slice(1), due: $5 }]; }
    ;

第三步：生成 Parser 并渲染 AST

main.js — 使用生成的 Parser
const parser = require('./todo-parser');

const source = `
project "工作" {
    task "完成 Parser 教程" @high due 2026-04-10
    task "写单元测试" @medium
}
`;

const ast = parser.parse(source);
console.log(JSON.stringify(ast, null, 2));

// 渲染为 HTML
function renderTodo(ast) {
    return ast.map(project => `
        <div class="project">
            <h2>${project.name}</h2>
            <ul>
                ${project.tasks.map(task => `
                    <li class="priority-${task.priority}">
                        ${task.title}
                        ${task.due ? ` <span class="due">截止: ${task.due}</span>` : ''}
                    </li>
                `).join('')}
            </ul>
        </div>
    `).join('');
}

总结

  • 正则表达式无法处理嵌套结构，这是计算理论的限制，不是工具的问题
  • 编译器核心流程：词法分析 → 语法分析 → 语义分析 → 代码生成
  • BNF/CFG 是描述文法的标准方式，分层定义可自然实现运算符优先级
  • Parser 生成器大幅提升开发效率：Flex+Bison（C/C++）、Jison（JS）、ANTLR（多语言）
  • 实战中，定义清晰的 Lexer 规则 + BNF 语法 即可生成可用的 Parser
