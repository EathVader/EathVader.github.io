微服务资源隔离之域名拆分

原创 · 2026-04-02 · 阅读约 15 分钟

问题背景

在微服务架构中，一个常见的反模式是：所有服务共用同一个域名和 API 网关。这在初期开发方便，但随着业务增长会暴露严重问题。

典型案例：非核心流量突发波及核心链路

问题场景
api.company.com
├── /order/*      ← 核心业务（下单、支付）
├── /user/*       ← 核心业务（登录、鉴权）
├── /report/*     ← 非核心（数据报表）
├── /export/*     ← 非核心（数据导出）
└── /webhook/*    ← 非核心（第三方回调）

某天，报表服务被大量并发查询，导出服务被批量触发...
结果：网关连接池耗尽 → 下单接口超时 → 核心业务受损！
⚠️ 核心问题：当核心业务和非核心业务共享同一域名时，它们共享同一套网络资源（DNS、CDN、网关、负载均衡、连接池）。非核心业务的流量突峰会直接挤占核心业务的资源。

域名拆分原则

原则一：相同功能的 API 划分到同一域名

按业务领域而非技术层级拆分。例如：

  • 所有订单相关 API → order.company.com
  • 所有用户相关 API → user.company.com
  • 所有报表相关 API → report.company.com

原则二：区分核心/非核心业务

分类特征示例SLA 要求
核心业务直接影响收入/用户体验下单、支付、登录99.99%
重要业务影响用户体验但不直接损失搜索、推荐、消息99.9%
非核心业务可延迟、可降级报表、导出、通知99%

原则三：独立资源隔离

不同域名对应独立的：

  • 网关实例：不同的 Nginx/APISIX/Kong 实例
  • 负载均衡：独立的 LB 配置和健康检查
  • 连接池：数据库连接池、上游服务连接池隔离
  • 限流策略：各自独立的限流阈值
  • 监控告警：独立的 SLO 和告警规则

命名规则：NAME.TAG.host.com

统一的命名规范有助于管理和自动化：

命名格式
格式: {NAME}.{TAG}.{BASE_DOMAIN}

NAME  - 业务名称（order, user, report, gateway）
TAG   - 环境/用途标签（api, admin, internal, public）
BASE  - 基础域名

示例:
order.api.company.com      # 订单 API（对外）
order.admin.company.com    # 订单管理后台
user.api.company.com       # 用户 API（对外）
report.internal.company.com # 报表服务（内部）
gateway.public.company.com  # 公共网关
标签用途访问范围示例
api对外 REST API公网order.api.company.com
admin管理后台内网/VPNorder.admin.company.com
internal内部服务调用内网report.internal.company.com
public公共资源公网static.public.company.com
grpcgRPC 服务内网order.grpc.company.com

带宽管理策略

核心业务：高基础带宽保障

  • 基础带宽：按峰值流量的 120% 配置，确保高峰期不瓶颈
  • 弹性扩容：自动根据 QPS 扩容网关实例
  • 优先级队列：核心接口在网关层设置更高优先级

非核心业务：动态调整

  • 基础带宽：按平均流量配置，不预留过多
  • 动态限流：当核心业务带宽使用率高时，自动收紧非核心限流
  • 降级策略：极端情况下可完全关闭非核心服务
Nginx 限流配置示例
# 核心业务网关 - 宽松限流
limit_req_zone $binary_remote_addr zone=core:10m rate=1000r/s;
server {
    listen 80;
    server_name core-api.company.com;

    location /order/ {
        limit_req zone=core burst=200 nodelay;
        proxy_pass http://order-service;
    }
}

# 非核心业务网关 - 严格限流
limit_req_zone $binary_remote_addr zone=batch:10m rate=100r/s;
server {
    listen 80;
    server_name batch.company.com;

    location /export/ {
        limit_req zone=batch burst=20 nodelay;
        proxy_pass http://export-service;
    }

    location /report/ {
        limit_req zone=batch burst=50 nodelay;
        proxy_pass http://report-service;
    }
}

实践注意事项

💡 拆分不是越多越好：域名拆分会增加 DNS 解析次数、SSL 证书管理成本、运维复杂度。建议按业务重要程度分为 2-3 层即可。
注意事项说明建议
DNS 解析延迟每个新域名增加一次 DNS 查询使用 DNS 预解析
SSL 证书管理每个域名需要独立证书使用通配符证书或 ACME 自动续期
CORS 跨域不同域名间请求触发 CORS配置正确的 CORS 头，或使用同域不同路径
Cookie 共享不同域名间 Cookie 不共享使用 Token 鉴权，或设置 .company.com 域
监控告警需要为每个域名配置独立监控统一监控平台 + 分级告警策略
灰度发布拆分后灰度策略更复杂使用网关层流量染色 + 路由规则
⚠️ 常见误区：域名拆分只是资源隔离的第一步。完整的隔离还需要：独立的数据库连接池、独立的消息队列消费组、独立的缓存实例。否则即使网关隔离了，下游资源仍可能被挤占。

总结

  • 核心问题：核心/非核心业务共用域名 → 共享资源 → 非核心流量突增波及核心链路
  • 拆分原则：按业务领域分组，按重要程度分级，相同功能 API 归到同一域名
  • 命名规范：NAME.TAG.host.com，标签区分用途（api/admin/internal/public）
  • 架构设计：核心/重要/非核心各自独立网关，资源完全隔离
  • 带宽策略：核心业务高基础保障，非核心业务动态调整 + 可降级
  • 注意事项：DNS 延迟、SSL 管理、CORS、Cookie 共享、监控告警
