1 B+树基础 / B+ Tree Basics

📚

类比 / Analogy

B+树就像一本多级目录的字典：最上面是总目录（根节点），中间是分类目录（内部节点），最下面是实际的词条（叶节点）。你通过目录一层层定位，最终找到想要的词条。

B+树：数据只存在叶节点，内部节点仅做索引
📐 B+树的核心性质
  • 所有路径等长（平衡树）→ 查找时间 O(log n)
  • 数据只存叶节点，内部节点仅存索引键
  • 每个节点至少半满（M/2 个键值对）
  • 叶节点通过链表相连 → 高效范围查询

2 并发正确性要求 / Correctness Requirements

不管用什么并发机制，都必须满足以下四个要求：

编号要求说明类比
R.1不读到中间状态读操作不会看到写了一半的数据🚫 不会看到只写了一半的文件
R.2不丢失已有数据数据被分裂移动时，读操作仍能找到它🔍 搬家后邮件仍能送达
W.1写写互斥两个写操作不会同时修改同一数据✏️ 两人不能同时编辑同一行
D.1无死锁不会出现循环等待🔄 不会互相等对方让路
🏦

类比 / Analogy

想象一个银行系统：R.1 = 查余额时不会看到转账转了一半的金额；R.2 = 账户迁移分行后仍能查到；W.1 = 两笔转账不会同时扣同一笔钱；D.1 = 不会两个柜员互相等对方处理完。

3 粗粒度锁 (MySQL 5.6) / Coarse-Grained Locking

MySQL 5.6 使用两种粒度的锁：索引级 S/X 锁 + 页级 S/X 锁。

读操作

/* 读操作 */
1. SL(index)          // 对整个索引加共享锁
2. 遍历到叶节点        // Travel down to leaf
3. SL(leaf)           // 对叶节点加共享锁
4. SU(index)          // 释放索引锁
5. 读取叶节点          // Read leaf
6. SU(leaf)           // 释放叶节点锁

悲观写操作

/* 悲观写操作 */
1. XL(index)          // 对整个索引加互斥锁 🔒
2. 遍历到叶节点        // Travel down to leaf
3. XL(leaf)           // 对叶节点加互斥锁
4. 修改树结构          // Modify tree structure
5. XU(index)          // 释放索引锁
6. 修改叶节点          // Modify leaf
7. XU(leaf)           // 释放叶节点锁
悲观写操作锁住整个索引，所有其他读写操作都被阻塞

乐观写操作

大多数写操作不会触发分裂/合并，所以可以乐观地只加 S 锁遍历，到叶节点再加 X 锁：

/* 乐观写操作 */
1. SL(index)          // 只加共享锁！
2. 遍历到叶节点
3. XL(leaf)           // 叶节点加互斥锁
4. SU(index)
5. 修改叶节点
6. XU(leaf)
💡 乐观 vs 悲观

乐观写：假设不会分裂，用 S 锁遍历 → 读写可并发
悲观写：假设会分裂，用 X 锁锁住整棵树 → 所有操作串行
MySQL 5.6 优先尝试乐观写，失败才退回悲观写。

4 只锁修改分支 / Lock Only Modified Branch

1976 年的论文提出：写操作不需要锁住整棵树，只需要锁住被修改的分支。核心概念是 Safe Node：

🛡️ Safe Node（安全节点）

如果当前操作不会影响祖先节点，则该节点是安全的：

  • 插入：键值对数量 < M → 不会分裂 → Safe ✅
  • 删除：键值对数量 > M/2 → 不会合并 → Safe ✅

遇到 Safe Node 时，可以释放所有祖先节点的锁！

/* 写操作：只锁修改分支 */
1.  current = root
2.  XL(current)
3.  While current is not leaf {
4.      XL(current->son)
5.      current = current->son
6.      If current is safe {
7.          XU(所有祖先节点)  // 释放！
8.      }
9.  }
10. 修改叶节点和上层节点
11. XU(current) + XU(祖先)
遇到 Safe Node 后释放祖先锁，其他线程可以并发访问其他分支
⚠️ 问题 / Problem

写操作在到达 Safe Node 之前，对途经的节点加的是 X 锁，仍然会阻塞读操作。能不能更好？

5 SX 锁横空出世 / SX Lock

🎮 交互演示：锁模式可视化

观察不同操作获取的锁类型：

选择操作类型查看锁...

1977 年，Rudolf Bayer 教授提出了介于 S 锁和 X 锁之间的 SX 锁：

请求 \ 持有SSXX
S✅ 兼容✅ 兼容❌ 冲突
SX✅ 兼容❌ 冲突❌ 冲突
X❌ 冲突❌ 冲突❌ 冲突
🚧

类比 / Analogy

S 锁 = 参观者通行证（多人可同时参观）
SX 锁 = 施工预告牌（参观者仍可通行，但其他施工队不能进入）
X 锁 = 施工围挡（所有人都不能进入）
SX 锁在确定要修改时可以升级为 X 锁。

/* 使用 SX 锁的写操作 */
1.  current = root
2.  SXL(current)          // SX 锁！读操作不被阻塞
3.  While current is not leaf {
4.      SXL(current->son)
5.      current = current->son
6.      If current is safe {
7.          SXU(祖先节点)
8.      }
9.  }
10. XL(修改的节点)         // SX → X 升级
11. 修改分支
12. XU(所有锁)

6 缓存一致性问题 / Cache Coherence Problem

即使用了 SX 锁，每次加锁都要修改锁变量，这会导致多核 CPU 上严重的缓存一致性开销。

📋

类比 / Analogy

4 个人各有一份文件副本。每当一个人在自己的副本上签名（加锁），其他人的副本就全部作废，必须重新复印。即使他们看的是不同的页面！

CPU 2 修改 n1 的锁 → 所有其他 CPU 缓存中的 n1 失效
📊 NUMA 架构下的延迟

跨 socket 的缓存同步延迟高达 200-300ns，而普通内存访问仅 50-100ns。加锁导致的缓存失效开销远超访存本身！

解决方向：减少加锁频率 → 自底向上加锁，只对真正修改的节点加锁。

7 Blink 树 / Blink Tree (1981)

Blink 树的核心创新：为每个节点添加一个右指针 (right link)，使得无锁遍历 + 自底向上加锁成为可能。

右指针让读操作在无锁状态下也能追踪被分裂移走的数据

读操作：无锁遍历！

/* Blink 树读操作 — 不需要加锁！ */
1. current = root
2. While current is not leaf {
3.     current = scanNode(current, v)  // 可能向右走
4.     current = current->son
5. }
6. 在叶节点中查找（可能继续向右走）

/* scanNode：如果目标在右兄弟，就向右移动 */
func scanNode(node, v) {
    If node->right->key[0] <= v
        return scanNode(node->right, v)
    return node
}

写操作：自底向上加锁

/* Blink 树写操作 */
1.  无锁遍历到叶节点（同读操作）
2.  将路径压入 stack
3.  XL(leaf)
4.  If leaf is safe:
5.      insert + XU(leaf)  // 完成！
6.  Else:
7.      分裂叶节点，创建新节点
8.      设置右指针
9.      parent = pop(stack)
10.     XL(parent)         // 向上加锁
11.     XU(leaf)
12.     重复插入过程...
🎯 Blink 树的突破
  • 读操作完全无锁 → 读-读、读-写零冲突
  • 自底向上加锁 → 只锁真正需要修改的节点
  • 右指针 → 分裂时数据只往右移，读操作通过右指针追踪
  • 加锁顺序"自左向右，自底向上" → 无死锁

8 OLFIT 树：版本号 / Version-based Optimistic (2001)

Blink 树假设"读写节点是原子的"，这在现实中不成立。OLFIT 树引入版本号解决这个问题。

📝

类比 / Analogy

就像 Google Docs 的版本号：你开始读文档时记下版本号 v5，读完后发现变成了 v6 → 说明有人改过了，你需要重新读。

📝 记录版本号
version = v
→
读取节点内容
→
检查版本号
changed?
→ 变了
🔄 重试
/* 写操作：修改节点时递增版本号 */
1. XL(node)
2. 修改节点内容
3. version++              // 递增版本号
4. XU(node)

/* 读操作：用版本号检测并发修改 */
1. v = node.version       // 记录版本号
2. 读取节点内容
3. If node.locked → 重试   // 正在被写
4. If node.version != v → 重试  // 已被修改
🔑 OLFIT 的核心思想
  • 写操作通过锁互斥 + 版本号通知读操作
  • 读操作完全无锁，通过版本号检测是否读到脏数据
  • 读操作不修改任何共享变量 → 零缓存一致性开销
  • 结合 Blink 树的右指针 → 完整的高性能并发 B+树

9 MySQL 5.7 的方案 / MySQL 5.7 Approach

MySQL 5.7 综合了前面的多种优化：SX 锁 + 只锁修改分支。

操作索引锁非叶节点叶节点
读操作SSS
乐观写SSX
悲观写SXX（修改分支）X
/* MySQL 5.7 悲观写操作 */
1. SXL(index)             // SX 锁：不阻塞读操作！
2. XL(修改分支)            // 只锁被影响的分支
3. XL(leaf)
4. 修改树结构
5. XU(修改分支)
6. SXU(index)
7. 修改叶节点
8. XU(leaf)
🤔 为什么 MySQL 没有用 Blink/OLFIT？

MySQL 的 B+树远比学术论文中的复杂：

  • 需要支持 Spatial Index 等复杂索引
  • 需要支持 purging 线程优先级
  • 存在锁住"左-当前-右"三个节点的场景
  • 需要支持 modify_prev/search_prev 等回溯操作

这些复杂需求使得完全无锁的方案难以直接应用。

10 总结与演进 / Summary

40 年演进路线

1970s
粗粒度锁 — 索引级 X 锁，写操作锁住整棵树
1976
只锁修改分支 — Safe Node 概念，释放安全祖先的锁
1977
SX 锁 — 写操作遍历时不阻塞读操作
1981
Blink 树 — 右指针 + 无锁读 + 自底向上加锁
2001
OLFIT 树 — 版本号乐观读，零缓存一致性开销
2012
Masstree — B+树 + Trie 混合，集大成者
至今
Lock-Free B+树 — Bw-Tree 等完全无锁设计

核心优化思路

优化方向手段效果
减小锁粒度索引级 → 分支级 → 节点级减少阻塞范围
减少锁冲突SX 锁、乐观写读写可并发
减少加锁频率无锁读 + 版本号消除缓存一致性开销
改变加锁方向自底向上 + 右指针只锁真正修改的节点
