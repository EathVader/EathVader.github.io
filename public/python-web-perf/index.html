Python Web 应用性能调优实战

来源：内部技术分享 · 2026-04-02

背景与目标

为了快速上线，早期代码往往怎么方便怎么来，留下很多性能隐患。Python 因为 GIL 的原因，在性能上有天然劣势——即使用了 gevent/eventlet 协程方案，也很容易因为耗时的 CPU 操作阻塞整个进程。

调优目标：
  • 用最少的机器处理相同流量，目标关闭 20% 的 stateless webserver
  • 尽量在框架代码上改动，不动业务逻辑代码
  • 低风险（历史经验告诉我们：动态一时爽，重构火葬场）

治标：定位慢接口

常见场景：开开心心做完一个 feature，sandbox 测试没问题，上线后 server load 飙升，各种 timeout 来了。

最有效的指标

看每个接口的 avg_latency × req_count，得到每个接口在一段时间内的总耗时。在柱状图上最长的那块就是对性能影响最大的接口。

计算方法
总耗时 = 平均延迟 × 请求数
Total Time = avg_latency × req_count

进一步的调试靠 cProfile 和读代码。但这种方式只能治标——调试个别慢的业务接口，无法提高整体性能。

治本：火焰图定位瓶颈

希望对运行时进程状态打 snapshot，每次快照记录当前的函数调用栈，叠合多次采样，出现次数多的函数必然就是瓶颈所在。这就是 Brendan Gregg 的 flamegraph 思想。

使用 pyflame

Uber 开源的 pyflame 可以非侵入式地对运行中的 Python 进程做 snapshot，输出成 SVG 火焰图。

Shell
pyflame -s 60 -r 0.01 ${pid} | flamegraph.pl > myprofile.svg
参数说明
-s 60总采样时间 60 秒
-r 0.01以 0.01 秒的频率做采样

火焰图解读

  • 横条越长：被采样到的次数越多，表示耗时占比越大
  • 从下往上：可以看到每一层上的函数耗时分布
  • IDLE 部分：pyflame 只能捕获当前获取了 GIL 的代码，IDLE 包括 IO wait、C 编写部分、进程空闲时间
核心优势：通过 pyflame 可以很快对进程运行时耗时分布有个大概的感觉，即使你完全不了解业务逻辑。

重构实战

架构背景

线上 Web 应用架构：

Web 端和 API Server 基于 Flask，后面是几组业务不同的 RPC Server，通过 msgpack 通信。为了方便，RPC Server 也是基于 Flask 的。

优化 1：RPC Server — 干掉 Flask

通过 pyflame 调试发现：Flask 的 overhead 很高。在 RPC 层，一些接口实际业务代码的采样次数只有总采样的 1/6 左右，其余都耗在了 Flask 层。

决策：RPC 层不处理 Web 逻辑，Flask 完全用不到，可以干掉。本着不动业务代码和低风险的原则，保持 HTTP + msgpack，直接实现 WSGI。
Python — 极简 WSGI RPC Server
import msgpack

def rpc_application(environ, start_response):
    """直接实现 WSGI，替代 Flask"""
    if environ['REQUEST_METHOD'] == 'POST':
        content_length = int(environ.get('CONTENT_LENGTH', 0))
        body = environ['wsgi.input'].read(content_length)
        
        # msgpack 解码请求
        data = msgpack.unpackb(body, raw=False)
        
        # 路由到对应的业务函数
        func_name = data.pop('func')
        result = dispatch(func_name, data)
        
        # msgpack 编码响应
        response_body = msgpack.packb(result, use_bin_type=True)
        
        start_response('200 OK', [
            ('Content-Type', 'application/msgpack'),
            ('Content-Length', str(len(response_body)))
        ])
        return [response_body]
    
    start_response('405 Method Not Allowed', [])
    return [b'Method not allowed']

# 用 gunicorn 启动
# gunicorn -w 4 rpc_module:rpc_application

改完 RPC Server 入口代码不到 200 行，用 wrk 做 helloworld benchmark，并发轻松变 3 倍。

优化 2：RPC Client — 替换 requests

在 pyflame 采样图上能看到 requests 代码里的耗时很长。尝试用 geventhttpclient 替换 requests。

Shell — Benchmark 结果
完成相同请求数：
geventhttpclient 只用了 requests 1/4 的时间
(在 gevent patch 过的情况下)

踩坑：Latency 反而上升了 30%

修改完 RPC Client 上线后，server load 降得很明显，但 latency 却直接上升了 30% 多？？？

排查过程

  1. 内网流量莫名增加：拿两台机器做 A/B testing，效果很明显
  2. 怀疑 connection pool 有问题：导致 TCP 连接没有复用
  3. 抓 SYN 包：tcpdump "tcp[tcpflags] & (tcp-syn) != 0"，对比两台机器，SYN 包数目没有太大差别

根因发现

抓包过程中偶然发现：geventhttpclient 在发送 HTTP 请求时，header 和 body 竟然是用两个 packet 发送的！

而 requests 底层用的标准库 httplib，会将 header buffer 起来和 body 通过一个 packet 发出去。所以每发一次请求，geventhttpclient 会多发一个 IP + TCP header（40 字节）。

教训：40 字节看起来很小，但在高并发场景下，每个请求多发一个 packet，累积起来流量和延迟的影响非常可观。

修复

把这个问题修了，上线后 latency 立刻恢复正常。顺手把改动推到了官方。

最终成果

优化成果

  • ✅ 关闭了 30% 的 stateless server（超过原定 20% 的目标）
  • ✅ 总共动到的代码也就几百行，业务开发无感知
  • ✅ 性价比极高

经验总结

  1. 完善监控和工具链：单机 benchmark QPS 数据只能估个天花板，关键是要有完善的监控帮助快速定位问题
  2. 火焰图是利器：pyflame 等非侵入式工具能快速定位瓶颈，即使不了解业务逻辑
  3. 框架 overhead 不容忽视：Flask 等 Web 框架在 RPC 场景下 overhead 可能占大头
  4. 替换库要谨慎：即使是 benchmark 表现更好的库，也要在生产环境充分验证
  5. 抓包是排查网络问题的终极手段：tcpdump 能发现很多意想不到的问题
