1 什么是 LRU Cache？

LRU（Least Recently Used） 是一种经典的缓存淘汰策略：当缓存满时，淘汰最近最少使用的数据。

Python 3.2+ 在 functools 模块中提供了 @lru_cache 装饰器，可以一键为函数添加缓存：

Python
from functools import lru_cache

@lru_cache(maxsize=128)
def fibonacci(n):
    if n < 2:
        return n
    return fibonacci(n - 1) + fibonacci(n - 2)

# 第一次调用：计算并缓存
print(fibonacci(10))  # 55

# 第二次调用：直接从缓存读取，O(1)
print(fibonacci(10))  # 55 (缓存命中)
LRU Cache：最近使用的数据保留在左侧，最少使用的数据在右侧等待被淘汰
💡 关键特性
  • 缓存命中（Cache Hit）：参数在缓存中，直接返回结果，O(1) 时间复杂度
  • 缓存未命中（Cache Miss）：参数不在缓存中，执行函数并缓存结果
  • 缓存淘汰（Cache Eviction）：缓存满时，淘汰最近最少使用的项
  • 线程安全：Python 3.7+ 的 @lru_cache 是线程安全的

2 Gunicorn 多 Worker 模型

Gunicorn（Green Unicorn）是 Python Web 应用最常用的 WSGI HTTP 服务器。它使用 预派生（pre-fork） 模型：

Shell
# 启动 4 个 Worker 进程
gunicorn app:app --workers 4 --bind 0.0.0.0:8000
Gunicorn 预派生模型：每个 Worker 进程拥有独立的内存空间和独立的 LRU Cache
⚠️ 核心问题

Gunicorn 的每个 Worker 是一个独立的 Python 进程，拥有独立的内存空间。这意味着：

  • 每个 Worker 的 @lru_cache 是完全独立的
  • Worker A 缓存了 fibonacci(10) = 55，Worker B 不知道
  • 同一个请求被不同 Worker 处理时，会重复计算

3 陷阱：每个 Worker 独立缓存

假设你有 4 个 Worker，maxsize=128 的 LRU Cache。理论上总缓存容量是 4 × 128 = 512 条记录，但实际上：

左侧：每个 Worker 独立缓存，命中率低；右侧：共享缓存，所有 Worker 共享数据

命中率分析

假设总共有 200 个不同的请求，均匀分配到 4 个 Worker：

指标独立 LRU Cache共享缓存 (Redis)
每个 Worker 处理的唯一请求~50 个—
每个 Worker 的缓存命中低（请求分散）高（全局共享）
总缓存利用率~25%~100%
重复计算次数高低

5 Flask 示例与 curl 验证

下面是一个使用 @lru_cache 的 Flask 应用：

app.py
from flask import Flask
from functools import lru_cache
import time

app = Flask(__name__)

@lru_cache(maxsize=128)
def expensive_query(user_id):
    """模拟耗时的数据库查询"""
    time.sleep(0.5)  # 模拟 I/O 延迟
    return {"id": user_id, "name": f"User_{user_id}", "cached": True}

@app.route('/user/')
def get_user(user_id):
    start = time.time()
    result = expensive_query(user_id)
    elapsed = time.time() - start
    result["elapsed"] = round(elapsed, 3)
    return result

if __name__ == '__main__':
    app.run()

单进程测试（开发模式）

Shell
# 第一次请求：缓存未命中，耗时 ~0.5s
$ curl -w "\nTime: %{time_total}s\n" http://localhost:5000/user/1
{"id": 1, "name": "User_1", "cached": true, "elapsed": 0.502}
Time: 0.503s

# 第二次请求：缓存命中，耗时 < 0.01s
$ curl -w "\nTime: %{time_total}s\n" http://localhost:5000/user/1
{"id": 1, "name": "User_1", "cached": true, "elapsed": 0.001}
Time: 0.002s  ✅ 缓存命中！

多进程测试（Gunicorn 4 Workers）

Shell
$ gunicorn app:app --workers 4 --bind 0.0.0.0:8000

# 第一次请求：由 Worker 1 处理，缓存未命中
$ curl http://localhost:8000/user/1
{"id": 1, "elapsed": 0.503}  # Worker 1 缓存

# 第二次请求：可能由 Worker 2 处理，缓存未命中！
$ curl http://localhost:8000/user/1
{"id": 1, "elapsed": 0.501}  # Worker 2 缓存，重新计算！❌

# 第三次请求：可能又由 Worker 3 处理...
$ curl http://localhost:8000/user/1
{"id": 1, "elapsed": 0.502}  # Worker 3 缓存，又计算一次！❌

# 第四次请求：回到 Worker 1，这次命中
$ curl http://localhost:8000/user/1
{"id": 1, "elapsed": 0.001}  # Worker 1 缓存命中 ✅
🚨 问题总结

在 4 个 Worker 的情况下，同一个请求可能被重复计算 最多 4 次！每个 Worker 都有自己的缓存副本，导致：

  • 缓存命中率降低为原来的 1/4
  • 数据库查询次数增加 4 倍
  • 内存浪费：相同数据在 4 个进程中各存一份

6 解决方案：Memcached / Redis

要解决多 Worker 缓存独立的问题，需要使用进程外缓存（Out-of-Process Cache），让所有 Worker 共享同一个缓存服务。

方案一：使用 Flask-Caching + Redis

app.py (Redis 方案)
from flask import Flask
from flask_caching import Cache

app = Flask(__name__)

# 配置 Redis 缓存（所有 Worker 共享）
app.config['CACHE_TYPE'] = 'RedisCache'
app.config['CACHE_REDIS_HOST'] = 'localhost'
app.config['CACHE_REDIS_PORT'] = 6379
app.config['CACHE_REDIS_DB'] = 0
app.config['CACHE_DEFAULT_TIMEOUT'] = 300

cache = Cache(app)

@cache.cached(timeout=300, key_prefix='user_')
def expensive_query(user_id):
    """现在所有 Worker 共享同一个 Redis 缓存"""
    time.sleep(0.5)
    return {"id": user_id, "name": f"User_{user_id}"}

@app.route('/user/')
def get_user(user_id):
    result = expensive_query(user_id)
    return result

方案二：使用 Memcached

app.py (Memcached 方案)
from flask import Flask
from flask_caching import Cache
import pylibmc

app = Flask(__name__)

# 配置 Memcached 缓存
app.config['CACHE_TYPE'] = 'MemcachedCache'
app.config['CACHE_MEMCACHED_SERVERS'] = ['127.0.0.1:11211']
app.config['CACHE_DEFAULT_TIMEOUT'] = 300

cache = Cache(app)

@cache.cached(timeout=300, key_prefix='user_')

    time.sleep(0.5)
    return {"id": user_id, "name": f"User_{user_id}"}

方案三：手动使用 Redis

app.py (手动 Redis)
import redis
import json
import time

r = redis.Redis(host='localhost', port=6379, db=0)

def get_user_cached(user_id):
    key = f"user:{user_id}"
    # 先查缓存
    cached = r.get(key)
    if cached:
        return json.loads(cached)
    # 缓存未命中，查询数据库
    result = expensive_db_query(user_id)
    # 写入缓存，5 分钟过期
    r.setex(key, 300, json.dumps(result))
    return result

7 方案对比

方案跨进程共享部署复杂度性能适用场景
@lru_cache❌ 否✅ 零配置⚡ 最快（内存）单进程应用
Redis✅ 是🟡 需安装 Redis⚡ 快（网络 I/O）生产环境推荐
Memcached✅ 是🟡 需安装 Memcached⚡ 快（纯内存）简单缓存场景
进程间共享内存✅ 是🔴 复杂⚡ 快不推荐，易出错
✅ 推荐方案

生产环境使用 Flask-Caching + Redis：

  • 所有 Worker 共享同一个 Redis 实例
  • 支持缓存过期、持久化、集群
  • 配置简单，只需改几行代码
  • 支持多种缓存后端（Redis、Memcached、文件系统）

核心要点

要点说明
LRU Cache 是进程内的@lru_cache 缓存在 Python 进程内存中，不同进程无法共享
Gunicorn 是多进程的每个 Worker 是独立的 Python 进程，有独立的内存空间
缓存命中率下降Worker 数越多，缓存命中率越低（近似 1/N）
解决方案使用 Redis 或 Memcached 等外部缓存服务
Flask-Caching统一的缓存接口，支持多种后端，推荐生产使用
