NAT 穿透与 P2P 通信

原创 · 2026-04-02 · 阅读约 20 分钟

什么是 NAT 穿透？

NAT（Network Address Translation，网络地址转换）是 IPv4 时代解决 IP 地址短缺的核心技术。它让多个内网设备共享一个公网 IP，但也带来了副作用：内网设备无法直接被外网访问。

NAT 穿透（NAT Traversal）就是要解决这个「两个都在 NAT 后面的设备如何直接通信」的问题。核心思路是借助一台公网服务器作为中介，帮助双方交换地址信息，然后建立直连。

四种 NAT 类型

NAT 的行为不是标准化的，不同厂商实现不同。按映射规则可分为四类：

NAT 类型映射规则过滤规则穿透难度
Full Cone内网地址 → 固定公网地址任何外网地址都可访问⭐
Address-Restrict内网地址 → 固定公网地址仅已通信的外网 IP 可访问⭐⭐
Port-Restrict内网地址 → 固定公网地址仅已通信的外网 IP:Port 可访问⭐⭐⭐
Symmetric不同目标 → 不同公网映射仅已通信的外网 IP:Port 可访问⭐⭐⭐⭐

STUN / TURN 协议原理

STUN（Session Traversal Utilities for NAT）

STUN 协议让客户端发现自己的 公网 IP 和端口。工作流程：

  1. 客户端向 STUN 服务器发送请求
  2. NAT 设备将源地址替换为公网地址
  3. STUN 服务器看到公网地址，将其返回给客户端
  4. 客户端得知自己的公网映射地址

TURN（Traversal Using Relays around NAT）

⚠️ 注意：TURN 不是真正的 P2P，所有流量都经过中继服务器，会带来额外的延迟和带宽成本。通常作为最后手段（fallback）。
STUN vs TURN 对比
STUN:  Client A ←——直连——→ Client B
              (通过 STUN 交换地址后直接通信)

TURN:  Client A ←→ TURN Server ←→ Client B
              (所有流量经服务器中转)

三种 NAT 组合的处理策略

实际场景中，两个客户端的 NAT 类型组合决定了穿透策略：

Client AClient B能否穿透策略
Full Cone任意类型✅ 可以A 的映射固定，B 可直接连接
Restrict / Port-RestrictRestrict / Port-Restrict✅ 可以双方同时打洞即可
SymmetricFull Cone✅ 可以Full Cone 端无需打洞
SymmetricSymmetric❌ 无法双方映射都不固定，必须用 TURN 中继
💡 经验法则：约 80% 的 NAT 组合可以通过 STUN 打洞成功。剩余 20%（主要是 Symmetric + Symmetric）需要 TURN 中继。

TCP vs UDP 穿透的区别

维度UDP 穿透TCP 穿透
连接模型无连接，直接发包即可需要三次握手建立连接
打洞方式双方同时发送 UDP 包双方同时发送 SYN 包
成功率较高（~80%）较低（~50-60%）
NAT 支持大多数 NAT 支持部分 NAT 不支持 TCP 打洞
超时时间映射保持时间短（~30s）映射保持时间长（连接期间）
典型应用WebRTC、VoIP、游戏较少使用，多回退到 TURN

UDP 打洞代码示例

udp_hole_punch.py
import socket
import threading

def create_udp_socket(port):
    """创建 UDP socket 并绑定到指定端口"""
    sock = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
    sock.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
    sock.bind(('0.0.0.0', port))
    return sock

def punch_hole(sock, target_addr):
    """向目标地址发送打洞包"""
    # 持续发送 UDP 包，触发 NAT 建立映射
    while True:
        sock.sendto(b'PUNCH', target_addr)
        time.sleep(0.5)

def listen_for_peer(sock):
    """监听对端发来的数据"""
    while True:
        data, addr = sock.recvfrom(1024)
        print(f"收到来自 {addr} 的消息: {data.decode()}")
        # 收到数据后，NAT 已建立映射，可以回复
        sock.sendto(b'ACK', addr)

# 使用流程:
# 1. 从 STUN/信令服务器获取对端公网地址
# 2. 同时启动打洞和监听
peer_addr = ('5.6.7.8', 8000)  # 从信令服务器获取
sock = create_udp_socket(5000)

threading.Thread(target=punch_hole, args=(sock, peer_addr), daemon=True).start()
listen_for_peer(sock)

TCP 打洞代码示例

tcp_hole_punch.py
import socket
import threading

def tcp_hole_punch(local_port, target_addr):
    """TCP 打洞：同时发起 SYN 连接"""
    sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    sock.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
    sock.bind(('0.0.0.0', local_port))

    # 设置非阻塞，因为 connect 可能成功也可能失败
    sock.setblocking(False)

    try:
        # 同时发起连接（打洞）
        sock.connect_ex(target_addr)
    except BlockingIOError:
        pass  # 预期行为：连接正在进行中

    return sock

# TCP 打洞的关键:
# 1. 双方必须使用相同的本地端口
# 2. 双方必须同时发起 connect
# 3. 如果 NAT 支持，SYN 包会穿透并建立连接
# 4. 否则 connect 会超时，需要回退到 TURN
⚠️ TCP 打洞注意事项：TCP 打洞要求两端使用相同的本地端口，且同时发起连接。很多 NAT 设备不支持这种行为，因此实际项目中 TCP P2P 通常直接回退到 TURN 中继。

WebRTC 中的 ICE 框架

现代 WebRTC 使用 ICE（Interactive Connectivity Establishment）框架，自动组合 STUN 和 TURN：

ICE 候选收集流程
ICE 候选收集顺序:
1. Host Candidate     → 本地 IP（直连）
2. Server Reflexive   → STUN 获取的公网 IP
3. Relayed Candidate  → TURN 中继地址

ICE 尝试顺序:
Host → Server Reflexive → Relayed
(优先直连，失败后逐级回退)

总结

  • NAT 穿透解决的是「两个 NAT 后设备如何直接通信」的问题
  • 四种 NAT 类型：Full Cone → Restrict → Port-Restrict → Symmetric，穿透难度递增
  • STUN 用于发现公网地址并打洞，TURN 用于无法穿透时的中继
  • UDP 打洞成功率约 80%，TCP 打洞成功率约 50-60%
  • Symmetric + Symmetric 组合无法穿透，必须使用 TURN
  • WebRTC 的 ICE 框架自动组合多种候选地址，优先直连、失败回退
