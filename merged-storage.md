## 一、分布式存储架构

### 1.1 存储类型

分布式存储系统主要分为三种类型：

| 类型 | 特点 | 代表 |
|------|------|------|
| **分布式文件存储** | 非结构化数据，提供文件语义 | GFS、HDFS、CephFS |
| **块存储** | 提供裸块设备，性能最高 | Ceph RBD、EBS |
| **对象存储** | 扁平命名空间，HTTP 接口 | Swift、Ceph RGW、S3 |

> 对象存储大多在分布式文件系统基础上实现。

### 1.2 核心设计维度

一个分布式文件存储系统需要满足：**基本读写功能、性能、可扩展性、可靠性、可用性**。

围绕这些需求，系统设计涉及以下关键模块：

| 模块 | 核心问题 |
|------|---------|
| **系统架构** | 有无中心节点？存储节点有无主从？ |
| **集群管理** | 节点上下线、心跳状态、逻辑拓扑（region/zone/node/partition） |
| **数据定位** | 如何根据文件名快速找到数据位置 |
| **存储引擎** | 数据在单机上以何种形式存储 |
| **存储副本** | 副本数量、位置、一致性保证 |
| **一致性协议** | 如何保持副本内容一致 |
| **数据恢复** | 节点故障后如何恢复数据 |
| **数据清理** | 直接删除 vs 标记删除 |
| **通信协议** | 节点间用 HTTP 还是 RPC |

### 1.3 两种核心架构模式

#### 模式一：有中心节点 + 存储节点有主从

**代表**：GFS、HDFS、小米 Pegasus

```
Client → Master（调度） → StorageNode（主） → StorageNode（从）
```

**Master 节点职责**：
1. **存储全局视图**：集群拓扑（存储池→分区→服务器→磁盘→目录），缓存在内存，定期持久化
2. **读写调度**：客户端写请求先到 Master 获取文件 ID（包含多副本位置信息），根据节点状态和权重选择合适节点
3. **存储节点选主**：为每个备份组选择主从节点
4. **数据恢复调度**：维持副本数量和内容正确性

**StorageNode 职责**：
1. **单机存储引擎**：以 Partition（目录）为副本划分单位；小文件合并写入大文件（随机写→顺序写，大幅提升吞吐）
2. **备份一致性**：主节点将数据推送到从节点，收到所有从节点成功响应后才持久化本地并返回成功（强一致）
3. **心跳汇报**：向 Master 汇报容量和资源使用情况，等待调度任务

**Client 职责**：
- 集群信息缓存（减少与 Master 交互）
- 异常处理（重试/超时，最大限度保证可用性）

**性能瓶颈分析**：
- 小文件场景：某些磁盘的吞吐能力（数据分布不均）
- 大文件场景：**主节点出口网卡**（3 副本时出口流量 = 入口 × 2）→ 解决方案：以 Partition 为单位做副本，每个节点都有主从角色，流量均衡

#### 模式二：无中心节点 + 存储节点无主从

**代表**：Swift、Ceph

```
Client(Proxy) → Ring(一致性Hash) → StorageNode（对等）
```

**Proxy/Client 职责**：
1. **备份一致性**：采用 Quorum 协议（R+W>N）。Swift 默认 N=3, W=2, R=1（弱一致）或 R=2+x-newest=true（强一致）
2. **系统可靠性**：引入 handoff 节点（备用节点），故障时重试写入

**StorageNode 组件**：

| 组件 | 职责 |
|------|------|
| **Object Server** | 单机存储引擎，文件内容存为文件，元数据存为文件属性，以时间戳命名 |
| **Object Replicator** | 保证副本数量和位置正确性；通过 hash 对比 + rsync 推送保证一致性 |
| **Object Auditor** | 检测比特级错误，隔离损坏文件 |
| **Object Updater** | 异步更新元数据（高负载时延迟处理） |

**关键设计——Ring**：
- 每个节点都拥有完整的 Ring 文件（集群配置信息）
- Ring 变更时需同步到所有节点
- 需要监控进程对比各节点 Ring 文件 MD5，保证一致

**性能瓶颈分析**：
- 小文件场景：磁盘吞吐 + 无小文件合并处理
- 大文件场景：**Proxy 节点出口网卡**（3 副本时出口 = 入口 × 3，比有主从的方案更严重）

### 1.4 数据定位方式

| 方式 | 原理 | 优点 | 缺点 | 代表 |
|------|------|------|------|------|
| **Hash 计算** | 按数据特征计算 hash，映射到磁盘 | 无需元信息，任何节点可计算 | 增减节点引起数据迁移 | 一致性 Hash |
| **元数据查询** | 存储映射关系，查询定位 | 灵活，可精确控制 | 元数据服务器可能成为瓶颈 | GFS/HDFS |

### 1.5 存储引擎：合并 vs 不合并

| 方式 | 优点 | 缺点 | 代表 |
|------|------|------|------|
| **不合并**（一个文件对应一个文件） | 实现简单 | 小文件读写受磁盘随机 I/O 制约 | Swift |
| **合并**（多文件追加写入大文件） | 随机写→顺序写，小文件性能大幅提升 | 需要二次映射（偏移定位），实现复杂 | GFS、自研系统 |

### 1.6 一致性模型

| 模型 | 定义 | 适用 |
|------|------|------|
| **强一致性** | 更新成功后，所有读操作立即获得最新数据 | 金融、交易 |
| **弱一致性** | 更新后需要一段时间才能读到最新数据 | 一般业务 |
| **最终一致性** | 经过一个时间窗口后保证一致 | 对象存储、CDN |

## 二、GFS/HDFS/Ceph 对比

| 维度 | GFS/HDFS | Ceph | Swift |
|------|----------|------|-------|
| **架构** | 有中心（NameNode/Master） | 无中心（CRUSH 算法） | 无中心（Ring） |
| **存储节点** | 有主从 | 有主从（OSD） | 无主从 |
| **数据定位** | 元数据查询 | CRUSH 算法（两级 Hash） | 一致性 Hash（Ring） |
| **一致性** | 强一致 | 可配置（强/最终） | 最终一致（Quorum） |
| **扩容** | 几乎无限制，平滑 | 受容错域制约，不完全平滑 | 需同步 Ring |
| **可靠利用率** | ~95% | ~50-70%（需提前扩容） | ~70% |
| **小文件** | 不擅长（NameNode 内存瓶颈） | 一般 | 不擅长 |
| **大文件** | 擅长 | 擅长 | 一般 |
| **存储类型** | 文件存储 | 文件/块/对象统一 | 对象存储 |
| **单点问题** | NameNode（可 HA） | 无 | 无 |
| **运维复杂度** | 中等 | 高 | 中等 |

### Ceph 的数据写入流程

```
对象 → PG（预定义的固定数量 Hash 分片）→ CRUSH 算法 → OSD（物理磁盘）
```

Ceph 的所有对象先被 pre-hash 到固定数量的桶（PG）中，然后根据集群物理架构（crushmap）选择落在具体的机器磁盘上。

## 三、Ceph 运维实战

### 3.1 扩容的"坑"

#### 3.1.1 扩容粒度受容错域制约

- Ceph 一次只能扩一个"容错域"（副本隔离级别：磁盘/机器/Rack/机房）
- 原因：去中心化导致 PG 位置随 crushmap 变化而变化
- 同时扩 2 台机器时，某些 PG 可能把 3 副本中的 2 个映射到新机器（无完整数据），导致 PG 停止服务

**解决方案**：初始规划时设定更大层次的容错域（如逻辑 Rack），扩容时扩一整个 Rack。

> **Tips**：很多公司数据日均增长量可能大于一台机器的存储容量，扩容粒度小会导致扩容速度赶不上写入速度。

#### 3.1.2 Crushmap 变更引发连锁反应

- 扩容进行中如果有硬盘损坏 → crushmap 改变 → PG 重新 re-hash → 大量 PG 位置重新计算
- 1000 台规模集群一天坏 2-3 块盘很正常
- crushmap 频繁变动 → 整体 I/O 下降（磁盘被 rebalance 占满）→ 部分数据暂时不可用

#### 3.1.3 PG 数量需随规模调整

- PG 理论上越多越好（数据分片随机性更好），但受 CPU/内存/网络限制
- 社区建议：~200 PG/OSD
- 集群从 10 台扩到 1000 台时，PG 数量不足会放大不平衡
- **调 PG 代价极大**：约 50% 的对象需要重新放置物理位置，服务质量严重下降

### 3.2 集群可靠利用率问题

**核心问题**：Ceph 集群整体使用率达到 **70%** 后就可能进入不稳定状态。

**原因**：
1. 去中心化 → 对象被"伪随机"分配到各磁盘 → 方差不可避免
2. 使用率高时，方差大的磁盘触及 95% 红线
3. 管理员调低 reweight → crushmap 变更 → 更多磁盘被写满 → 恶性循环
4. **Ceph 设计**：有盘满了就拒绝整个集群的写入服务（无法保证新对象不落在满盘）

**对比**：
- HDFS：中心节点可选择空闲机器写入，95% 仍可服务
- EMC Isilon：98-99% 仍可服务
- Ceph：**50% 就要开始准备扩容**

**成本影响**：预估 50PB 存储需要 300 台机器，但必须提前采购 200-300 台备用机器。

### 3.3 运维稳定性挑战

#### 3.3.1 80% 使用率后的恶性循环

```
磁盘变满 → 管理员调 reweight → crushmap 变更 → 
更多磁盘被写满 → 再调 reweight → Ceph 永远无法稳定 → 
管理员必须 7×24 盯着集群
```

> 这种情况对运维人员的士气是很大的伤害。

#### 3.3.2 缺少元数据

- Ceph 对象**没有 last_access_time**
- 冷热数据区分需要二次开发
- 垃圾数据清理、冷数据归档都需要额外工作

#### 3.3.3 运维建议

- 积累运维文档，踩过的坑不要踩第二次
- 新手容易犯错导致事故，需要技术积累管理
- Red Hat 版本的 Ceph 文档逻辑感更强，推荐参考

## 四、选型建议

### 4.1 中心化 vs 去中心化

| 维度 | 中心化（GFS/HDFS） | 去中心化（Ceph/Swift） |
|------|-------------------|----------------------|
| **扩容** | ✅ 几乎无限制，平滑 | ⚠️ 受容错域制约，不完全平滑 |
| **利用率** | ✅ 95% 仍可服务 | ❌ 50-70% 需扩容 |
| **单点** | ❌ NameNode 单点（可 HA） | ✅ 无单点 |
| **数据放置** | ✅ 精确控制，可选空闲节点 | ⚠️ 伪随机，方差不可控 |
| **运维** | ✅ 相对简单 | ❌ 复杂，需强团队 |
| **扩展上限** | ❌ 受 Master 物理极限 | ✅ 理论无限 |

> **没有银弹，只有 tradeoff。** 去中心化消除了单点问题，但带来了扩容不平滑、利用率低、运维复杂等代价。

### 4.2 场景化选型

| 场景 | 推荐 | 理由 |
|------|------|------|
| **大数据分析（MapReduce/Spark）** | HDFS | 生态完善，大文件顺序读写优秀 |
| **云平台块存储** | Ceph RBD | 统一存储，与 OpenStack 深度集成 |
| **对象存储（海量小文件）** | Ceph RGW / Swift | S3 兼容接口，水平扩展 |
| **需要文件+块+对象统一** | Ceph | 唯一支持三种存储类型的开源方案 |
| **团队运维能力有限** | HDFS | 运维相对简单，社区成熟 |
| **成本敏感** | 需综合评估 | Ceph 硬件利用率低，需考虑空置成本 |

### 4.3 关键决策因素

1. **数据规模与增长速度**：日增量是否超过单机容量？
2. **一致性要求**：强一致 vs 最终一致？
3. **存储类型需求**：文件/块/对象？
4. **团队能力**：是否有能力运维 Ceph？
5. **成本预算**：是否能接受 50% 的空置率？
6. **初始规划**：容错域、PG 数量等需要提前规划好，后期调整代价极大
